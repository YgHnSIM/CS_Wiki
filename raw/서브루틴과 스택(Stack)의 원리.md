# 서브루틴과 스택(Stack)의 원리

## 0. 핵심 논지

서브루틴은 단순히 "반복되는 코드를 따로 빼낸 것"이 아니다.

더 본질적인 문제는 이것이다.

> 어떤 코드 조각으로 잠시 이동했다가, 
> 일이 끝나면 정확히 호출 지점 다음으로 돌아오는 방법은 무엇인가?

현대 컴퓨터에서는 이 일을 보통 `CALL`, `RET`, `SP(Stack Pointer)`, `스택 프레임`이 처리한다. 하지만 초기 컴퓨터에는 이런 장치가 없었다. 그래서 

> 서브루틴의 역사는 곧 **복귀 주소(return address)를 어떻게 기억하고 복원할 것인가**의 역사라고 볼 수 있다.

이 글은 다음 흐름으로 서브루틴과 스택을 설명한다.

1. 서브루틴의 핵심 문제는 **복귀 주소 관리**다.
2. 스택이 없던 시절에는 Wheeler Jump 같은 **자기 수정 코드** 기법을 사용했다.
3. 스택은 **중첩** 호출과 **재귀** 호출을 가능하게 한 결정적 장치다.
4. 현대의 스택 프레임, 인터럽트, 보안 문제는 모두 이 복귀 주소 관리 문제에서 이어진다.

---

## 1. 서브루틴의 본질: 복귀 가능한 코드

서브루틴(Subroutine)은 프로그램 안에서 **반복적으로 사용할 수 있도록 분리한 코드 조각**이다.

하지만 서브루틴이 단순한 점프와 다른 이유는 **돌아와야 한다**는 데 있다.

단순 점프는 다음과 같다.

```text
현재 위치 -> 다른 위치로 이동
```

반면 서브루틴 호출은 다음 흐름을 가져야 한다.

```text
호출 지점
-> 서브루틴으로 이동
-> 서브루틴 실행
-> 호출 지점 다음 명령으로 복귀
```

따라서 서브루틴의 핵심 질문은 다음과 같다.

> 서브루틴이 끝난 뒤 어디로 돌아갈지 어떻게 기억할 것인가?

이 "돌아갈 곳"을 '**복귀 주소(return address)**'라고 한다.

---

## 2. CALL/RET 이전의 세계

현대 컴퓨터에서는 보통 다음 명령어가 있다.

```text
CALL subroutine
RET
```

`CALL`은 복귀 주소를 저장하고 서브루틴으로 이동한다. 
`RET`은 저장된 복귀 주소를 꺼내 원래 흐름으로 돌아간다.

하지만 초기 컴퓨터, 특히 EDSAC 같은 기계에는 오늘날식 `CALL`, `RET`, 호출 스택이 없었다. 그럼에도 사람들은 코드를 재사용하고 싶었다.

초기 저장 프로그램 컴퓨터에서는 코드와 데이터가 같은 메모리에 있었다.

```text
100: A 100 F      ; 명령어
101: G 200 F      ; 명령어
300: 42           ; 데이터
301: 17           ; 데이터
```

중요한 점은 다음이다.

> 명령어도 메모리에 저장된 값이다.

따라서 프로그램은 메모리에 있는 데이터를 바꾸듯이, 메모리에 저장된 명령어 자체도 바꿀 수 있었다. 이것이 '**자기 수정 코드(self-modifying code)**'의 전제다.

초기 서브루틴 기법은 이 성질을 적극적으로 이용했다.

---

## 3. 폐쇄형 서브루틴이란 무엇인가

초기 프로그래밍에서 중요한 개념 중 하나가 '**폐쇄형 서브루틴(closed subroutine)**'이다.

폐쇄형 서브루틴은 다음 성질을 가진다.

> 호출되면 자기 일을 수행하고, 끝나면 호출 지점 다음으로 스스로 돌아오는 서브루틴

즉 실행 흐름이 다음처럼 닫힌다.

```text
호출 -> 서브루틴 실행 -> 호출 지점 다음으로 복귀
```

반대로 복귀 위치가 고정된 코드는 재사용하기 어렵다.

예를 들어 서브루틴 끝에 다음 명령이 고정되어 있다고 하자.

```text
250: E 102 F
```

이 경우 서브루틴은 항상 102번지로만 돌아간다. 300번지나 500번지에서 호출하면 올바르게 복귀할 수 없다.

폐쇄형 서브루틴의 과제는 **호출 위치가 달라져도 그때마다 알맞은 복귀 주소를 만들어내는 것**이다.

## 4. EDSAC과 Wheeler Jump

EDSAC에서 사용된 대표적인 기법이 **Wheeler Jump**다. David Wheeler가 고안한 이 방식은 `CALL/RET`도 스택도 없던 환경에서 복귀 주소를 만들어내기 위한 방법이었다.

핵심 아이디어는 다음과 같다.

> 복귀 주소를 따로 저장하는 대신, 
> 호출부의 명령어 자체를 재료로 삼아 
> 서브루틴 끝에 복귀 점프 명령을 만들어 넣는다.

### 4.1 호출부 구조

호출부가 100번지에 있고, 서브루틴이 200번지에서 시작한다고 하자.

```text
100: A 100 F      ; 자기 자신의 명령어 값을 accumulator에 더함
101: G 200 F      ; 200번지 서브루틴으로 이동
102: ...          ; 복귀 지점
```

여기서 복귀 지점은 102번지다. 호출에 사용된 명령이 두 개이기 때문이다.

첫 줄 `A 100 F`는 개념적으로 100번지에 저장된 자기 자신, 즉 `A 100 F`라는 명령어 값을 accumulator에 넣는 역할을 한다.

그다음 `G 200 F`가 실행되면 제어 흐름은 200번지의 서브루틴으로 이동한다.

### 4.2 서브루틴 시작부

서브루틴은 다음과 같이 시작할 수 있다.

```text
200: A 3 F        ; accumulator 안의 A 100 F를 E 102 F로 바꿈
201: T 250 F      ; 그 결과를 250번지에 저장
202: ...          ; 서브루틴 본문
...
250: E 102 F      ; 실행 중 만들어진 복귀 점프
```

처음부터 250번지에 `E 102 F`가 들어 있는 것은 아니다. 서브루틴 시작부가 실행되면서 250번지의 명령어가 만들어진다.

### 4.3 핵심 식

Wheeler Jump의 핵심은 다음 식으로 요약할 수 있다.

```text
A m F + Memory[3] = E (m+2) F
```

뜻은 다음과 같다.

- `A m F`는 호출부 첫 명령이다.
- `m+2`는 호출부 다음의 복귀 지점이다.
- `Memory[3]`에는 `A m F`를 `E (m+2) F`로 바꾸는 보정값이 들어 있다.

EDSAC 명령어를 개념적으로 다음처럼 보면 이해하기 쉽다.

```text
[ opcode ][ address ][ length ]
```

`A 100 F`는 하나의 명령어이지만, 내부적으로는 opcode, address, length 필드를 가진 값이다. 여기에 적절한 상수를 더하면 다음 변화가 동시에 일어난다.

```text
[ A ][ 100 ][ F ]
+ [ E-A ][ 2 ][ 0 ]
= [ E ][ 102 ][ F ]
```

즉 하나의 덧셈으로 opcode는 `A`에서 `E`로 바뀌고, 주소는 `100`에서 `102`로 바뀐다.

### 4.4 전체 흐름

```text
m:     A m F       ; 자기 자신을 accumulator에 넣음
m+1:   G n F       ; n번지 서브루틴으로 이동
m+2:   ...         ; 복귀 지점

n:     A 3 F       ; A m F를 E (m+2) F로 변환
n+1:   T p F       ; p번지에 복귀 점프 저장
...
p:     E (m+2) F   ; 호출부 다음 위치로 복귀
```

이 방식 덕분에 같은 서브루틴을 여러 위치에서 호출할 수 있었다.

## 5. Wheeler Jump의 역사적 의미와 한계

Wheeler Jump는 현대의 관점에서는 낯설고 위험해 보인다. 
하지만 당시에는 매우 중요한 도약이었다.

### 5.1 의미

Wheeler Jump는 다음을 가능하게 했다.

1. 같은 서브루틴을 **여러 위치에서** 재사용할 수 있었다.
2. 프로그램을 긴 일직선 명령열이 아니라 **기능 단위 조합**으로 만들 수 있었다.
3. 서브루틴 **라이브러리와 API 문서화**의 출발점이 되었다.
4. 현대 함수 호출의 핵심 문제인 복귀 주소 관리를 구체적으로 드러냈다.

EDSAC의 서브루틴 라이브러리는 단순한 코드 모음이 아니었다. 사용자는 필요한 수학 루틴을 골라 자신의 프로그램에 결합할 수 있었고, 각 루틴을 어떻게 호출해야 하는지 설명하는 카탈로그도 필요했다. 이는 오늘날 라이브러리와 API 문서의 초기 형태로 볼 수 있다.

### 5.2 한계

그러나 Wheeler Jump에는 뚜렷한 한계가 있었다.

첫째, **중첩 호출이 어렵다.**

서브루틴 끝의 복귀 점프 자리가 하나뿐이면, 서브루틴 실행 중 다른 호출이 발생했을 때 기존 복귀 정보가 덮어써질 수 있다.

둘째, **재귀 호출이 어렵다.**

재귀는 같은 서브루틴이 자기 자신을 다시 호출하는 구조다. 하지만 복귀 주소를 저장하는 자리가 하나라면 이전 호출의 복귀 정보가 새 호출에 의해 사라진다.

셋째, **코드와 데이터의 분리가 약하다.**

명령어를 실행 중에 고치는 방식은 강력하지만, 현대 관점에서는 보안, 디버깅, 캐시 일관성, 유지보수 측면에서 문제가 많다.

넷째, **명령어 필드의 경계에 의존한다.**

`A m F + Memory[3] = E (m+2) F`가 성립하려면 주소 필드에서 발생한 carry가 opcode 필드로 넘어가 명령어를 깨뜨리지 않아야 한다.

이 한계들이 다음 단계의 필요성을 만든다.

> 복귀 주소를 서브루틴 내부의 한 자리에 덮어쓰지 말고, 
> 호출될 때마다 별도의 장소에 차곡차곡 저장할 수는 없을까?

그 답이 스택이다.

## 6. 스택의 등장은 무엇을 해결했는가

**스택(Stack)**은 나중에 넣은 것을 먼저 꺼내는 LIFO(Last In, First Out) 구조다.

서브루틴 호출은 자연스럽게 LIFO 구조를 따른다.

```text
main이 A를 호출
A가 B를 호출
B가 C를 호출

복귀 순서:
C -> B -> A -> main
```

가장 나중에 호출된 C가 가장 먼저 끝나야 한다. 
따라서 복귀 주소를 스택에 저장하면 중첩 호출을 자연스럽게 처리할 수 있다.

Wheeler Jump에서는 복귀 주소를 서브루틴 내부의 고정된 자리에 만들었다.

```text
서브루틴 내부의 p번지: E 복귀주소 F
```

스택 방식에서는 호출될 때마다 복귀 주소를 새로 쌓는다.

```text
CALL A -> return address push
CALL B -> return address push
CALL C -> return address push

RET C -> pop
RET B -> pop
RET A -> pop
```

이 차이가 결정적이다.

> 스택은 복귀 주소를 하나만 기억하는 구조에서, 호출 깊이만큼 여러 개 기억하는 구조로의 전환이다.

덕분에 중첩 호출, 재귀 호출, 예외 처리, 인터럽트 처리가 훨씬 자연스러워졌다.

## 7. CALL, RET, SP의 작동 원리

현대적인 서브루틴 호출에서 핵심 장치는 보통 다음 셋이다.

- `CALL`: 서브루틴 호출 명령
- `RET`: 복귀 명령
- `SP(Stack Pointer)`: 스택의 현재 꼭대기를 가리키는 레지스터

### 7.1 CALL 명령

`CALL`이 실행되면 CPU는 대체로 다음 일을 한다.

1. `CALL` 바로 다음 명령어의 주소를 계산한다.
2. 그 주소를 스택에 저장한다.
3. PC(Program Counter)를 서브루틴 시작 주소로 바꾼다.

개념적으로는 다음과 같다.

```text
PUSH return_address
PC <- subroutine_address
```

### 7.2 RET 명령

`RET`이 실행되면 CPU는 대체로 다음 일을 한다.

1. 스택 꼭대기에서 복귀 주소를 꺼낸다.
2. 그 주소를 PC에 넣는다.
3. 호출 지점 다음 명령부터 실행을 재개한다.

개념적으로는 다음과 같다.

```text
PC <- POP
```

### 7.3 SP의 역할

`SP`는 스택의 현재 위치를 가리키는 하드웨어 레지스터다.

| 구분 | 역할 |
|---|---|
| CPU 하드웨어 | `SP` 레지스터 제공, `PUSH`/`POP`/`CALL`/`RET` 실행, 주소 증감 처리 |
| 운영체제 | 프로세스나 스레드별 스택 영역 할당, 스택 크기 제한 관리 |
| 컴파일러 | 스택 프레임 구조 결정, 매개변수와 지역 변수 배치, 호출 규약 준수 |

## 8. 역사적 발전의 큰 흐름

서브루틴과 스택의 역사는 하나의 직선적 발명이 아니라 여러 해법의 축적이다.

### 8.1 1945-1946년: 튜링의 ACE 제안

Alan Turing은 ACE 설계 구상에서 서브루틴과 복귀 주소 스택에 가까운 아이디어를 논의했다. 이는 오늘날 CPU의 `SP`와 완전히 같은 구현은 아니지만, 복귀 주소를 여러 겹 저장해야 한다는 문제의식을 매우 이른 시기에 보여준다.

주의할 점은, 이 시기의 아이디어를 곧바로 현대적 스택 포인터의 상업적 구현으로 보면 안 된다는 것이다. 더 정확히는 **복귀 주소 스택이라는 개념적 선구**로 보는 편이 좋다.

### 8.2 1949-1951년: EDSAC과 Wheeler Jump

EDSAC에서는 하드웨어 스택 없이 Wheeler Jump 방식으로 폐쇄형 서브루틴을 구현했다. 이 방식은 자기 수정 코드를 이용해 호출 지점에 맞는 복귀 점프를 생성했다.

이 단계의 핵심은 다음이다.

> 스택은 아직 없었지만, 복귀 주소를 자동으로 만들어야 한다는 문제는 이미 분명했다.

### 8.3 1950-1960년대: 여러 호출 방식의 경쟁

초기 컴퓨터들은 복귀 주소를 저장하기 위해 여러 방식을 사용했다.

- 서브루틴 내부의 특정 메모리 위치에 복귀 주소 저장
- 레지스터에 복귀 주소 저장
- 별도의 하드웨어 return stack 사용
- 메모리 스택 사용

어떤 방식이 좋은지는 언어, 하드웨어 비용, 메모리 구조, 컴파일러 기술에 따라 달랐다.

### 8.4 1960년대: 스택 지향 아키텍처와 고급 언어

Burroughs B5000은 스택 지향 아키텍처의 대표적 사례다. 이 기계는 단순히 복귀 주소 저장을 위해 스택을 쓴 정도가 아니라, ALGOL 같은 고급 언어의 블록 구조와 표현식 평가를 하드웨어 차원에서 지원하려 했다.

즉 B5000은 다음 관점에서 중요하다.

> 스택이 단순한 보조 저장소가 아니라, 
> 고급 언어 실행 모델을 떠받치는 아키텍처 원리로 확장되었다.

### 8.5 1970년대: 마이크로프로세서와 SP의 대중화

Intel 8080 같은 마이크로프로세서에서는 `SP`, `PUSH`, `POP`, `CALL`, `RET`이 프로그래머가 직접 다루는 기본 장치로 자리 잡았다.

8080 계열에서 스택은 보통 높은 주소에서 낮은 주소 방향으로 자란다. `CALL`은 다음 명령어 주소를 스택에 저장하고, `RET`은 스택에서 16비트 주소를 꺼내 PC에 넣는다.

이 시기부터 스택은 컴퓨터 구조와 어셈블리 교육의 표준 주제가 되었다.

---

## 9. Intel 8080에서의 물리적 감각

초기 8비트 마이크로프로세서에서 주소는 보통 16비트였다. 
따라서 복귀 주소 하나를 저장하려면 8비트 바이트 두 개를 다뤄야 했다.

개념적으로 `CALL`은 다음 일을 한다.

```text
return_address의 상위 바이트 저장
return_address의 하위 바이트 저장
PC를 서브루틴 주소로 변경
```

8080의 스택은 낮은 주소 방향으로 자라므로, 
값을 넣을 때 `SP`가 감소하고 값을 꺼낼 때 `SP`가 증가한다.

```text
PUSH: SP 감소 -> 메모리에 저장
POP:  메모리에서 읽음 -> SP 증가
```

중요한 점은 프로그래머가 스택의 균형을 맞춰야 한다는 것이다. 서브루틴 안에서 임시로 레지스터를 `PUSH`했다면, `RET` 전에 반드시 반대 순서로 `POP`해야 한다.

그렇지 않으면 `RET`은 복귀 주소가 아니라 엉뚱한 데이터를 PC에 넣을 수 있다.

---

## 10. 스택 프레임: 함수 실행 환경의 구조화

스택은 처음에는 복귀 주소를 저장하기 위한 장치로 이해할 수 있다. 하지만 현대 함수 호출에서는 더 많은 정보를 담는다.

함수가 호출될 때마다 스택에는 해당 함수만을 위한 독립적인 영역이 생기는데, 이를 **스택 프레임(Stack Frame)**이라고 한다.

일반적인 스택 프레임에는 다음 항목들이 들어갈 수 있다.

1. 매개변수
2. 복귀 주소
3. 이전 프레임 포인터
4. 저장해야 할 레지스터
5. 지역 변수
6. 임시 계산 값

### 10.1 프레임 포인터의 필요성

`SP`는 `PUSH`와 `POP`에 따라 계속 변한다. 그래서 함수 내부에서 지역 변수나 매개변수를 안정적으로 찾기 어렵다.

이를 해결하기 위해 기준점 역할을 하는 **FP(Frame Pointer)** 또는 **BP(Base Pointer)**를 사용할 수 있다.

```text
지역 변수: FP - 4, FP - 8, ...
매개변수:   FP + 8, FP + 12, ...
```

현대 컴파일러는 최적화를 위해 프레임 포인터를 생략하기도 하지만, 개념적으로 스택 프레임을 이해할 때 FP는 매우 유용하다.

---

## 11. 인터럽트와 문맥 저장

스택은 함수 호출뿐 아니라 인터럽트 처리에도 쓰인다.

**인터럽트(Interrupt)**는 CPU가 프로그램을 실행하던 중 입출력 장치, 타이머, 예외 상황 등에 의해 잠시 다른 처리 루틴으로 이동해야 하는 사건이다.

인터럽트가 발생하면 CPU와 운영체제는 원래 실행 상태로 돌아올 수 있도록 현재 문맥을 저장해야 한다.

저장될 수 있는 항목은 다음과 같다.

- PC 또는 명령 포인터
- 상태 레지스터 또는 flags
- 범용 레지스터
- 필요하다면 현재 스택 관련 정보

흐름은 다음과 같다.

```text
현재 문맥 저장
-> 인터럽트 서비스 루틴(ISR) 실행
-> 문맥 복구
-> 원래 프로그램 재개
```

여기서도 본질은 같다.

> 잠시 다른 코드로 이동하되, 나중에 정확히 원래 상태로 돌아와야 한다.

서브루틴 호출의 복귀 주소 문제가 시스템 전체의 문맥 저장 문제로 확장된 것이다.

## 12. 보안: 스택이 공격 표면이 되는 이유

스택은 실행 흐름을 기억한다. 따라서 스택이 손상되면 실행 흐름도 손상될 수 있다.

### 12.1 스택 오버플로우

**스택 오버플로우(Stack Overflow)**는 무한 재귀나 너무 큰 지역 변수 때문에 할당된 스택 영역을 넘어서는 현상이다.

### 12.2 스택 버퍼 오버플로우

**스택 버퍼 오버플로우(Stack Buffer Overflow)**는 공격자가 스택에 있는 복귀 주소를 덮어써서, 함수가 `RET`할 때 공격자가 원하는 주소로 점프하게 만드는 공격이다.

이 공격이 가능한 이유는 복귀 주소가 스택에 저장되기 때문이다.

현대 시스템은 이를 막기 위해 여러 방어 기법을 사용한다.

- stack canary: 복귀 주소 근처에 훼손 감지 값을 둔다.
- NX bit: 스택에 있는 데이터를 코드로 실행하지 못하게 한다.
- ASLR: 코드와 스택의 주소를 예측하기 어렵게 한다.
- shadow stack: 복귀 주소를 별도의 보호된 스택에도 저장한다.

Wheeler Jump 시대에는 복귀 주소가 코드 자체에 만들어졌다. 현대에는 복귀 주소가 스택에 저장된다. 저장 위치는 달라졌지만, 공격자가 실행 흐름을 바꾸려 한다는 점에서는 같은 문제의 연장선에 있다.

## 13. 최종 정리

서브루틴과 스택의 발전은 다음 세 단계로 정리할 수 있다.

### 13.1 Wheeler Jump

복귀 주소를 명령어 덧셈과 자기 수정 코드로 만든다.

```text
A m F + Memory[3] = E (m+2) F
```

장점은 서브루틴 재사용이다. 한계는 중첩 호출과 재귀 호출이 어렵다는 점이다.

### 13.2 CALL/RET과 스택

복귀 주소를 호출될 때마다 스택에 쌓는다.

```text
CALL -> PUSH return address
RET  -> POP return address
```

장점은 중첩 호출과 재귀 호출이 자연스럽다는 점이다.

### 13.3 스택 프레임

복귀 주소뿐 아니라 함수 실행 환경 전체를 스택에 구조화한다.

```text
매개변수
복귀 주소
이전 FP
저장 레지스터
지역 변수
```

이 단계에서 스택은 단순한 자료구조가 아니라 프로그램 실행의 핵심 인프라가 된다.

결국 스택은 다음 질문에 대한 역사적 답이다.

> 컴퓨터는 잠시 다른 일을 하고도, 어떻게 정확히 원래 하던 일로 돌아올 수 있는가?
