## 목적

CS Wiki의 지식 지도는 링크 수가 많은 문서를 크게 그리는 네트워크가 아니라, 문서 사이의 **관계 유형과 그 이유**를 탐색하는 인터페이스다. 기존 문서는 수정하지 않아도 본문 언급, 관련 항목, 근거와 학습 순서가 자동으로 구분된다. 새 문서는 필요할 때 안정 식별자, 역사 시점, 추상화 층위와 편집자가 검토한 의미 관계를 덧붙인다.

이 스키마는 전체 지도를 한 번에 펼치지 않는다. 문서 중심 이웃, 두 문서 사이의 설명 가능한 경로, 학습 노선, 전체 의미 지도, 역사 시간축과 근거 추적이 같은 정규화 데이터에서 서로 다른 질문에 답하게 한다. 전체 의미 지도는 군집 개요에서 시작해 군집 내부, 군집 사이의 회랑, 한 문서의 1·2홉 맥락으로 차례로 깊어진다.

## 자동으로 생성되는 관계

| 관계 | 생성 위치 | 방향 | 의미 |
|---|---|---|---|
| `mentions` | 일반 본문의 위키링크 | 현재 문서 → 대상 | 본문이 다른 문서를 언급한다. |
| `related` | 마지막 `## 관련 항목` | 무방향 | 편집자가 함께 읽을 문서로 연결했다. 상호 링크도 한 간선으로 합친다. |
| `supports` | 프론트매터 `sources` | 소스·참고 자료 → 현재 문서 | 해당 자료가 현재 문서 전체의 등록 근거다. 개별 문장에 대한 증명이나 강도를 뜻하지 않는다. |
| `path_next` | 학습 경로의 인접 단계 | 이전 단계 → 다음 단계 | 특정 경로에서 권장하는 읽기 순서다. |

출처 절, 관계 표, 이미지 임베드와 코드 예제의 링크는 단순 본문 언급으로 중복 계산하지 않는다. 같은 도메인이라는 이유만으로 모든 문서를 서로 연결하지 않으며 도메인은 의미 지도에서 군집을 만드는 속성으로만 사용한다.

## 안정 식별자와 노출 범위

새 문서에는 가능한 한 `graph_id`를 부여한다.

```yaml
graph_id: computing-capability
graph_visibility: public
```

`graph_id`는 위키 전체에서 유일한 소문자 ASCII slug다. 한번 공개한 값은 제목·파일명·URL을 바꾸더라도 유지한다. 정규 소스와 참고 자료는 기존 `source_id`가 우선하며, 두 값이 모두 없는 기존 문서는 카테고리와 파일명으로 호환 ID를 만든다.

`graph_visibility`의 `public`은 전역 지도에 표시한다. `context`는 직접 연결된 문서를 살필 때만 보여 주고, `hidden`은 운영용 문서처럼 그래프 탐색에서 제외한다. `index`, `overview`, `log`는 필드가 없어도 기본적으로 숨긴다.

## 역사 시점과 분석 층위

역사 렌즈는 위키 문서의 작성일이 아니라 실제 출판·사건 시점을 사용한다.

```yaml
publication_year: 1937
event_start: 1936
event_end: 1945
historical_note: "1936년 제출, 1937년 출판"
historical_layer: theory
capability_layers: [computability, complexity]
```

`publication_year`는 원전의 출판 연도, `event_start`와 `event_end`는 사건이나 발전 국면의 기간이다. 하나의 사건 시점이면 `event_start`만 기록하며, `event_end`만 단독으로 쓸 수 없다. 출판 시점과 관찰·사건 기간이 다르면 두 값을 함께 보존한다. `historical_note`는 제출·출판, 초판·확인 판본, 관찰 기간처럼 숫자만으로 잃는 구분을 300자 이내에서 설명한다. 이 메모는 연도를 대신하거나 근거 없는 연대를 제안하는 필드가 아니다.

역사 층위는 `theory`, `machine`, `architecture`, `software`, `system`, `service`, `measurement` 가운데 하나를 사용한다. 역사 렌즈에서 각각 이론, 기계, 아키텍처, 소프트웨어, 시스템, 서비스, 측정 레인에 배치되며, 필드가 없는 기존 문서는 `unclassified` 레인에 남는다.

컴퓨팅 능력 층위는 [[컴퓨팅 능력이란 무엇인가]]의 구분에 맞춰 `computability`, `complexity`, `programmability`, `realized-performance`, `scalability`, `resource-efficiency`, `reliable-results`를 사용한다. 한 문서가 여러 층위를 연결한다면 배열에 모두 기록하되, 연대가 불확실한 경우 추정값을 넣지 않고 미상으로 둔다.

### 고정 시대 경계

역사 렌즈는 현재 문서 분포에 맞춰 경계를 이동시키지 않고 다음 아홉 구간을 고정한다.

| 안정 ID | 경계 | 탐색 질문 |
|---|---|---|
| `before-1800` | 1800년 이전 | 수와 규칙을 반복 가능한 기호 절차로 어떻게 고정했는가? |
| `1800-1935` | 1800–1935 | 계산 절차를 기계와 논리 기호로 어떻게 옮겼는가? |
| `1936-1945` | 1936–1945 | 기계적 절차의 경계와 프로그램 저장을 어떻게 정의했는가? |
| `1946-1959` | 1946–1959 | 범용 설계를 반복 가능한 프로그램 작성 능력으로 어떻게 바꿨는가? |
| `1960-1979` | 1960–1979 | 늘어난 회로와 자원을 실제 프로그램 능력으로 어떻게 전환했는가? |
| `1980-1999` | 1980–1999 | 잠재 성능, 데이터 이동과 같은 답의 조건을 어떻게 함께 다뤘는가? |
| `2000-2014` | 2000–2014 | 코어와 서버를 늘릴 때 전력·통신·꼬리 지연을 어떻게 통제했는가? |
| `2015-plus` | 2015년 이후 | 범용성을 조절하면서 특정 작업의 성능과 효율을 어떻게 높였는가? |
| `undated` | 연도 미상 | 연도를 추정하지 않고 어떤 층위와 질문에 연결해 읽을 수 있는가? |

사건이 여러 구간에 걸치면 `event_start`가 속한 구간을 주 구간으로 삼고 겹치는 구간도 검색 맥락으로 기록한다. 연도 필드가 하나도 없으면 문서를 `undated`에 정확히 한 번 배치한다. 제목·파일명·본문의 네 자리 숫자, `created`, `updated`, 같은 주제의 인접 문서로 연도를 자동 추정하지 않는다. 이후 원전 대조로 연도가 확인됐을 때만 메타데이터를 추가해 해당 문서를 이동한다.

## 명시 관계 작성

자동 링크만으로 표현할 수 없는 인과, 구조, 비교와 역사 관계는 본문에 표로 기록한다. 표는 `## 출처`보다 앞에 둔다.

```markdown
## 관계

| 관계 | 대상 | 설명 | 근거 |
|---|---|---|---|
| enables | [[저장 프로그램 컴퓨터]] | 프로그램 교체 비용을 재배선에서 기억장치 갱신으로 바꾼다. | [[First Draft of a Report on the EDVAC]] |
```

한 행은 `현재 문서 → 대상`으로 읽는다. 대상은 반드시 존재하는 위키링크여야 하고, 설명은 선 하나를 문장으로 풀어 읽을 수 있도록 관계의 이유를 적는다. `responds_to`, `enables`, `precedes`, `constrains`에는 관계를 직접 뒷받침하는 정규 소스나 참고 자료를 반드시 연결한다. 근거가 비어 있으면 lint와 그래프 빌드가 실패한다.

관계 표의 근거는 `type/source` 또는 `type/reference` 페이지여야 한다. 개념·분석 문서를 관계의 직접 근거 자리에 둘 수 없고, 같은 문서 안에서 관계 종류와 대상이 같은 행을 별칭만 바꿔 반복할 수 없다. 설명이나 근거가 늘어나면 기존 행에 합친다.

허용 어휘는 다음과 같다.

- 구조와 범위: `broader`, `narrower`, `related`, `synthesizes`
- 학습과 구현: `prerequisite_for`, `implements`, `exemplifies`
- 변화와 제약: `enables`, `constrains`, `responds_to`, `precedes`
- 측정과 논증: `measures`, `supports`, `contradicts`

새 관계 종류가 필요하면 비슷한 표현을 즉석에서 늘리지 않는다. 기존 종류로 표현할 수 없는 사례를 먼저 모은 뒤 관계 이름, 정방향·역방향 문장, 시각적 계열과 경로 탐색 비용을 함께 정의한다.

### 역사·인과 전환 합성

역사 렌즈가 인과 전환으로 승격하는 것은 편집자가 `## 관계`에서 직접 작성한 `responds_to`, `enables`, `precedes`, `constrains` 네 종류뿐이다. `mentions`, `related`, `supports`, `path_next`는 각각 언급, 함께 읽기, 근거, 학습 순서를 뜻하므로 인과관계로 재해석하지 않는다. 같은 시대·도메인·역사 층위에 있다는 사실만으로도 전환을 만들지 않는다.

같은 현재 문서가 `responds_to`와 `enables`를 모두 가질 때 다음과 같이 **한계 → 대응 → 새 능력**을 하나의 검토 전환으로 합성한다.

- `responds_to`의 대상 문서: 한계
- 관계를 기록한 현재 문서: 대응
- `enables`의 대상 문서: 새 능력

양쪽 관계가 모두 있을 때만 완전 전환으로 표시하고, 한쪽만 있으면 누락된 역할을 꾸미지 않은 부분 전환으로 남긴다. 같은 대응 문서의 부분 전환에 나중에 반대쪽 관계를 보강해도 전환 ID와 공유 URL은 바뀌지 않는다. `precedes`는 앞·뒤의 검토된 순서를, `constrains`는 제약과 제약받는 대상을 독립 전환으로 표시하며, 두 유형의 기준 시점과 위치는 의미상 출발 역할을 따른다. 관계의 설명, 근거, 기록 방향을 보존하며 알려진 연도가 기대 순서와 어긋나면 간선을 뒤집지 않고 연대 충돌로 드러낸다.

### 문서 근거 계보의 의미 경계

근거 계보 렌즈는 서로 다른 두 층위를 분리한다.

1. **문서 근거 묶음**: 일반 문서 프론트매터의 `sources`에서 만들어진 `supports` 관계다. 소스·참고 자료 페이지가 해당 문서 전체의 근거로 등록됐다는 사실만 뜻한다.
2. **검토된 관계 주장**: `## 관계` 표에 방향, 설명과 근거가 함께 기록된 편집 관계다. 자동 본문 언급이나 학습 순서, 같은 도메인은 이 층위로 승격하지 않는다.

따라서 근거 수를 진실도·신뢰도나 출처 독립성 점수로 환산하지 않는다. 같은 자료를 등록한 두 문서는 **공통 자료를 사용한다**고만 표현하며 같은 결론, 동의나 인과관계를 암시하지 않는다. `status: active`도 위키 운영 절차를 통과했다는 상태이지 모든 문장의 영구적 사실성을 보증하는 등급이 아니다.

소스·참고 자료 페이지의 `primary_sources`와 `supporting_sources`는 해당 자료 페이지 안의 재현 계보다. 각각 원자료와 보조·접근 자료를 뜻하며, 이를 일반 지식 문서의 개별 주장에 대한 직접·간접 근거 등급으로 옮기지 않는다. `source_kind`, `snapshot_status`, `retrieved`, `version`, `source_urls`는 출처를 다시 확인할 조건을 설명한다. 특히 `external-only`는 자료의 질이 낮다는 뜻이 아니라 로컬 보존 사본 없이 외부 URL에 의존한다는 뜻이다.

## 빌드 결과와 확장 규칙

정적 빌드는 `data/knowledge-graph.json`에 노드, 유형 관계, 17개 학습 경로와 필터용 분류를 출력한다. 각 간선에는 방향, 자동 추론·편집 관계 여부, 관계 계열, 탐색 비용, 발생 횟수와 최대 여섯 개의 대표 문맥을 기록한다. 발생 횟수는 모두 보존하되 문맥 표본 수를 제한하므로 문서와 반복 링크가 늘어나도 데이터 크기가 무제한으로 증가하지 않는다.

학습 노선 지도는 같은 그래프에서 편집자가 정한 경로 순서만 `data/learning-map.json`으로 다시 압축한다. 노선, 고유 역, 노선별 역 순서와 환승 멤버십을 분리하므로 한 문서가 여러 질문에 등장해도 문서 정보는 한 번만 저장한다. 노선 표시 순서는 `site/catalog.mjs`의 편집 순서를 보존하고, URL은 노선 slug와 역의 안정 그래프 ID를 사용한다. 데이터 URL의 버전은 CSS·JavaScript뿐 아니라 위키 본문과 그래프·노선 산출물까지 함께 해시하므로 문서나 경로만 추가된 배포에서도 이전 지도가 캐시에 남지 않는다.

전체 의미 지도는 `data/atlas/manifest.json`과 개요, 검색용 lookup, 군집·회랑·문서 초점 조각으로 나뉜다. 첫 화면은 모든 문서와 간선을 받지 않고 manifest와 군집 개요만 불러온다. 군집을 선택하면 그 군집의 문서와 경계 맥락, 회랑을 선택하면 두 군집을 실제로 잇는 최대 40개 관계, 문서를 선택하면 그 문서를 고정한 1·2홉 맥락만 지연 로드한다. 관계에는 종류·방향·자동 추론 또는 편집 관계 여부·대표 본문 문맥과 근거를 보존하므로 군집 사이의 선을 “왜 연결되는가”라는 문장과 목록으로 풀어 읽을 수 있다.

현재 군집의 기본 영역은 문서 프론트매터에 먼저 적힌 `domain/*` 태그로 정한다. 나머지 도메인은 검색·필터와 군집의 의미 범위를 설명하는 집계 속성으로 유지한다. `graph_visibility: public` 문서만 전역 군집·검색·통계와 자체 초점 조각에 포함한다. `context` 문서는 전역에서 숨기되 공개 문서와 직접 연결된 경우에만 해당 문서의 1홉 초점 맥락으로 포함하고, `context`에서 다시 이어지는 다른 문맥 문서를 2홉으로 확장하지 않는다. 새 문서가 아직 역사·컴퓨팅 능력 메타데이터를 갖지 않으면 값을 추정하지 않고 해당 필터에 `메타데이터 없음`을 표시한다. 군집 화면은 관계가 많아져도 최대 240개 노드와 240개 대표 관계만 그리며, 편집 관계·근거·학습 순서·관련 항목을 균형 있게 우선하고 자동 본문 언급은 소수의 표본만 남긴다. 전체 정규화 관계는 원 데이터와 문서·회랑 초점에서 계속 보존된다.

문서 초점 조각은 여러 문서를 한 큰 파일에 누적하지 않고 안정 문서 ID마다 별도 파일로 만든다. 따라서 문서가 수천 개로 늘어나도 한 사용자가 요청하는 조각은 최대 56개 노드와 180개 관계로 제한되고, 파일 수만 문서 수에 비례해 늘어난다. 각 군집은 자바스크립트가 없어도 들어갈 수 있는 정적 주소와 전체 문서 링크를 가지며, Canvas에 표시한 노드·관계와 같은 상태를 키보드 및 화면 낭독기용 HTML 목록으로 함께 제공한다. 모든 조각은 manifest와 같은 콘텐츠 지문을 사용해 새 배포에서 오래된 캐시가 섞이면 거부한다.

역사 렌즈는 `data/history/manifest.json`과 시대 개요만 첫 화면에 싣고, 조회 버킷과 시대별 조각은 필요할 때 불러온다. 직접 문서·전환 주소는 안정 ID의 결정적 해시로 최대 96레코드인 조회 버킷 하나를 찾고, 전체 검색은 같은 상한의 버킷들을 제한된 동시성으로 읽는다. 문서가 늘면 2의 거듭제곱 개수로 버킷을 다시 나누므로 단일 `lookup.json`이 끝없이 커지지 않는다. `graph_visibility: public`인 문서는 역사 메타데이터가 없어도 `undated`와 `unclassified`에 자동 수용하며, `context`와 `hidden` 문서는 전역 역사 렌즈에서 제외한다. 따라서 새 문서를 공개하면 별도 시대 목록을 손으로 갱신할 필요는 없지만, 연도·층위·능력과 인과관계는 원전 대조와 편집 판단 없이 자동 보완하지 않는다.

한 시대 조각은 최대 120개 문서와 96개 전환 요약으로 제한한다. 문서가 이 상한을 넘으면 안정 정렬에 따라 다음 조각을 만들고, 첫 조각은 `/map/history/<시대>/`, 이후 조각은 `/map/history/<시대>/<쪽>/` 정적 주소를 갖는다. `undated`도 같은 규칙과 `/map/history/undated/` 주소를 사용하므로 연도 미상 문서가 늘어나도 하나의 무제한 파일이 되지 않는다. 조회 버킷의 문서 위치는 안정 `graph_id`를 시대·쪽·조각에 연결하고, 모든 JSON 조각은 같은 콘텐츠 지문을 사용한다.

전환 하나의 관계가 많아지는 경우도 별도로 제한한다. 시대·조회 조각에는 역할과 관계를 대표하는 최대 6개 간선·32KiB 요약만 두고 생략 사실과 전체 수를 표시한다. 전체 역할·간선·방향·설명·근거는 최대 48레코드·64KiB인 `data/history/transitions/<전환 ID>/page-<쪽>.json` 조각에 정확히 보존한다. 각 조각은 간선과 그 양 끝 역할 문맥을 함께 담아 단독으로 읽을 수 있고, 브라우저는 한 쪽씩 불러와 캐시하며 이전·다음으로 이동한다. 자바스크립트가 없어도 요약에서 관계 표를 가진 대응 문서로 이동해 전체 관계를 읽을 수 있다. 새 문서나 고차수 관계가 추가되면 시대·조회·상세 조각 수만 늘어나며 한 번의 요청과 화면 표시 상한은 유지된다.

근거 계보는 `data/evidence/manifest.json`과 제한된 개요, 직접 조회, prefix 검색, 문서·관계 초점, 근거 역색인 조각으로 나뉜다. 첫 화면은 전체 문서–근거 행렬을 받지 않는다. 안정 문서·자료·관계 ID는 결정적 조회 버킷 하나에서 찾고, 제목·별칭 검색은 두 글자 이상 입력했을 때 정규화한 앞 2–4글자에 해당하는 prefix 버킷의 쪽만 차례로 읽는다. 모든 조회·검색 조각은 레코드 수와 64KiB 크기를 함께 제한하며, 한 원자 레코드가 상한을 넘으면 조용히 자르지 않고 빌드를 실패시킨다.

문서 초점에는 최대 8개·32KiB 미리보기만 두고 전체 문서–근거 연결은 최대 48레코드·64KiB 상세 쪽에 정확히 한 번씩 저장한다. 하나의 근거 자료가 많은 문서와 관계에서 사용되는 경우에도 같은 규칙으로 역색인을 나눈다. 각 상세 레코드는 연결과 반대편 문서·자료 문맥을 함께 포함해 그 쪽만 읽어도 의미가 완결된다. 브라우저 캐시는 단일 제한 LRU만 사용하며 현재 필요한 쪽 외의 전체 상세 목록을 메모리에 누적하지 않는다.

자바스크립트가 없는 정적 계보도 같은 확장성 경계를 갖는다. 근거 자료, 자료를 등록한 문서와 직접 근거가 명시된 검토 관계는 초점별로 각각 12개 단위의 같은 쪽 번호에 나누며, 한 관계는 해당 초점의 한 쪽에서만 렌더링한다. 쪽 이동은 첫 쪽·마지막 쪽과 현재 앞뒤 두 쪽만 표시하므로 고차수 허브에서도 쪽마다 전체 쪽 번호를 반복해 HTML을 제곱 규모로 키우지 않는다. 앞뒤 주변 링크를 따라 모든 쪽에 도달할 수 있다.

`public` 일반 문서는 근거가 아직 0개여도 누락하지 않고 미연결 상태로 수용하며, 공개 소스·참고 자료는 전역 자료 검색과 보존 통계에 자동 편입한다. `context` 자료는 공개 문서나 검토 관계가 실제로 직접 사용한 초점 안에서만 보이며 그 초점에서 따라갈 정적 자료 주소와 역색인은 만들되 전역 검색·통계·진입 목록·사이트맵에서는 제외한다. `hidden` 자료는 두 범위 모두에서 제외한다. 각 공개 문서·자료·검토 관계와 직접 사용된 문맥 자료에는 자바스크립트 없이도 접근할 수 있는 정적 주소와 실제 문서 링크가 있으며, 3레일 화면은 **원자료·재현 정보 → 정규 소스·참고 자료 → 지식 문서·검토 관계** 순서를 색뿐 아니라 번호, 방향 기호와 텍스트로 함께 표현한다.

새 일반 문서는 공개 범위라면 전체 지식 그래프와 검색에 자동으로 들어가지만, 학습 노선의 역을 자동 추측하지는 않는다. 읽는 순서는 편집 판단이므로 해당 문서를 `site/catalog.mjs`의 기존 경로에 배치하거나 새 경로를 선언했을 때만 노선 지도에 들어간다. 이후 역·환승 통계, 노선별 정적 주소와 자바스크립트 없이 이동할 수 있는 역 앵커는 빌드가 자동 생성한다. 중복 노선 ID, 두 역 미만의 노선, 같은 노선 안의 중복 역, 존재하지 않거나 비공개인 역, 중복·누락·순서가 맞지 않는 멤버십은 빌드 또는 브라우저 인덱스 검증에서 거부한다.

새 문서가 추가되면 다음 검사가 자동으로 실패를 알려야 한다.

- 중복되거나 형식이 잘못된 `graph_id`
- 허용 목록에 없는 역사·컴퓨팅 능력 층위
- 시작 연도 없이 기록한 종료 연도와 종료 연도가 시작 연도보다 이른 기간
- 300자를 넘는 `historical_note`
- 정의되지 않은 관계 종류, 없는 대상 문서, 설명이 비어 있는 관계
- 근거가 비어 있는 역사·인과 관계, 존재하지 않는 관계 근거 문서와 노드·간선 식별자 충돌
- 학습 노선의 중복 ID·역, 비공개·없는 역과 손상된 순서·환승 멤버십
- 의미 지도 manifest와 조각의 스키마·콘텐츠 버전 불일치
- 군집·회랑·문서 초점 조각의 없는 노드·관계 참조와 중복 ID
- 정적 군집 주소에서 빠진 문서, 안전하지 않은 조각 경로와 표시·압축 전송량 상한 초과
- 역사 시대의 중복·누락 문서, 고정 경계 밖 배치, 안전하지 않은 조각 경로와 120개 문서·96개 전환 요약 상한 초과
- 조회 버킷의 96레코드 상한, 전환 요약의 6개 대표 간선·32KiB 상한, 상세 쪽의 48레코드·64KiB 상한과 역할·간선 누락·중복
- 자동 관계가 인과 전환에 포함되거나 전환의 설명·근거·기록 방향이 소실된 경우
- 관계 표 근거가 소스·참고 자료가 아니거나 같은 문서의 관계 종류·대상이 중복된 경우
- 근거 계보의 문서·관계·자료 조회 누락, 문서–근거·관계–근거 연결의 중복, prefix 검색 버킷 불일치
- 근거 계보의 8개·32KiB 미리보기와 48레코드·64KiB 상세 쪽, 96레코드·64KiB 조회·검색 쪽 상한 초과
- 근거 계보 정적 화면의 12개 관계·자료 상한, 제한된 주변 쪽 이동, 직접 사용된 `context` 자료 주소와 전역 비노출 계약 불일치
- `primary_sources`와 `supporting_sources`를 일반 문서 주장 강도나 신뢰도 점수로 잘못 승격한 경우

시각화는 이 데이터를 지연 로드하며, 자바스크립트가 작동하지 않아도 같은 관계를 문서 링크 목록으로 읽을 수 있어야 한다. 색만으로 관계를 구분하지 않고 선 모양, 방향, 관계 이름을 함께 제공한다.
