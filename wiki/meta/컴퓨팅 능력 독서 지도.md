## 개요

[[컴퓨팅 능력이란 무엇인가]]를 통찰력 있게 읽으려면 “컴퓨터가 얼마나 빨라졌는가”라는 한 줄 연표보다 질문이 어떻게 늘어났는지를 따라가야 한다. 무엇을 계산할 수 있는가, 입력이 커질 때 비용은 어떻게 증가하는가, 새 계산을 기계에 부여하는 비용은 얼마인가, 실제 작업에서 얼마나 빠른가, 자원을 늘리면 무엇이 확장되는가, 전력과 데이터 이동에 얼마를 쓰는가, 결과가 정확하고 제시간에 도착하는가를 차례로 구분해야 한다.

이 지도는 새 학습 경로를 하나 더 만들지 않는다. 기존 22개 경로와 원전·개념·분석을 목적에 따라 조합하는 **메타 경로**다. 한 문서를 여러 경로에서 만났다면 처음 읽은 곳에서만 정독하고, 다음 경로에서는 그 문서를 건너뛴다. 정의 허브는 처음에, 역사 종합은 마지막에 두면 중복 없이 전체 구조를 볼 수 있다.

> 계산 가능 ≠ 사용 가능 ≠ 달성 성능 ≠ 유효한 답 ≠ 확장 가능 ≠ 에너지 지속 가능 ≠ 예측 가능한 서비스

## 이 지도 사용법

모든 문서는 세 종류로 나누어 읽는다.

1. **원전**은 당시 연구자가 무엇을 문제로 삼았고 어떤 조건에서 주장했는지 확인한다.
2. **개념**은 시대와 시스템이 달라도 다시 사용할 수 있는 구분과 어휘를 얻는다.
3. **분석**은 서로 다른 원전이 답한 질문을 비교하고, 한 병목의 완화가 다음 병목을 어떻게 드러냈는지 종합한다.

원전의 수치나 결론은 판본, 작업 부하와 시스템 경계를 떼어 내어 일반 법칙처럼 읽지 않는다. 특히 추상 계산 모델과 실제 성능, 고정 크기와 확대 크기, 최고 부하와 실제 운용, 규범 연산의 결정성과 프로그램 전체 재현성, 구성요소 지연과 종단 서비스 지연을 구분한다.

## 빠른 진입

시간이 제한되어 있다면 다음 세 단계로 읽는다.

1. 정의 허브인 컴퓨팅 능력이란 무엇인가에서 일곱 층위와 측정 조건을 확인한다.
2. 현재 질문에 가장 가까운 분석 한 편을 고른다.
   - 계산의 원리적 경계: [[기계가 계산한다는 말의 이론적 의미]]
   - 알고리즘·하드웨어의 역할 분담: [[더 빠른 하드웨어는 더 나은 알고리즘을 대신할 수 있는가]]
   - 새 작업을 부여하는 비용: [[범용성은 어떻게 컴퓨팅 능력이 되었는가]]
   - 하드웨어 잠재력과 실제 성능: [[더 빠른 프로세서는 왜 더 빠른 프로그램을 보장하지 않는가]]
   - 속도와 결과 계약: [[더 빠른 계산은 같은 답을 내는가]]
   - 자원 증가와 문제 규모: [[병렬 컴퓨팅은 시간을 줄이는가 문제를 키우는가]]
   - 성능과 전력 예산: [[전력 장벽은 성능 향상의 의미를 어떻게 바꾸었는가]]
   - LLM 처리량과 사용자 체감 서비스: [[초당 토큰 수는 왜 LLM 서비스 능력을 설명하지 못하는가]]
   - 일반 서비스의 평균과 꼬리: [[평균 성능은 왜 서비스의 컴퓨팅 능력을 설명하지 못하는가]]
3. [[컴퓨팅 능력의 발달사]]에서 선택한 질문이 전체 병목 이동의 역사에서 어디에 놓이는지 확인한다.

이 경로의 목표는 세부 사실을 모두 기억하는 것이 아니라, “더 강한 컴퓨터”라는 표현을 들었을 때 먼저 작업·규모·정확성·기한·에너지·시스템 경계를 묻는 습관을 얻는 것이다.

## 핵심 분석 시리즈: 가능성에서 서비스 능력까지

전체 논증을 따라가려면 아래 순서를 권한다. 각 문서는 앞 문서가 당연하게 두었던 조건을 다음 문서의 질문으로 바꾼다.

| 순서 | 문서 | 이 단계에서 추가되는 질문 |
|---|---|---|
| 입구 | 컴퓨팅 능력이란 무엇인가 | 무엇을 고정해야 두 시스템의 능력을 비교할 수 있는가? |
| 1 | 기계가 계산한다는 말의 이론적 의미 | 기계적 절차로 가능한 것과 불가능한 것은 무엇인가? |
| 연결 | 더 빠른 하드웨어는 더 나은 알고리즘을 대신할 수 있는가 | 계산 가능하더라도 입력 규모에 따라 비용이 어떻게 증가하며, 알고리즘과 하드웨어는 무엇을 각각 바꾸는가? |
| 2 | 범용성은 어떻게 컴퓨팅 능력이 되었는가 | 가능한 계산을 새 프로그램으로 표현·변경·재사용하는 비용은 얼마인가? |
| 3 | 더 빠른 프로세서는 왜 더 빠른 프로그램을 보장하지 않는가 | 작업·컴파일러·ISA·메모리가 하드웨어 잠재력을 얼마나 실현하는가? |
| 4 | 더 빠른 계산은 같은 답을 내는가 | 결과 비트·오차·과업 품질 가운데 무엇을 보존해야 같은 계산인가? |
| 5 | 병렬 컴퓨팅은 시간을 줄이는가 문제를 키우는가 | 같은 문제를 빨리 풀 것인가, 같은 시간에 더 큰 문제를 풀 것인가? |
| 6 | 전력 장벽은 성능 향상의 의미를 어떻게 바꾸었는가 | 제한된 전력에서 어떤 회로·시간·작업에 자원을 배분할 것인가? |
| 7 | 평균 성능은 왜 서비스의 컴퓨팅 능력을 설명하지 못하는가 → 빠른 서비스는 왜 가용한 서비스를 보장하지 않는가 | 분포의 꼬리, 실패 모형과 복구까지 포함해 결과를 제시간에 얼마나 안정적으로 제공하는가? |
| 종합 | 컴퓨팅 능력의 발달사 | 병목 이동이 구조·소프트웨어·측정 경계를 어떻게 함께 바꾸었는가? |

[[더 빠른 하드웨어는 더 나은 알고리즘을 대신할 수 있는가]]는 [[계산 복잡도]], [[다항 시간 환원]], [[NP-완전]]을 실제 성능 비교의 조건과 연결한다. 이는 계산 가능성과 실현 성능 사이에서 하드웨어의 상수배 향상과 입력 규모에 따른 증가율을 혼동하지 않게 하는 다리다.

## 원전 심화: 일곱 층위의 33개 코어

| 프로그래밍 가능성 | [[First Draft of a Report on the EDVAC]] → [[The Preparation of Programs for an Electronic Digital Computer]] → [[The FORTRAN Automatic Coding System]] → [[No Silver Bullet—Essence and Accidents of Software Engineering]] → [[The SPACE of Developer Productivity]] | 명령의 저장, 서브루틴·카탈로그, 고급 표기와 최적화가 새 작업을 부여하는 비용을 어떻게 낮추며, 그 비용 감소를 어떤 개발 결과로 측정하는가? | 설계·최초 실행·실용 운영을 한 종류의 최초성으로 합치지 않는다. 위키가 확인한 *The Preparation*의 1957년 제2판과 IBM 704용 Fortran을 후대 전체의 대표로 넓히지 않는다. Brooks의 1987년 구분과 SPACE의 2021년 측정 틀도 활동량 하나를 보편 생산성 지표로 만들지 않는다. |

| 신뢰 가능한 결과 | [[What Every Computer Scientist Should Know About Floating-Point Arithmetic]] → [[IEEE 754-2019 Standard for Floating-Point Arithmetic]] → [[Mixed Precision Training]] → [[MLPerf Training Benchmark]] → [[MLPerf Inference Benchmark]] → [[The Tail at Scale]] → [[Why Do Internet Services Fail and What Can Be Done About It]] → [[Recovery-Oriented Computing (ROC)]] | 같은 답의 계약은 무엇이며, 규범 연산·모델 품질·추론 시나리오·종단 서비스의 기한·복구를 어떻게 검증하는가? | Goldberg는 754-1985의 해설이므로 1991년 정오표와 함께 읽는다. 혼합 정밀도 학습의 품질 문턱은 비트 일치와 같지 않으며, MLPerf Training과 Inference의 판본별 규칙은 모든 과업이나 운영 부하를 대표하지 않는다. IEEE 공개 기록은 표준 전문이 아니고, 조건부 팬아웃 사례와 세 서비스의 장애 관찰을 모든 서비스의 분포·원인 비율로 넓히지 않는다. ROC의 복구 관점도 특정 기법의 보편적 개선률을 뜻하지 않는다. |

측정 실습에는 SPEC CPU 2026과 LINPACK을 사용해 실행 시간·처리량, `Rpeak`·`Rmax`, 검증 계약을 비교할 수 있다. 캐시 계보는 Slave Memories 논문, 전력 측정은 2007년 Green500 지침, 서비스 경계는 The Datacenter as a Computer를 보조로 읽는다. 이 자료들은 코어 원전의 질문을 구체화하지만, 현재 규정이나 모든 작업의 대표 성능을 대신하지 않는다.

|---|---|---|
| 전체 방향 잡기 | **컴퓨팅 능력의 발달사** | 일곱 층위와 역사적 병목 이동을 한 흐름으로 본다. |
| 이론에서 소프트웨어 계층까지 | **컴퓨팅의 기원** → **계산 가능성과 한계** → **저장 프로그램 컴퓨터** → **초기 소프트웨어와 호출 구조** → **goto와 구조적 프로그래밍** → **Fortran과 컴파일러** → **Unix와 C** → **라이브러리에서 API와 재사용으로** → **범용 기계와 프로그래밍 가능성** | 계산의 표현, 저장, 번역, 제어, 재사용과 이식성이 실제 범용성을 만든 과정을 본다. |
| 프로그래밍 가능성에서 개발 결과까지 | **Fortran과 컴파일러** → **라이브러리에서 API와 재사용으로** → **범용 기계와 프로그래밍 가능성** → **개발자 생산성과 측정** | 도구·재사용이 낮춘 표현 비용과 요구사항·품질·협업·몰입을 분리해 읽는다. |
| 알고리즘과 실용 성능 | **계산 가능성과 한계** → **알고리즘 복잡도와 실용성** → **더 빠른 프로세서의 조건** | 절차의 존재, 문제 사이 환원, 점근 자원 증가와 실제 시스템 병목을 구분한다. |
| 표현과 해석의 횡단 기반 | **논리 회로와 이진 표현** → **데이터 표현과 유니코드** | 동일한 비트도 회로·타입·인코딩·경계 규칙에 따라 다른 의미를 갖는다는 기반을 얻는다. |
| 잠재 성능에서 서비스 능력까지 | **더 빠른 프로세서의 조건** → **부동소수점 성능과 정확성** → **혼합 정밀도와 AI 품질** → **병렬 확장성의 두 질문** → **전력 장벽과 에너지 효율** → **서비스 성능과 꼬리 지연** | 실현 성능, 수치·과업 품질 계약, 자원 확장, 전력 예산과 종단 분포를 하나의 측정 문제로 연결한다. |
| Transformer 계산에서 서비스 추론까지 | **혼합 정밀도와 AI 품질** → **LLM 추론 시스템과 서비스 품질** → **서비스 성능과 꼬리 지연** | 모델의 정밀도·품질 계약에서 자기 주의의 메모리 이동과 자동회귀 서비스 지연으로 측정 경계를 넓힌다. |
| 장애를 포함한 서비스 결과 | **서비스 성능과 꼬리 지연** → **결함 허용과 가용성** | 정상 상태 분포, 실패 모형, 공통 원인과 탐지·복구 시간을 한 서비스 결과 계약으로 연결한다. |

전체를 읽을 때는 컴퓨팅 능력의 발달사 경로를 먼저 훑고, 관심 묶음을 하나 선택해 깊게 읽은 다음, 이미 읽은 중복 문서를 건너뛰며 마지막에 역사 종합 문서로 돌아온다. 논리·인코딩 경로는 일곱 층위 중 하나에만 속하지 않고 표현과 해석 전반을 가로지르는 기반이다.

## 읽기 완료 기준

각 원전이나 분석을 읽은 뒤 다음 여섯 칸을 한 줄씩 채울 수 있으면 그 문서를 읽은 것으로 본다.

| 기록할 항목 | 확인 질문 |
|---|---|
| 작업 | 무엇을 계산하거나 어떤 요청을 처리하는가? |
| 규모 | 입력, 문제, 병렬 자원, 팬아웃 가운데 무엇이 얼마나 큰가? |
| 결과 계약 | 비트 동일성, 허용 오차, 잔차, 완전성, 기한 가운데 무엇을 요구하는가? |
| 시스템 경계 | 프로세서, 메모리, 컴파일러, 네트워크, 전력, 클라이언트 중 어디까지 포함하는가? |
| 고정 조건 | 작업 크기, 실행 시간, 전력, 정밀도, 비용 가운데 무엇을 고정하는가? |
| 지표 | 시간, 처리량, 가속비, 작업/J, 활성 면적, 백분위 가운데 무엇을 보고하는가? |

마지막 과제는 “시스템 A의 컴퓨팅 능력이 B의 두 배다”라는 문장을 택해, 빠진 작업·결과 계약·고정 조건·측정 경계를 지적하고 비교 가능한 문장으로 다시 쓰는 것이다.

## 종합 질문

1. 한 역사적 전환은 계산 가능한 문제를 늘렸는가, 새 작업의 표현 비용을 낮췄는가, 같은 작업의 자원 비용을 줄였는가?
2. 프로세서에서 메모리, 전력, 서비스로 병목이 이동한 사례에서 왜 지표와 시스템 경계도 함께 바뀌었는가?
3. 낮은 정밀도, 부분 결과나 중복 실행으로 속도를 얻었다면 어떤 결과 계약과 자원 비용이 달라졌는가?
4. 같은 병렬 자원 증가는 고정 문제의 지연, 고정 시간의 문제 규모, 서비스 유효 처리량에 각각 어떤 능력을 만드는가?
5. 특화 가속기는 범용성을 줄이는가, 아니면 범용 소프트웨어가 접근할 수 있을 때 시스템 전체의 사용 가능한 범위를 넓히는가?
6. 과학 시뮬레이션과 대화형 서비스에 필요한 최소 측정 벡터를 각각 설계하면 어떤 값과 조건이 들어가야 하는가?

## 다음 조사 후보

현대 AI 추론 확장은 Transformer·입출력 인지 어텐션에서 반복 단위 스케줄링, 블록 기반 KV 캐시, 접두사 캐싱·추측 디코딩, 프리필·디코드 분리와 TTFT·TPOT SLO 아래 goodput, 표준화된 추론 벤치마크까지 반영했다. 다음 보강 후보는 같은 서비스 품질에서의 요청당 에너지·재현 가능한 자원 비용, 실제 운영 요청 분포와 표준 벤치마크 사이의 차이, 낮은 비트 가중치가 메모리 이동과 품질 계약을 바꾸는 조건이다.
