## 핵심 주장

[[계산 결과에서 실행 가능한 절차로]]는 계산 능력의 재사용을 “같은 코드를 다시 쓴 역사”보다 넓게 본다. [[수학 표]]는 이미 구한 결과를 저장하고, [[라이브러리]]는 결과를 만드는 절차를 저장한다. [[저장 프로그램 컴퓨터]]는 실행할 명령을 내부 기억장치의 기호로 만들며, [[캐시 메모리]]는 실제 실행 중 다시 참조될 가능성이 있는 명령과 데이터의 사본을 가까운 층에 둔다.

이 네 형태는 하나가 다음 형태로 곧바로 발전한 단일 계보가 아니다. 서로 다른 시대와 문제에서 등장했고 저장 대상과 사용자도 다르다. 그러나 함께 비교하면 컴퓨팅 능력이 단순히 연산량을 늘리는 데서 나오지 않고, **언제 계산하고 무엇을 남기며 누가 다시 사용할 것인가를 설계하는 데서도 나온다**는 공통점을 볼 수 있다.

## 네 가지 재사용 단위

| 형태 | 저장하는 것 | 계산 시점 | 입력 범위 | 사용자의 행위 | 줄이는 비용 | 대표적인 실패 방식 |
|---|---|---|---|---|---|---|
| 수학 표 | 미리 계산한 값 | 배포·사용 이전 | 표에 실린 유한한 격자와 보간 가능한 범위 | 값을 조회하고 필요하면 보간한다. | 반복 산술과 개인별 계산 노동 | 잘못된 값·단위·판본의 복제, 조회·보간 오류 |
| 라이브러리 | 결과를 만드는 실행 절차 | 호출할 때 | 루틴의 매개변수와 전제 조건이 허용하는 범위 | 루틴을 찾아 적재·호출·조합한다. | 재구현, 재검증과 프로그램 작성 노동 | 잘못된 호출 규약, 정밀도·버전·전제 조건 불일치 |
| 저장 프로그램 | 기계가 해석할 명령과 데이터 | 적재 후 실행할 때 | 기억장치와 명령 체계로 표현 가능한 절차 | 프로그램을 적재·복사·변경하고 실행한다. | 작업 변경을 위한 배선·장치 재구성 | 잘못된 프로그램, 주소·표현 오류, 기계 의존성 |
| 캐시 | 하위 기억장치에 있는 명령·데이터의 사본 | 실행 중 실제 요구가 생길 때 | 프로그램이 만들어 내는 주소 참조 흐름 | 보통 캐시를 직접 조작하지 않고 같은 주소를 재참조한다. | 느린 하위 계층의 평균 접근 대기 | 미스, 충돌, 잘못된 쓰기·일관성 처리 |

표의 행, 루틴의 매개변수, 프로그램의 명령 주소, 캐시의 태그는 모두 “무엇을 다시 찾을 것인가”를 식별하지만 같은 인터페이스는 아니다. 표와 라이브러리는 사람이 선택하고 문서화한 재사용 자산인 반면, 캐시는 실행 중 관찰된 접근을 하드웨어나 시스템이 자동으로 복제하는 계층이다. 저장 프로그램은 특정 재사용 자산이라기보다 값과 명령을 저장·변경 가능한 대상으로 만드는 기반이다.

## 결과를 미리 계산해 나누어 쓰기

수학 표는 계산 비용을 사용 이전에 집중한다. 표 제작자는 값의 범위와 간격, 단위, 자릿수, 반올림과 [[검산]] 방법을 정하고 결과를 인쇄하거나 배포한다. 사용자는 전체 계산을 되풀이하는 대신 값을 조회하며, 정확한 입력이 없으면 인접 항목 사이를 보간한다.

이 방식의 이득은 한 번의 제작비를 많은 조회에 나누는 데서 생긴다. 같은 값이 충분히 자주 필요하고 배포된 표를 많은 사용자가 신뢰할 수 있을 때 큰 효과가 있다. 반대로 입력 범위가 넓거나 정밀도 요구가 자주 바뀌면 표의 크기와 제작·검산 비용이 커진다. 잘못된 값이 중앙에서 인쇄되면 같은 오류가 여러 사용자에게 복제된다는 점도 결과 재사용의 위험이다.

## 값을 만드는 절차를 불러 쓰기

EDSAC의 서브루틴 라이브러리는 미리 계산한 모든 결과가 아니라 결과를 만드는 명령열을 보존했다. 사용자는 [[라이브러리 카탈로그]]에서 루틴의 동작, 사용법, 실행 시간, 저장 공간과 정밀도를 확인하고 호출했다. 계산은 호출할 때 수행되므로, 하나의 루틴이 전제 조건 안에서 여러 입력을 처리할 수 있다.

절차 재사용은 표의 유한한 입력 격자보다 넓은 범위를 다룰 수 있지만 실행 비용을 없애지는 않는다. 또한 이름 붙은 코드가 존재하는 것만으로 재사용이 성립하지 않는다. 호출 규약, 입력 표현, 부작용, 정밀도와 비용을 설명하는 인터페이스가 필요하다. 이 점은 [[서브루틴 라이브러리에서 API로]]가 다루는 문서화와 구현 은닉의 문제로 이어진다.

## 명령을 저장하고 작업 변경 비용을 낮추기

저장 프로그램 방식은 명령을 내부 기억장치에 둘 수 있게 해, 작업 변경을 물리적 배선이나 장치 설정에서 기호의 적재와 수정 문제로 옮겼다. 프로그램은 복사·보존·수정할 수 있는 대상이 되었고, 그 위에서 로더·번역기·라이브러리 같은 “프로그램을 다루는 프로그램”이 가능해졌다.

그러나 저장된 프로그램이 자동으로 재사용 가능한 라이브러리가 되는 것은 아니다. 다른 사람이 안정적으로 다시 쓰려면 프로그램을 기능 단위로 나누고 주소와 호출 상태를 관리하며 사용법을 문서화해야 한다. 저장 프로그램은 절차 재사용을 가능하게 하는 기반이고, 라이브러리는 그 가능성을 공동체의 반복 가능한 자산으로 조직한 한 방식이다.

## 실행 중 필요한 사본을 가까이 두기

캐시는 미리 선택한 수학 함수나 이름 붙은 루틴이 아니라, 실행 중 실제로 요구된 명령과 데이터의 사본을 빠른 층에 둔다. 같은 주소가 다시 요구되면 느린 하위 기억장치 접근을 피할 수 있다. 이 이득은 프로그램의 시간적·공간적 지역성, 주소 사상, 교체와 쓰기 정책에 달려 있다.

따라서 캐시는 결과 재사용이나 코드 재사용과 구분되는 **접근 재사용**이다. 어떤 값이나 코드가 의미상 중요해서가 아니라 다시 참조될 가능성이 있기 때문에 복제한다. 사용자가 명시적으로 선택하는 재사용 자산을 넘어, 시스템이 실행의 관찰된 패턴을 이용해 비용을 줄이는 단계라고 볼 수 있다.

## 비용은 사라지지 않고 이동한다

네 형태가 줄이는 비용과 새로 만드는 비용을 함께 보면 다음과 같은 이동이 드러난다.

- 수학 표는 사용자의 계산 시간을 제작자의 계산·검산 노동과 인쇄 공간으로 옮긴다.
- 라이브러리는 재구현 노동을 호출 시 실행 비용, 인터페이스 학습과 버전 관리로 옮긴다.
- 저장 프로그램은 작업 변경의 물리적 비용을 프로그램 작성·번역·적재와 디버깅 비용으로 옮긴다.
- 캐시는 하위 기억장치의 대기를 빠른 저장 공간, 태그·교체·일관성 관리 비용으로 옮긴다.

이 관점에서 [[컴퓨팅 능력이란 무엇인가#자원 효율|자원 효율]]은 단순히 자원을 덜 쓰는 성질이 아니다. 시간, 공간, 사람의 노동과 검증 비용 가운데 무엇을 먼저 지불하고 무엇을 반복해서 절약할지 선택하는 구조다. 한 계층의 비용 절감은 다른 계층의 저장·관리·검증 부담을 만들 수 있다.

## 재사용과 신뢰성

재사용은 검증된 결과나 절차를 널리 활용하게 하지만 오류의 영향 범위도 넓힌다. 잘못된 수학 표는 여러 계산에 같은 오차를 넣고, 잘못된 라이브러리 루틴은 이를 호출하는 프로그램에 결함을 전파한다. 저장 프로그램은 명령을 쉽게 수정할 수 있게 하지만 잘못된 수정과 주소 오류의 가능성도 연다. 캐시는 동일한 값의 여러 사본을 만들기 때문에 쓰기와 일관성 규칙이 필요하다.

따라서 [[컴퓨팅 능력이란 무엇인가#신뢰 가능한 결과|신뢰 가능한 결과]]는 재사용 그 자체에서 자동으로 나오지 않는다. 표에는 검산·판본·정오표가, 라이브러리에는 테스트·명세·버전이, 프로그램에는 번역·디버깅·실행 환경이, 캐시에는 태그·쓰기·일관성 정책이 필요하다. 재사용 범위가 커질수록 오류를 발견하고 수정된 자산을 다시 배포하는 체계도 중요해진다.

## 해석의 경계

수학 표를 라이브러리의 직접 조상이라고 부르거나 캐시를 수학 표의 현대판이라고 부르면 역사와 구조의 차이가 사라진다. 이 분석이 주장하는 것은 직접 계보가 아니라 비교 가능한 설계 질문이다.

1. 어떤 계산 또는 접근이 반복될 것이라고 예상하는가?
2. 결과, 절차, 명령과 사본 가운데 무엇을 저장하는가?
3. 비용을 사용 전, 호출 시점과 실행 중 어느 시점에 지불하는가?
4. 저장된 자산의 범위·정밀도·호환성과 최신성을 어떻게 명시하는가?
5. 오류가 재사용될 때 어느 계층에서 발견하고 교정하는가?

이 질문은 [[소프트웨어 재사용의 역사]]가 다루는 코드의 이름·검색·조합보다 한 단계 넓다. 계산 결과도 절차도 아니지만 실행 비용을 줄이는 사본까지 포함해, 컴퓨팅 능력을 **재사용 가능한 단위와 비용 배치의 설계**로 읽는다.
