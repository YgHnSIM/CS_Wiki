## 문제 제기

분산 서비스가 빠르게 응답한다는 말과 여러 복제본이 같은 상태를 본다는 말은 다른 계약이다. 전자는 보통 요청이 기한 안에 끝났는지를, 후자는 상태 변경이 어떤 순서로 확정되고 다른 사용자에게 어떻게 보이는지를 말한다. 장애 뒤 빨리 다시 응답한다는 주장에는 여기에 복구된 상태의 안전성까지 더해진다.

이 차이를 놓치면 세 가지 오류가 생긴다. CAP을 고정된 “둘 중 하나” 선택표로 오해하고, 합의 프로토콜의 안전성을 사용자 SLO로 오해하며, 전역 시간 지원을 네트워크·트랜잭션 비용을 없애는 장치로 오해하는 것이다. [[CAP 정리]], [[복제 로그와 합의]], [[외부 일관성과 시간 불확실성]]은 서로 다른 층의 질문을 준다.

## 1. 분할 중 무엇을 약속할 것인가

[[Brewer's Conjecture and the Feasibility of Consistent, Available, Partition-Tolerant Web Services]]가 보인 것은 비동기 네트워크 분할 모형에서 선형화 가능한 원자적 일관성과 모든 비고장 노드의 응답 보장을 동시에 만족시킬 수 없다는 점이다. 이는 평상시 p99나 월간 가동률을 계산하는 식이 아니다. 분할 중에 어떤 읽기·쓰기 응답을 허용하고, 오래된 값·충돌·실패 응답을 사용자 계약에서 어떻게 다룰지를 묻는 한계다.

따라서 “가용성을 선택했다”는 문장만으로는 정보가 부족하다. 어떤 노드·요청을 분모에 넣는지, 응답이 최신 상태를 뜻하는지, 거부·재시도·나중 병합을 어떻게 세는지, 분할을 어떻게 감지하는지를 함께 밝혀야 한다. [[가용성과 복구]]의 운영 지표는 이 형식적 가용성과 별도로 사용자 기능·품질·복구 시간을 측정한다.

## 2. 같은 로그와 빠른 진행은 다른 성질이다

[[In Search of an Understandable Consensus Algorithm]]의 Raft는 리더 선출·로그 복제·안전성을 분리한다. 필요한 복제본이 같은 로그 항목을 확정하고 같은 결정적 상태 기계를 적용하면, 상태 순서는 보존된다. 그러나 리더를 선출할 수 없거나 정족수 통신이 끊기면 안전성을 유지한 채 진행이 멈출 수 있다.

이 때문에 다음 질문을 섞으면 안 된다.

| 질문 | 직접 확인할 대상 | 사용자 지연과의 관계 |
|---|---|---|
| 서로 다른 명령이 같은 로그 위치에 확정되는가 | 로그 안전성·확정 규칙 | 안전해도 응답이 빠르다는 뜻은 아니다 |
| 새 요청이 앞으로 나아가는가 | 리더 선출·정족수·통신 상태 | 진행이 멈추면 대기·실패·재시도가 지연 분포에 나타난다 |
| 사용자가 유효한 결과를 받는가 | 기한, 기능·정확성, 오류·취소 규칙 | 합의보다 넓은 서비스·제품 계약이다 |
| 누가 거짓말할 수 있는가 | crash fault 또는 Byzantine fault 가정 | Raft의 보장을 비잔틴 내성으로 확대할 수 없다 |

[[꼬리 지연 시간]]은 이 마지막 서비스 경계에서 대기·재시도·팬아웃이 종단 분포를 어떻게 바꾸는지 본다. 합의의 안전성은 그 분포의 한 원인일 수 있지만 p99 하나로 대체할 수는 없다.

## 3. 전역 순서는 시간 비용을 숨기지 않는다

[[Spanner - Google's Globally-Distributed Database]]는 데이터 조각의 Paxos 복제, 여러 그룹의 2단계 커밋, TrueTime의 시간 불확실성 범위를 조합해 외부 일관성을 제공한다. 이때 시간 불확실성은 무시하는 오차가 아니라, 커밋 뒤 실제 시간이 안전한 순서 경계를 지났음을 판단하기 위해 기다릴 수 있는 값이다.

따라서 Spanner의 commit wait는 강한 시간 순서 계약과 맞바꾼 대기다. 이 대기만큼이 전체 지연이라는 뜻도 아니고, 일반적인 분산 서비스가 모두 같은 방식을 써야 한다는 뜻도 아니다. 리더 위치, 복제 확인, 트랜잭션 참여자, 경합, 네트워크와 요청 부하를 포함한 종단 측정에서만 서비스 비용을 판단할 수 있다.

## 4. 설계와 측정의 최소 계약

| 층 | 먼저 고정할 질문 | 함께 남길 관측값 |
|---|---|---|
| 일관성 | 읽기·쓰기·트랜잭션이 어떤 순서와 가시성을 보장하는가 | 오래된 읽기, 충돌, 중복·손실, 선형화·외부 일관성 여부 |
| 실패 모형 | 정지·재시작·분할·임의 행동 중 무엇을 가정하는가 | 정족수, 리더 선출, 인증, 허용 실패 수 |
| 진행성 | 어느 조건에서 요청이 확정되거나 거부되는가 | 대기·시간 초과·재시도·취소와 그 원인 |
| 종단 성능 | 사용자가 어떤 기한 안에 무엇을 받는가 | 평균·여러 백분위, goodput, 오류·부분 기능 |
| 복구 | 언제 안전한 상태·품질로 돌아왔는가 | 탐지·격리·상태 검증·복구 시간·영향 범위 |
| 자원 | 강한 계약을 위해 무엇을 썼는가 | 복제·네트워크·저장·CPU·전력, 유휴 여유와 트랜잭션 비용 |

이 표는 강한 일관성이 항상 옳거나 느슨한 일관성이 항상 빠르다고 결론내리지 않는다. 제품의 정확성, 사용자 경험, 지리적 배치, 장애 정책이 적절한 계약을 정한다. 비교 가능한 평가는 선택한 계약·실패 모형·부하·측정 경계를 먼저 고정해야 한다.

## 결론

분산 서비스의 능력은 “응답했다”라는 한 사건으로 측정되지 않는다. CAP은 분할 중의 형식적 경계를, Raft는 정지 장애 모형에서의 상태 순서를, Spanner는 전역 순서에 시간 불확실성과 대기가 어떻게 들어오는지를 보여 준다. 여기에 사용자 기한·품질·복구·자원 비용을 더해야 실제 서비스 결과 계약이 된다.

그러므로 빠른 응답 수치와 강한 일관성 주장을 함께 제시할 때는, 어떤 요청이 어떤 상태 보장을 받고, 어떤 실패에서 대기·거부되며, 어느 시점에 안전한 결과로 회복되는지를 같은 실험·운영 경계 안에 기록해야 한다.
