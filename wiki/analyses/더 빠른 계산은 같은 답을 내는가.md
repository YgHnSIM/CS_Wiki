## 문제 제기

계산 시간을 절반으로 줄였지만 마지막 숫자가 달라졌다면 프로그램은 두 배 빨라진 것인가. 이 질문에는 먼저 “같은 답”의 뜻이 필요하다. 모든 비트가 같은지, 참조값과의 오차가 같은 범위인지, 과학적 결론이 같은지, 벤치마크의 검증 규칙을 통과했는지는 서로 다른 기준이다.

[[부동소수점 정확성]]의 관점에서 성능은 답과 독립된 속도가 아니다. 같은 작업의 속도를 비교하려면 입력뿐 아니라 결과가 충족해야 할 정확성·재현성·품질 계약을 고정해야 한다. 계약을 벗어난 더 빠른 실행은 같은 계산의 가속이 아니라 다른 계산 조건과의 교환이다.

## 1. 기계 산술은 실수 산술의 축소판이 아니다

수학의 실수는 무한히 많은 값을 포함하지만 컴퓨터의 부동소수점 형식은 유한하다. [[What Every Computer Scientist Should Know About Floating-Point Arithmetic]]은 이 차이를 표현과 반올림, 상쇄와 예외 처리의 문제로 분석한다. 각 기본 연산의 결과가 목적 형식으로 반올림되기 때문에 실수에서 성립하는 대수적 변형이 기계 계산에서 마지막 비트까지 같은 결과를 보장하지 않을 수 있다.

이 사실은 부동소수점 계산이 임의적이라는 뜻이 아니다. [[IEEE 754-2019 Standard for Floating-Point Arithmetic]]의 공개 설명은 입력값, 연산 순서와 목적 형식이 정해진 규범 연산의 결과와 예외가 유일하게 결정된다고 밝힌다. 표준화가 만든 것은 “실수와 같은 계산”이 아니라 **유한 정밀도 산술을 서로 다른 구현에서 추론할 수 있는 공통 계약**이다.

## 2. 표준화는 재현성의 토대이지 완성은 아니다

표준 연산이 같아도 프로그램은 연산 순서를 선택한다. `a + b + c`를 어느 두 항부터 더할지, 합계를 직렬로 누적할지 병렬 트리로 묶을지, 중간 결과를 어느 형식에 저장할지에 따라 반올림 지점이 달라진다. 따라서 IEEE 754 준수와 프로그램 전체의 비트 동일성은 같은 주장이 아니다.

반대로 비트가 항상 같은 실행도 틀린 식이나 잘못된 입력을 반복하면 정확하지 않다. 재현성과 정확성을 교차하면 네 경우가 생긴다.

| 결과 | 재현 가능 | 재현 불가능 |
|---|---|---|
| 요구 정확도 충족 | 같은 조건에서 같은 유효 답을 반복 | 실행마다 비트는 달라도 모두 허용 오차 안일 수 있음 |
| 요구 정확도 미달 | 일관되게 잘못된 답 | 결과도 흔들리고 계약도 위반 |

좋은 측정은 어느 칸을 허용하는지 밝힌다. 규제·회귀 검사처럼 비트 동일성이 필요한 작업도 있고, 병렬 과학 계산처럼 오차 한계와 보존량을 만족하면 여러 반올림 경로를 허용하는 작업도 있다.

## 3. 최적화는 수치 계약을 전제로 한다

초기 [[The FORTRAN Automatic Coding System|Fortran]]의 설득력은 사람이 읽기 쉬운 수학적 표기를 기계가 효율적인 코드로 바꿀 수 있다는 데 있었다. 이때 [[컴파일러 최적화]]의 목표는 단순히 명령 수를 줄이는 것이 아니라 프로그램이 요구한 의미를 보존하면서 비용을 줄이는 것이다.

그러나 어떤 의미를 보존할지는 언어와 실행 옵션이 정한다. 반올림을 고려하지 않은 채 항을 재배열하거나 공통 부분을 합치면 연산 수와 데이터 이동은 줄어도 결과 비트와 예외 발생 시점이 달라질 수 있다. 병렬화 역시 부분합의 묶음과 순서를 바꾼다. 빠른 변환을 무조건 금지할 수도, 실수 대수에서 같은 식이라는 이유만으로 모두 허용할 수도 없다.

따라서 최적화 계약은 적어도 다음을 구분해야 한다.

1. 정확히 같은 연산 순서와 비트를 유지해야 하는가.
2. 지정된 오차·잔차 안에서 다른 순서를 허용하는가.
3. 정밀도, 예외와 특별한 값의 처리를 바꿀 수 있는가.
4. 결과의 과업 의미나 품질을 어느 정도까지 교환할 수 있는가.

이 조건을 명시하면 컴파일러와 병렬 런타임은 허용된 범위 안에서 속도·메모리·에너지를 최적화할 수 있다. 조건이 없으면 빠른 결과와 틀린 결과를 구분할 기준도 없다.

## 4. 벤치마크는 계산과 검증을 묶는다

[[The Linpack Benchmark]]의 `Rpeak`는 하드웨어 사양으로 계산한 이론적 최고 부동소수점 실행률이고, `Rmax`는 HPL 작업을 정해진 정밀도와 정확도 조건 아래 실행해 얻은 달성 성능이다. 계산을 많이 수행했다는 사실만으로 Rmax가 되지 않는다. 결과 검증을 통과한 연산만 같은 벤치마크의 성능으로 인정된다.

[[SPEC CPU 2026 Overview]]도 작업 부하와 측정 지표뿐 아니라 정답 검증, 재현 가능성, 비교 규칙을 벤치마크의 조건으로 둔다. 특정 최적화가 실행 시간을 줄였어도 요구한 출력을 만들지 못하면 유효한 SPEC 성능 결과가 아니다. 반면 SPEC의 허용 규칙을 통과한 결과가 모든 과학 계산에 충분한 수치 정확성을 보장하는 것도 아니다. 각 벤치마크는 자기 작업과 검증 경계 안에서만 의미가 있다.

분석용으로 이를 다음처럼 표현할 수 있다.

`유효 성능 = 결과 계약을 통과한 작업량 / 시간`

이 식은 새로운 표준 지표가 아니라 성능의 분모·분자 앞에 검증 조건이 있다는 점을 드러내는 정의다. 검증을 실패한 작업을 분자에 더하거나 정확도를 낮춘 뒤 같은 작업량으로 세면 능력을 과대평가한다.

## 5. FLOP/s도 답의 종류를 말하지 않는다

[[Roofline An Insightful Visual Performance Model]]은 최고 FLOP/s, 메모리 대역폭과 연산 집약도를 결합해 달성 가능한 성능의 병목을 보여준다. 그러나 FLOP/s는 몇 번의 부동소수점 연산을 얼마나 빨리 수행했는지를 나타낼 뿐, 그 연산 순서가 안정적인 알고리즘을 이루는지와 결과가 요구 오차 안인지 직접 말하지 않는다.

더 낮은 정밀도, 다른 알고리즘이나 근사 계산은 더 높은 처리량과 에너지 효율을 제공할 수 있다. 이는 중요한 컴퓨팅 능력의 확대일 수 있지만 **정확성 조건을 고정한 속도 향상**과 **정확성 조건을 바꾼 처리량 교환**을 분리해 보고해야 한다. “같은 답”의 동등성 기준이 성능 수치의 단위를 완성한다.

## 6. 네 겹의 결과 계약

| 계약 층위 | 고정할 것 | 대표 실패 |
|---|---|---|
| 산술 | 형식, 반올림, 예외, 연산 순서와 목적 형식 | 같은 식을 다른 순서로 계산해 비트가 달라짐 |
| 알고리즘 | 오차 경계, 안정성, 종료 조건, 입력 전처리 | 작은 연산 오차가 큰 출력 오차로 증폭 |
| 과업 | 잔차, 허용 오차, 보존량, 분류·순위·품질 | 수치 차이가 실제 결론이나 품질을 바꿈 |
| 측정 | 검증 규칙, 비교 입력, 실패·재시도 처리 | 틀린 실행을 처리량에 포함 |

네 층위는 서로 대체되지 않는다. 표준 산술은 알고리즘을 대신 검증하지 않고, 작은 잔차가 항상 해의 작은 전방 오차를 뜻하지 않으며, 벤치마크 통과는 측정하지 않은 응용의 정확성을 보장하지 않는다.

## 역사적 의미

컴퓨팅 능력의 초기 성능 서사는 초당 연산 수와 실행 시간에 집중하기 쉽다. 그러나 부동소수점 표준화는 “연산”이라는 단위 자체가 공통 의미를 가져야 기계 사이의 성능을 비교할 수 있음을 보여줬다. 컴파일러와 병렬 구조가 연산 순서를 더 적극적으로 바꾸면서 정확성 계약은 하드웨어의 세부 사항에서 언어·알고리즘·측정 규칙의 공동 문제로 넓어졌다.

LINPACK과 SPEC이 결과 검증을 성능 절차 안에 둔 것은 이 역사적 변화를 제도화한 사례다. 더 빠른 기계의 의미는 단순히 더 많은 산술 명령을 실행하는 기계가 아니라, **요구한 결과 계약을 지키면서 더 짧은 시간·더 적은 에너지로 더 큰 작업을 수행하는 시스템**으로 바뀌었다.

따라서 “더 빠른 계산은 같은 답을 내는가”에 대한 답은 조건부다.

> 같은 입력뿐 아니라 같은 결과 계약을 만족한다면 더 빠른 계산이다. 계약이 달라졌다면 속도 향상과 정확성·품질의 교환을 따로 보고해야 한다.
