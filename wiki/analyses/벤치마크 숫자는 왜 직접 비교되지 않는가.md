## 문제 제기

벤치마크 표에는 대개 숫자와 단위가 먼저 보인다. 그러나 `초`, `요청/초`, FLOP/s, W, J/token이 같다고 해서 두 줄이 같은 질문에 답하는 것은 아니다. 성능 수치는 항상 [[벤치마크 계약]]의 결과다. 계약을 빼고 숫자만 비교하면 한 시스템의 처리량, 다른 시스템의 단일 작업 지연, 또 다른 시스템의 이론 상한을 한 순위로 섞게 된다.

이 문제는 수치를 버리자는 주장이 아니다. 오히려 어떤 수치가 강한 증거인지 분명히 하기 위한 기준이다. 같은 작업·결과 조건·경계·규칙을 공유하는 두 결과는 직접 비교할 수 있고, 계약이 다른 두 결과는 각자의 적용 범위를 보존한 채 읽어야 한다.

## 1. 같은 단위가 같은 질문을 뜻하지 않는다

| 비교하려는 수치 | 실제로 다른 질문 | 직접 비교가 어려운 이유 |
|---|---|---|
| SPECspeed와 SPECrate | 한 작업의 완료 시간과 여러 복제본의 단위 시간 작업량 | 실행 시간과 처리량은 동시성·자원 배치가 다른 성능 목적이다. |
| LINPACK `Rmax`와 `Rpeak` | 특정 HPL 실행에서의 달성 성능과 사양에서 계산한 이론 상한 | 하나는 검증된 관측값, 다른 하나는 실행을 대신하지 않는 계산값이다. |
| Roofline 지붕과 실행 FLOP/s | 주어진 연산 집약도·대역폭 아래의 가능한 상한과 실제 실행 결과 | 지붕은 병목 분석을 돕지만 입력·구현·측정 구간의 관측값을 대체하지 않는다. |
| LLM의 토큰/초와 Server·Interactive goodput | 원시 토큰 생성률과 품질·TTFT·TPOT 조건을 통과한 요청률 | 입력·출력 길이, 도착 과정, 지연 한계와 유효 요청의 정의가 다르다. |
| 평균 전력(W)과 요청당 에너지(J) | 특정 시점 또는 구간의 전력과 시간 적분된 작업당 에너지 | 실행 시간이 달라지거나 계측 경계가 달라지면 같은 W가 같은 J를 뜻하지 않는다. |

[[SPEC CPU 2026 Overview]]은 이 차이를 명시적으로 제도화한다. SPECspeed와 SPECrate는 같은 프로그램 묶음을 쓰더라도 서로 다른 수행 방식을 측정한다. [[The Linpack Benchmark]]도 `Rmax`를 특정 문제 크기와 검증 아래의 달성 성능으로, `Rpeak`를 이론적 최대치로 분리한다. 이 둘을 나란히 적는 것은 유용하지만, 값의 종류를 바꾸지 않는다.

## 2. 결과 계약은 속도와 분리되지 않는다

수치 계산에서 “같은 결과”는 비트 동일성, 수치 오차, 잔차, 과업 품질처럼 여러 수준을 가질 수 있다. LINPACK의 검증은 지정한 선형대수 작업의 유효성을 확인하지만 모든 부동소수점 의미를 증명하지는 않는다. SPEC의 출력 검증도 정한 규칙 안에서만 유효하다. 그러므로 [[더 빠른 계산은 같은 답을 내는가]]가 다루는 결과 계약은 벤치마크 계약의 한 요소이며, 작업 부하·규칙·시스템 구성까지 같아야 “더 빠른 같은 계산”이라는 결론이 완성된다.

기계 학습에서는 품질 조건이 더 직접적으로 시간의 정의를 바꾼다. [[MLPerf Training Benchmark]]은 처리량 대신 목표 품질에 도달한 시간을 비교하고, [[MLPerf Inference Benchmark]]은 모델·데이터·품질·시나리오를 고정한 뒤 지연과 처리량을 계산한다. LLM의 경우 TTFT·TPOT와 생성 길이·품질 조건을 생략한 높은 토큰률은 사용자가 경험한 같은 서비스의 향상을 증명하지 못한다.

## 3. 시스템 경계와 시간 구간이 분모를 바꾼다

에너지 수치는 성능 수치보다 계측 경계에 더 민감하다. [[Power Measurement Tutorial for the Green500 List]]은 LINPACK 실행 중의 평균 시스템 전력과 같은 실행의 성능을 결합한다. [[MLPerf Inference Power Measurement]]은 전력 분석기와 공통 수집 절차로 벽면 AC의 전체 시스템 경계를 정의하며, GPU TDP나 구성요소 텔레메트리를 그 값으로 대체하지 않는다.

[[TokenPowerBench - Benchmarking the Power Consumption of LLM Inference]]는 이 원리를 LLM에 적용한다. GPU·노드·전체 시스템 경계와 프리필·디코드·유휴 구간을 구분하고, 전력을 시간에 따라 적분해 에너지를 얻는다. 따라서 `J/token`의 분모가 입력 토큰인지 출력 토큰인지, 어떤 단계와 어떤 시스템이 포함됐는지 밝히지 않은 값은 다른 결과와 직접 비교하기 어렵다.

## 4. 내부 비교 가능성과 외적 타당성은 다른 질문이다

계약이 같으면 **내부 비교 가능성**이 높다. 같은 규칙·판본·작업·품질·경계에서 수행한 A와 B의 차이는 해당 계약 안에서 해석할 수 있다. 반면 실제 서비스나 과학 작업에 얼마나 도움이 되는지는 **외적 타당성**의 질문이다. 표준 벤치마크가 반복 가능한 비교를 만들기 위해 실제 사용자의 모든 입력·지역·장애·비용 조건을 대표하지 않는 것은 결함이라기보다 범위의 선택이다.

따라서 좋은 보고서는 두 주장을 분리한다.

1. “이 구성은 판본 V의 작업 W와 규칙 R 아래 지표 M에서 결과 X를 냈다.”
2. “작업 W가 목표 환경을 어느 정도 대표하므로, 이 결과는 범위 Y의 설계 선택에 참고할 수 있다.”

첫 문장은 측정의 증거이고, 둘째 문장은 적용 범위에 대한 해석이다. 첫 문장이 있어도 둘째 문장이 자동으로 따라오지 않으며, 반대도 성립하지 않는다.

## 5. 숫자를 비교 가능한 문장으로 쓰는 법

성능 결과를 기록할 때는 다음의 순서가 안전하다.

1. **대상**: 작업·입력·데이터·요청 시나리오와 결과 조건을 쓴다.
2. **실행**: 시스템 구성, 소프트웨어·컴파일러, 병렬도, 규칙·판본과 반복 방식을 쓴다.
3. **경계**: 시간 관측의 시작·종료와 전력·에너지 계측의 포함 범위를 쓴다.
4. **값**: 단위, 분모, 집계(평균·백분위·최대·총량)와 값의 종류(관측·이론 상한·분석 모델)를 쓴다.
5. **결론 범위**: 같은 계약 안의 직접 비교인지, 다른 계약을 병렬로 소개하는 것인지 구분한다.

이 형식은 숫자를 덜 인상적으로 만드는 대신, 다음 실험·구매·최적화가 무엇을 바꿔야 하는지 드러낸다. 예를 들어 Roofline 분석은 메모리 병목을 개선할 후보를 제시하고, MLPerf 결과는 품질과 지연을 통과한 특정 시나리오의 처리 능력을 보이며, 전력 계측은 동일 실행의 자원 비용을 보인다. 이들은 경쟁하는 숫자가 아니라 서로 다른 질문을 연결하는 증거다.
