## 개요

“더 빠른 프로세서”는 서로 다른 세 가지 뜻으로 쓰일 수 있다. 클럭이 더 높은 프로세서, 한 사이클에 더 많은 명령이나 연산을 처리할 수 있는 프로세서, 특정 프로그램을 더 짧은 시간에 끝내는 프로세서다. 앞의 두 가지는 하드웨어의 잠재력을 나타내지만 세 번째 결과를 자동으로 보장하지 않는다.

실제 프로그램의 실행 시간은 작업 부하가 컴파일러를 거쳐 어떤 명령열이 되는지, 각 명령이 몇 사이클을 요구하는지, 사이클이 얼마나 짧은지, 캐시 미스와 메모리 대기가 얼마나 생기는지의 합성 결과다. 이를 분석용으로 단순화하면 다음처럼 쓸 수 있다.

`실행 시간 ≈ 명령 수 × 명령당 평균 사이클 × 사이클 시간 + 겹치지 못한 데이터 이동·대기`

이 식의 어느 항도 프로세서 사양 하나로 고정되지 않는다. [[Slave Memories and Dynamic Storage Allocation]]의 기억장치 계층과 [[Design and Implementation of RISC I]]의 명령어 집합·컴파일러 공동 설계는 서로 직접 이어지는 계보가 아니지만, **부품의 최고 속도와 프로그램의 달성 속도 사이에 변환 조건이 있다**는 점을 서로 다른 방향에서 보여준다.

## 잠재력에서 실현 성능까지

| 잠재 자원 또는 선택 | 성능으로 전환되는 조건 | 전환을 막는 대표 비용 |
|---|---|---|
| 짧은 사이클·높은 클럭 | 임계 경로와 파이프라인이 실제 칩에서 목표대로 작동 | 긴 신호 경로, 의존성, 파이프라인 정지 |
| 단순하고 규칙적인 명령 | 컴파일러가 레지스터와 명령 순서를 효과적으로 사용 | 명령 수·코드 크기 증가, 빈 지연 슬롯 |
| 빠른 소용량 기억장치 | 작업이 짧은 간격으로 명령과 데이터를 재사용 | 캐시 미스, 충돌, 교체와 쓰기 비용 |
| 높은 최고 연산률 | 필요한 데이터가 충분한 대역폭과 낮은 지연으로 공급 | 메모리 장벽, 낮은 연산 집약도 |
| 벤치마크 우위 | 실제 사용 작업과 입력·컴파일 조건이 비교 대상과 유사 | 작업 부하 편향, 서로 다른 시스템 경계 |

이 표는 “병목이 어디에 있는가”를 묻기 위한 틀이다. 한 항을 개선하면 다른 항의 비중이 커질 수 있으므로 개선 효과는 프로그램과 실행 조건마다 달라진다.

## 1. 캐시는 속도보다 재사용을 전제로 한다

Wilkes의 1965년 논문은 느린 주기억장치에서 요구된 단어를 빠른 소용량 기억장치에 자동으로 축적하는 구조를 설명했다. 같은 명령이나 데이터가 곧 다시 쓰이면 이후 접근은 빠른 층에서 끝난다. 반대로 재사용이 적거나 여러 주소가 같은 위치를 계속 밀어내면 빠른 층의 존재만으로는 대기를 줄일 수 없다.

따라서 [[캐시 메모리]]의 성능은 “캐시가 있다”나 “용량이 크다”보다 적중 시간, 미스율과 미스 패널티의 결합으로 이해해야 한다. 주소 사상과 교체 정책은 어떤 참조가 적중할지 바꾸고, 쓰기 정책은 데이터 일관성과 하위 계층 트래픽을 바꾼다. 빠른 하드웨어를 유효하게 만드는 핵심 변수 중 하나가 프로그램의 접근 패턴인 셈이다.

## 2. RISC는 명령을 줄이는 일이 아니라 비용을 재배치하는 일이다

RISC I은 초기 VLSI에서 흔한 명령을 짧고 규칙적으로 실행하려 했다. 레지스터 간 연산, `LOAD`/`STORE`, 고정 길이 형식, 작은 제어부와 컴파일러 스케줄링을 결합하면 제어 경로와 사이클을 단순화할 수 있었다. 그러나 [[축소 명령어 집합 컴퓨터|RISC]]의 단순성은 코드 전체의 실행 비용을 자동으로 줄이지 않았다.

RISC I 보고서는 데이터 메모리 접근 감소와 레지스터 윈도의 효과를 주장하면서도, 고정 길이 명령과 `NOP` 때문에 코드 크기와 명령어 인출이 늘 수 있다고 기록했다. 실제 칩은 시뮬레이션에서 가정한 400ns 사이클에 도달하지 못했는데, 배치에서 놓친 긴 신호 경로가 있었기 때문이다. 이는 ISA 수준의 설계 목표, 시뮬레이션 모델과 제작된 회로의 성능을 구분해야 함을 보여준다.

RISC I이 명령어 캐시를 후속 조건으로 요구한 점은 특히 중요하다. 데이터 접근을 레지스터로 옮겨 줄였더라도 명령어 수와 인출량이 늘면 다른 메모리 병목이 생긴다. 한 층의 단순화가 전체 비용을 없앤 것이 아니라 다른 층으로 이동시킨 것이다.

## 3. 컴파일러는 기계의 일부처럼 성능에 참여한다

고급 언어 프로그램과 프로세서 사이에는 [[컴파일러 최적화]]가 있다. 컴파일러는 어떤 명령을 선택하고, 값을 어느 레지스터에 두며, 의존성이 없는 작업을 어떤 순서로 배치할지 결정한다. 같은 ISA와 하드웨어에서도 이 선택에 따라 명령 수, 평균 사이클과 메모리 접근이 달라질 수 있다.

RISC I은 단순하고 규칙적인 구조를 컴파일러가 활용할 수 있다는 가정에 기대었다. 지연 분기 슬롯에 유용한 명령을 넣지 못하면 `NOP`가 남고, 레지스터를 충분히 활용하지 못하면 메모리 트래픽이 늘어난다. 하드웨어의 잠재력은 자동으로 나타나는 것이 아니라 코드 생성기가 작업의 구조를 그 자원에 맞게 표현할 때 실현된다.

수치 프로그램에서는 코드 생성이 답의 경계에도 참여한다. [[What Every Computer Scientist Should Know About Floating-Point Arithmetic]]과 [[IEEE 754-2019 Standard for Floating-Point Arithmetic]]을 함께 보면 규정 연산은 입력값·연산 순서·목적 형식 아래 결정되지만, 최적화가 순서나 중간 형식을 바꾸면 전체 결과가 달라질 수 있음을 알 수 있다. 따라서 더 빠른 목적 코드는 [[부동소수점 정확성|요구한 수치 계약]]을 지킬 때에만 같은 프로그램의 실현 성능 향상이다.

## 4. 캐시와 RISC는 서로 다른 병목 대응이다

두 설계 흐름을 하나의 직접 계보로 묶을 수는 없다. Wilkes의 논문은 빠른 소용량 기억장치로 느린 기억장치 접근을 완충하는 계층을 제안했고, RISC I은 흔한 명령 경로와 초기 VLSI 구현의 복잡성을 ISA·레지스터·컴파일러 사이에 재배분했다.

그럼에도 둘을 함께 읽으면 중요한 공통점이 드러난다.

1. 성능은 가장 빠른 부품보다 가장 자주 드는 비용에 좌우된다.
2. 작업 부하의 동적 행동은 정적인 사양표보다 중요할 수 있다.
3. 한 층의 개선은 다른 층의 트래픽과 복잡성을 바꾼다.
4. 하드웨어와 소프트웨어의 경계는 고정되어 있지 않다.
5. 설계 목표·시뮬레이션과 실제 측정을 구분해야 한다.

이는 [[메모리 장벽]]을 캐시나 RISC가 “해결했다”는 뜻이 아니다. 오히려 1995년의 [[Hitting the Memory Wall]]은 프로세서와 DRAM의 속도 격차가 계속 커질 때 캐시만으로 모든 접근 패턴을 보호할 수 없음을 경고했다.

## 5. 성능은 무엇으로 확인해야 하는가

[[SPEC CPU 2026 Overview]]가 설명하듯 CPU 벤치마크는 프로세서만 시험하지 않는다. 실행 결과에는 컴파일러, 메모리 계층과 시스템 구성이 함께 반영된다. 그러므로 “CPU A가 CPU B보다 빠르다”는 문장은 작업 부하, 입력, 컴파일 옵션, 메모리 구성과 지표가 같을 때에만 제한된 의미를 갖는다.

[[Roofline An Insightful Visual Performance Model]]은 훗날 이 문제를 최고 연산 성능과 `메모리 대역폭 × 연산 집약도`라는 두 상한으로 표현했다. 계산 자원이 더 빨라도 작업의 데이터 재사용이 낮으면 메모리 지붕이 성능을 제한한다. 반대로 데이터가 충분히 공급될 때에만 연산 지붕을 향한 벡터화와 명령 수준 병렬성의 효과가 드러난다.

그러나 높은 달성 FLOP/s도 결과의 유효성을 직접 말하지 않는다. LINPACK과 SPEC이 결과 검증을 성능 절차에 포함하듯, 속도 비교에는 비트 동일성·수치 허용 오차·과업 품질 중 무엇을 고정했는지가 필요하다. 이 조건은 [[더 빠른 계산은 같은 답을 내는가]]에서 별도로 분석한다.

2단계에서 다루는 시스템 경계는 단일 프로그램, 컴파일러, 프로세서와 메모리 계층까지다. 병렬 시스템의 규모와 에너지는 후속 단계에서 확장했으며, 분산 서비스의 [[꼬리 지연 시간]]과 사용자 경계의 결과 조건은 [[평균 성능은 왜 서비스의 컴퓨팅 능력을 설명하지 못하는가]]에서 다룬다.

## 종합

더 빠른 프로세서가 더 빠른 프로그램을 보장하지 않는 이유는 “프로세서 속도”가 프로그램 실행을 구성하는 여러 비용 가운데 일부만 나타내기 때문이다. 실현 성능은 다음 관계의 산물이다.

`작업 부하 × 컴파일러 × ISA × 마이크로아키텍처 × 메모리 계층`

어느 항도 홀로 성능을 보장하지 않으며, 한 항의 개선은 다른 항의 병목을 드러낼 수 있다. 따라서 프로세서의 컴퓨팅 능력을 평가하려면 클럭이나 최고 처리율보다 **어떤 작업이 어떤 코드와 접근 패턴으로 바뀌고, 실제 구성에서 어디를 기다리는지**를 측정해야 한다.
