## 문제 제기

보안 경계는 “외부 입력을 받는다”는 한 줄에서 끝나지 않는다. 같은 바이트열은 디코더에 따라 다른 문자열이 될 수 있고, 같은 문자열처럼 보이는 값도 다른 코드 포인트 시퀀스일 수 있으며, 같은 숫자처럼 보이는 값도 부호·폭·단위·연산 순서에 따라 다른 메모리 크기나 인덱스가 될 수 있다. 시스템이 검증한 값과 실제로 소비한 값이 다르면, 검증은 경계 바깥에서 이루어진 셈이 된다.

[[CWE-20 Improper Input Validation]]은 이 문제를 길이·타입·문법·일관성·업무 규칙·동등성의 속성으로 정리한다. [[CWE-190 Integer Overflow or Wraparound]]은 수치 입력과 파생된 크기가 표현 범위를 넘을 때 작은 할당·잘못된 인덱스·경계 밖 접근으로 이어질 수 있음을 보인다. 두 자료를 함께 읽으면 텍스트 보안과 메모리 안전성은 서로 다른 도메인이라도 “해석 계약의 단절”이라는 공통 구조를 가진다.

## 1. 소비 전에 하나의 표현을 정한다

안전한 처리 흐름은 보통 다음 순서를 명시한다.

| 단계 | 질문 | 실패했을 때의 안전한 처리 |
|---|---|---|
| 입력 수집 | 어느 경로·인코딩·신뢰 수준에서 왔는가 | 출처와 원 바이트·메타데이터를 구분해 기록한다 |
| 디코드·파싱 | 이 바이트열이 정한 형식의 구조인가 | 잘못된 시퀀스·누락 필드·애매한 문법을 명시적으로 거부한다 |
| 정준화 | 비교 정책에서 동등한 표현을 어떻게 정하는가 | 정책상 필요한 경우 한 번 정준화하고 그 결과를 다음 단계에 사용한다 |
| 검증 | 타입·길이·범위·필드 일관성·업무 규칙을 만족하는가 | 허용된 명세 밖의 입력을 거부하거나 명시된 오류로 끝낸다 |
| 인가 | 이 주체가 이 자원·작업을 사용할 수 있는가 | 검증된 값과 분리된 권한 판단을 적용한다 |
| 출력·전달 | 다음 파서가 데이터를 어떤 문맥에서 읽는가 | 대상 문맥의 인코딩·이스케이프·프로토콜 규칙을 적용한다 |

이 표는 모든 제품에 같은 구현 순서를 강제하는 표준이 아니다. 핵심은 각 단계의 입력·출력 표현과 실패 의미를 정하고, 한 단계에서 확인한 속성을 다음 단계가 무효화하지 않게 하는 일이다. 예를 들어 replacement character로 바꾼 텍스트를 허용할지, 정규화 전·후 어떤 값을 감사 로그에 남길지, 외부 API가 반환한 크기를 다시 검증할지는 제품 계약이다.

## 2. 정규화는 검증도, 인가도 아니다

[[유니코드 정규화]]는 같은 글자처럼 보이거나 동등한 문자열을 목적에 맞는 형식으로 맞춘다. 하지만 정규화가 모든 시각적 혼동을 제거하지는 않는다. [[Unicode Security Mechanisms]]가 다루는 [[동형이의 문자]]와 mixed-script 위험은 식별자 허용 범위, 경고, 표시 정책을 따로 요구한다.

따라서 다음 두 극단 모두 피해야 한다.

- “NFC로 바꿨으니 같은 사용자다” — 정규화 동등성과 계정·권한의 동일성은 다르다.
- “유사 문자 차단만 하면 안전하다” — 정상적인 다국어 이름, 업무 규칙, 계정 중복, 표시·복구 절차가 함께 설계돼야 한다.

정책이 정규화된 문자열을 키로 삼는다면 정규화한 뒤에 검증·저장·비교·인가를 일관되게 수행한다. 반면 원문 보존·표시가 필요하면 원 표현과 정준 표현의 역할·로그·변환 시점을 분리한다. 이 원칙은 텍스트를 “필터링해서 안전하게 만들었다”는 주장보다 재현 가능하고 검토 가능한 경계를 만든다.

## 3. 수치도 해석될 때 경계가 된다

입력 `n`이 `0 ≤ n ≤ limit`을 만족해도, `n × item_size`, `offset + length`, `signed → unsigned` 변환이 안전하다는 결론은 나오지 않는다. 예를 들어 항목 수와 항목 크기가 각각 허용 범위 안이어도 곱이 표현 범위를 넘으면 실제 할당 크기는 의도보다 작아질 수 있다. 이후 코드가 원래 항목 수만큼 쓰면 메모리 경계를 넘을 수 있다.

[[C Integer and Shift Semantics]]가 보여 주듯 C에서는 unsigned 래핑과 signed overflow의 의미가 다르다. “기계가 2의 보수이므로 값이 되돌아간다”는 직관을 C의 signed 연산 규칙으로 옮길 수 없다. 따라서 검사는 결과 뒤에 한 번 붙이는 습관이 아니라, 연산 전 조건·타입 폭·부호·변환·오류 경로를 명시하는 계약이어야 한다.

## 4. 검증과 방어의 역할을 분리한다

| 오해 | 왜 부족한가 | 필요한 추가 계약 |
|---|---|---|
| 악성 문자 몇 개를 제거했다 | 다른 인코딩·구문·표현이 같은 소비자에서 다른 의미를 만들 수 있다 | 명세 기반 파싱·allowlist 검증·대상 문맥 이스케이프 |
| 입력값의 최대만 검사했다 | 음수·부호 변환·곱셈·덧셈으로 파생된 값이 범위를 넘을 수 있다 | 최소·최대·단위·연산 전 범위 검사·checked arithmetic |
| 정규화했으니 식별자가 안전하다 | 동형이의 문자와 권한 동일성은 정규화 동등성과 다르다 | 스크립트·표시·중복·계정 복구·인가 정책 |
| 클라이언트에서 검사했다 | 요청을 수정하거나 다른 경로로 제출할 수 있다 | 서버와 최종 권한 집행 지점의 재검증 |
| 입력이 잘 형성됐다 | 출력·쿼리·명령 문맥에서 안전하다는 뜻은 아니다 | 문맥별 출력 인코딩·파라미터화·권한 확인 |

이 구분은 보안 장치를 많이 나열하기 위한 것이 아니다. 각 조치가 어떤 표현을 받아 어떤 속성을 보장하는지 남기면, 실패가 생겼을 때 “검증이 없었는지”, “검증 뒤 재해석됐는지”, “권한·출력 경계가 다른지”를 분리해 고칠 수 있다.

## 결론

입력 검증은 문자열을 세척하는 작업이 아니라, 데이터가 다음 해석자에게 넘어가기 전에 어떤 계약을 만족해야 하는지 확인하는 일이다. Unicode의 디코드·정규화·동형이의 문자와 C의 타입 폭·정수 연산·버퍼 크기는 모두 이 계약이 끊길 수 있는 위치다.

따라서 신뢰 가능한 결과를 제공하는 시스템은 입력의 원 표현, 정준 표현, 파생된 수치, 소비자와 권한 결정을 분리해 기록한다. 같은 데이터를 어디에서 어떤 규칙으로 다시 해석하는지까지 추적할 때, 보안 경계는 단순 필터가 아니라 검토 가능한 시스템 계약이 된다.
