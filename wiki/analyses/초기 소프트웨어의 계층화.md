## 핵심 주장

[[초기 소프트웨어의 계층화]]는 소프트웨어가 단순한 명령 목록이 아니라 여러 수준의 표현과 실행 절차가 포개진 구조로 형성되었다는 분석이다. [[EDSAC]]은 이 흐름을 잘 보여준다. 종이 테이프 입력, [[Initial Orders]], [[디버깅]] 루틴, [[서브루틴]], [[라이브러리]], [[마이크로프로그래밍]], [[프로그래밍 언어]]가 각각 다른 층위를 이룬다.

## 계층별 의미

| 층위 | 초기 사례 | 현대적 대응 |
|---|---|---|
| 시작 절차 | Initial Orders | 펌웨어, 부트로더, [[로더]] |
| 번역 계층 | 기능 문자와 주소 해석 | [[어셈블러]], 컴파일러, 인터프리터 |
| 관찰 계층 | 트레이스·[[포스트모템 루틴]]·체크포인트 루틴 | 로그, 디버거, 분산 추적 |
| 재사용 계층 | 폐쇄형 서브루틴, [[Wheeler Jump]] | 함수, 모듈, 패키지 |
| 공유 자산 | [[라이브러리 카탈로그]] | 표준 라이브러리, 패키지 레지스트리 |
| 인터페이스 문서화 | 루틴의 동작·사용법·비용 명세 | [[API]] 문서, 레퍼런스 |
| 배치·결합 계층 | Initial Orders 2의 `θ` | [[재배치]], [[링커]], 동적 로더 |
| 호출 상태 계층 | 복귀 점프, 호출 스택 | [[복귀 주소]], [[스택]], [[스택 프레임]] |
| 제어 구현 | 마이크로프로그래밍 | 마이크로코드, 펌웨어, ISA 구현 |
| 표현 체계 | Fortran, FLOW-MATIC | 고급 언어, DSL, API |

## 분석

이 계층화의 핵심은 인간의 의도와 기계 실행 사이의 거리를 줄이는 것이다. 사용자는 낮은 수준의 비트와 회로를 직접 다루기보다, 점점 더 높은 수준의 표현을 사용한다. 그 표현은 다시 더 낮은 수준의 절차가 해석하고 실행한다. 이 구조는 초기 컴퓨터에서도 이미 드러났고, 현대 소프트웨어 스택에서도 반복된다.

계층화는 편리함만을 위한 장치가 아니다. 각 층은 오류를 관찰하고, 복잡성을 나누고, 재사용 가능한 단위를 만들고, 공동체가 공유할 수 있는 규칙을 형성한다. 따라서 소프트웨어의 탄생은 코드를 쓰기 시작한 사건이 아니라, 사고를 여러 층의 실행 가능한 구조로 조직하기 시작한 과정으로 이해된다.

새 소스들은 이 계층화에 배치와 호출 상태라는 두 층을 더 선명하게 추가한다. [[Initial Orders]] 2의 `θ`는 코드가 어디에 놓일지 나중에 결정하는 [[재배치]] 문제를 드러내고, [[Wheeler Jump]]와 [[스택]]의 대비는 "호출 후 어디로 돌아가는가"라는 제어 흐름 문제가 독립적인 소프트웨어·하드웨어 설계 주제로 성장했음을 보여준다.

[[The Preparation of Programs for an Electronic Digital Computer]]는 이 계층화에 인터페이스 문서화와 작성 자동화라는 층을 더한다. EDSAC의 라이브러리 카탈로그는 루틴의 내부 구현이 아니라 사용자가 알아야 할 명세를 중심에 두었고, [[자동 프로그래밍]]은 주소 보정과 서브루틴 조립 같은 반복 작업을 변환 루틴으로 넘기려 했다. 이 흐름은 [[서브루틴 라이브러리에서 API로]] 이어진다.
