## 핵심 주장

[[초기 소프트웨어의 계층화]]는 소프트웨어가 단순한 명령 목록이 아니라 입력, 적재, 실행, 관찰, 재사용처럼 서로 다른 책임을 가진 절차들로 분화되었다는 분석이다. 다만 한 시점의 실행 구조와 수십 년에 걸친 역사적 확장을 섞어서는 안 된다. [[EDSAC]] I에서 실제로 함께 작동한 계층과, 이후 [[마이크로프로그래밍]], [[프로그래밍 언어]], 운영체제로 확장된 계보를 나누어 살펴봐야 한다.

## EDSAC I 내부 실행 계층

| 층위 | EDSAC I의 사례 | 역할 |
|---|---|---|
| 입력 표현 | 종이 테이프의 기능 문자와 주소 표기 | 사람이 준비한 명령을 입력 매체에 기록 |
| 시작·번역 | [[Initial Orders]] | 표기를 기계 명령으로 조립하고 메모리에 적재한 뒤 실행을 넘김 |
| 배치·결합 | Initial Orders 2의 `θ` | 상대 주소를 적재 위치에 맞춰 보정하고 루틴을 결합 |
| 실행 | 주 프로그램과 [[서브루틴]] | 문제별 계산 절차를 수행 |
| 호출·복귀 | [[Wheeler Jump]]와 [[복귀 주소]] | 재사용 루틴을 호출하고 원래 실행 지점으로 복귀 |
| 관찰·진단 | 트레이스·[[포스트모템 루틴]]·체크포인트 루틴 | 실행 상태를 드러내고 오류를 국소화 |
| 공유 자산 | 종이 테이프 루틴과 [[라이브러리 카탈로그]] | 검증된 절차와 사용 명세를 공동으로 재사용 |

이 표의 항목은 오늘날의 부트로더, [[로더]], [[어셈블러]], [[링커]], 디버거, 함수, 패키지와 닮은 책임을 보인다. 그러나 EDSAC I의 작은 루틴과 물리적 테이프 환경을 현대 도구와 동일시해서는 안 된다. 당시에는 여러 책임이 Initial Orders와 작업 관행 안에 압축되어 있었고, 이후에야 독립된 도구와 인터페이스로 분화되었다.

## 후대 역사 연표

| 시기·계보 | 대표 사례 | 계층화의 확장 |
|---|---|---|
| EDSAC I 프로그래밍 환경 | Initial Orders, 진단 루틴, 서브루틴 라이브러리 | 입력·적재·관찰·재사용의 분화 |
| EDSAC 경험의 문서화 | [[The Preparation of Programs for an Electronic Digital Computer]] | 인터페이스 명세, 비용 정보, [[자동 프로그래밍]] |
| Wilkes의 후속 제어부 연구 | 마이크로프로그래밍과 EDSAC II | 기계 명령 아래의 CPU 제어 구현을 규칙적 절차로 조직 |
| 1950년대 고급 언어 | Fortran과 그 컴파일러 | 소스 표현, 번역, [[컴파일러 최적화]]의 분화 |
| 1960–1970년대 설계 방법 | 컴포넌트, [[정보 은닉]], [[구조적 프로그래밍]] | 재사용 단위, 모듈 경계, 제어 구조를 체계화 |
| 1970년대 이후 시스템 계보 | Unix, C, 도구 조합과 이식 | 운영체제 인터페이스와 언어·도구 생태계로 확장 |

> [!NOTE] 계층과 연표의 구분
> 마이크로프로그래밍은 EDSAC I의 응용 소프트웨어 계층이 아니다. Wilkes가 뒤이어 제안하고 EDSAC II의 제어부에 구현한 별도 하드웨어 제어 계보다. Fortran과 Unix 역시 EDSAC I 안에서 동시에 실행된 층이 아니라, 초기의 책임 분화가 후대에 확장된 사례다.

## 분석

계층화의 핵심은 인간의 의도와 기계 실행 사이의 책임을 분리하는 것이다. 사용자는 낮은 수준의 비트와 회로를 매번 직접 다루지 않고 더 높은 수준의 표현을 사용하며, 다른 절차가 그 표현을 적재·번역·실행한다. 이 구조는 초기 컴퓨터에서 부분적으로 드러났고, 이후 독립된 도구와 시스템으로 확장되었다.

계층화는 편리함만을 위한 장치가 아니다. 각 층은 오류를 관찰하고, 복잡성을 나누고, 재사용 가능한 단위를 만들고, 공동체가 공유할 수 있는 규칙을 형성한다. 따라서 소프트웨어의 탄생은 코드를 쓰기 시작한 사건이 아니라, 사고를 여러 층의 실행 가능한 구조로 조직하기 시작한 과정으로 이해된다.

EDSAC I 내부에서 [[Initial Orders]] 2의 `θ`는 코드가 어디에 놓일지 적재 시점에 결정하는 [[재배치]] 문제를 드러낸다. [[Wheeler Jump]]는 "호출 후 어디로 돌아가는가"를 명령 수정으로 해결했다. 후대의 [[스택]]과 [[스택 프레임]]은 같은 호출 상태 문제를 더 일반적인 중첩 호출과 지역 상태 관리로 확장했지만, EDSAC I의 Wheeler Jump 자체가 현대적 호출 스택을 사용한 것은 아니다.

[[The Preparation of Programs for an Electronic Digital Computer]]는 이 계층화에 인터페이스 문서화와 작성 자동화라는 층을 더한다. EDSAC의 라이브러리 카탈로그는 루틴의 내부 구현이 아니라 사용자가 알아야 할 명세를 중심에 두었고, [[자동 프로그래밍]]은 주소 보정과 서브루틴 조립 같은 반복 작업을 변환 루틴으로 넘기려 했다. 이 흐름은 [[서브루틴 라이브러리에서 API로]] 이어진다.

## 역사적 확장

마이크로프로그래밍은 EDSAC I의 소프트웨어 스택을 한 층 더 쌓은 것이 아니라, 같은 구조화 원리를 CPU 제어부에 적용한 후속 계보다. EDSAC II는 이 방식을 제어 구현에 사용했으며, 기계어 명령처럼 하나의 실행 단위로 보이는 동작도 더 낮은 제어 절차로 구성될 수 있음을 보여주었다.

[[Mass Produced Software Components]], [[On the Criteria To Be Used in Decomposing Systems into Modules]], [[Software Reuse]]는 재사용 계층을 더 두껍게 만든다. 재사용은 루틴을 보관하는 데서 끝나지 않고, 컴포넌트 제품군, 카탈로그, 정보 은닉, 검색 가능한 추상화, 특수화와 통합 절차를 요구한다. 이 흐름은 [[소프트웨어 재사용의 역사]]에서 별도로 분석한다.

[[A Case against the GO TO Statement]]와 [[Structured Programming]]은 후대에 제어 구조화를 독립된 설계 문제로 만들었다. 초기 소프트웨어가 "어디로 점프하고 어디로 돌아올 것인가"를 해결했다면, 구조적 프로그래밍은 "그 실행 경로를 사람이 어떤 구조로 이해할 것인가"를 묻는다. 이 전환은 [[goto와 점프에서 구조적 프로그래밍으로]]에서 별도로 분석한다.

[[The FORTRAN Automatic Coding System]]은 1950년대에 컴파일러와 최적화를 독립된 도구 층으로 발전시켰다. Fortran 번역기는 사람이 쓴 [[소스 프로그램]]을 [[목적 프로그램]]으로 바꾸면서, 단순 명령 변환을 넘어 반복 구조와 인덱스 레지스터 사용을 분석했다. 이 흐름은 [[Fortran과 컴파일러]]에서 별도로 분석한다.

[[The UNIX Time-Sharing System]]과 [[The Development of the C Language]]는 1970년대 이후 계층화가 운영체제와 도구 생태계로 확장되는 장면을 보여준다. Unix는 파일, 프로세스, 파이프, 셸을 일관된 인터페이스로 묶었고, C는 이 구조를 어셈블리보다 높은 수준에서 구현하고 다른 기계로 옮길 수 있게 했다. 이 흐름은 [[Unix와 C]]에서 별도로 분석한다.
