## 문제 제기

더 빠른 하드웨어와 더 나은 알고리즘은 모두 실행 시간을 줄일 수 있지만, 같은 종류의 개선은 아니다. 하드웨어가 같은 프로그램을 `c`배 빠르게 실행하면 같은 입력에 필요한 시간이 대체로 상수배 줄어든다. 알고리즘이 `T(n)`의 증가율을 바꾸면, 입력 `n`이 커질수록 개선 비율 자체가 달라질 수 있다.

[[On the Computational Complexity of Algorithms]]은 계산 가능성과 자원 증가율을 분리했고, [[The Complexity of Theorem-Proving Procedures]]과 [[Reducibility Among Combinatorial Problems]]은 문제들 사이의 조건부 난이도 관계를 다뤘다. [[There's Plenty of Room at the Top]]은 미세화 둔화 이후 성능 향상이 소프트웨어·알고리즘·아키텍처의 공동 개선에서 나올 수 있다고 보지만, 그 개선이 자동적·균일하지 않다고 경고한다.

## 1. 상수배와 증가율은 다른 질문이다

단순화해 기존 시간이 `T_old(n)`, 새 시간이 `T_new(n)`이라고 하자.

- 같은 알고리즘을 `c`배 빠른 하드웨어에서 실행하면 이상화한 시간은 `T_old(n) / c`다.
- 알고리즘을 바꾸면 비교 비율은 `T_old(n) / T_new(n)`이고, 이 값은 `n`에 따라 커지거나 작아질 수 있다.

예를 들어 `n³`에서 `n²`로 바꾸는 개선은 입력이 커질수록 비율이 커지지만, 작은 입력에서는 변환·초기화·메모리 접근의 상수가 더 큰 새 알고리즘이 늦을 수 있다. 반대로 하드웨어 가속은 넓은 작업에 즉시 도움이 될 수 있지만, 입력 증가에 따른 차수를 바꾸지는 않는다.

## 2. 환원은 실행 시간 표가 아니다

[[다항 시간 환원]]은 문제 `A`가 문제 `B`의 해법으로 풀릴 수 있음을 보인다. 이것은 `B`의 다항 시간 해법이 `A`의 다항 시간 해법을 준다는 조건부 관계이지, 두 문제의 실제 실행 시간이 비슷하다는 뜻이 아니다. 변환 자체의 상수, 만들어진 인스턴스의 크기, 메모리 배치와 구현은 별도로 측정해야 한다.

[[NP-완전]]도 마찬가지다. 최악 사례의 분류는 어떤 문제를 풀기 위해 무엇을 가정해야 하는지 알려 주지만, 특정 규모의 인스턴스가 어려운지, 휴리스틱·근사·문제 구조화가 유용한지, 어느 하드웨어에서 빨리 실행되는지를 한 번에 결정하지 않는다.

## 3. 실제 성능은 알고리즘과 시스템의 합성이다

| 개선 수단 | 주로 바꾸는 것 | 반드시 확인할 조건 |
|---|---|---|
| 알고리즘·문제 공식화 | 연산 수·메모리 사용량의 증가율, 허용하는 결과 형태 | 정확성·근사 품질, 입력 구조, 변환 비용, 교차점 |
| 데이터 구조·구현 | 상수, 지역성, 병렬성, 할당·동기화 비용 | 캐시·메모리·네트워크, 컴파일러, 라이브러리 |
| 하드웨어·아키텍처 | 연산·대역폭·병렬 자원과 에너지 제약 | 작업의 연산 집약도, 데이터 이동, 이용률, 전력 |

[[Roofline An Insightful Visual Performance Model]]과 [[Hitting the Memory Wall]]이 보여 주듯, 연산 수가 줄어도 데이터 이동이 병목이면 기대한 시간 단축이 작을 수 있다. 반대로 알고리즘이 같은 연산 수라도 재사용과 병렬성을 바꾸면 하드웨어 자원을 더 잘 쓸 수 있다. 그래서 “알고리즘 대 하드웨어”는 승자를 고르는 질문이 아니라, 작업·규모·결과 계약·시스템 경계에서 어느 비용이 지배적인지 찾는 질문이다.

## 4. 비교를 위한 최소 실험 설계

1. 같은 문제 정의와 정확성·근사 품질에서 비교할 알고리즘을 정한다.
2. 입력 크기와 구조를 여러 구간으로 나눠 교차점과 증가 추세를 기록한다.
3. 동일하거나 명시적으로 다른 하드웨어·컴파일러·라이브러리·병렬 자원을 기록한다.
4. 시간뿐 아니라 메모리, 데이터 이동, 에너지와 실패·재시도 조건을 측정한다.
5. 최악 사례 이론, 평균·실측 결과, 특정 시스템의 벤치마크를 같은 종류의 주장으로 합치지 않는다.

## 결론

더 빠른 하드웨어는 더 나은 알고리즘을 자동으로 대신하지 않고, 더 나은 알고리즘도 구현·데이터 이동·정확성 비용을 지우지 않는다. 하드웨어는 같은 계산의 상수와 병렬 자원을 바꾸고, 알고리즘은 문제를 푸는 자원의 증가율과 때로는 결과 계약을 바꾼다. 좋은 비교는 둘을 한 숫자로 환원하기보다, 어떤 입력 규모와 결과 조건에서 어떤 병목을 얼마나 줄였는지 밝힌다.
