## 개요

병렬 컴퓨터에 프로세서를 더했을 때 “얼마나 빨라졌는가”라는 질문은 불완전하다. 같은 입력을 더 짧은 시간에 처리했는지, 같은 시간에 더 큰 입력이나 더 높은 해상도를 처리했는지에 따라 가속의 의미가 달라지기 때문이다. [[Validity of the Single Processor Approach to Achieving Large Scale Computing Capabilities]]에서 유래한 암달의 법칙과 [[Reevaluating Amdahl's Law]]의 확대 크기 가속은 이 두 질문을 대표한다.

두 모델은 병렬 컴퓨팅의 가능성을 두고 한쪽이 맞고 다른 쪽이 틀린 경쟁 법칙이 아니다. 서로 다른 값을 고정한 반사실적 비교다.

> 암달은 “같은 일을 얼마나 빨리 끝내는가”를 묻고, Gustafson은 “같은 시간에 얼마나 큰 일을 하는가”를 묻는다.

## 같은 프로세서 수, 다른 비교 대상

| 관점 | 고정하는 조건 | 자원이 늘 때 변하는 것 | 가속비가 뜻하는 것 |
|---|---|---|---|
| 암달의 고정 크기 | 전체 작업량·입력 | 실행 시간 | 같은 문제의 시간 단축 비율 |
| Gustafson의 확대 크기 | 병렬 실행 시간 | 병렬 작업량·문제 크기 | 커진 문제의 직렬 가상 시간 대비 비율 |

암달 모델에서 순차 시간 `s`와 병렬 가능 시간 `p`는 프로세서 한 개로 같은 고정 작업을 수행한 시간을 기준으로 하며 `s + p = 1`이다.

`S_fixed(N) = 1 / (s + p/N)`

Gustafson 모델의 `s'`와 `p'`는 `N`개 프로세서에서 확대된 문제를 실행한 시간을 기준으로 하며 `s' + p' = 1`이다.

`S_scaled(N) = s' + Np' = N - (N - 1)s'`

두 식에서 순차 비율의 분모와 문제 크기가 다르다. 같은 문자 하나로 치환해 두 모델의 예측을 정면 비교하면 질문 자체가 바뀌었다는 사실을 놓친다.

## 암달이 정확히 경고한 것

Amdahl의 1967년 논지는 병렬성이 쓸모없다는 수학적 선언이 아니었다. 당시 생산 프로그램에서 순차 데이터 관리와 불규칙한 문제 구조가 상당한 비용을 차지하므로, 병렬 연산률만 높이고 순차 처리율과 데이터 이동을 개선하지 않으면 전체 가속이 제한된다는 경고였다.

고정된 입력을 정해진 기한보다 빨리 끝내야 하는 작업에는 이 질문이 그대로 중요하다. 같은 시뮬레이션의 응답 시간을 줄이거나, 하나의 요청 지연을 낮추고, 이미 정해진 데이터셋을 더 빨리 처리하는 경우 문제를 키워서 시간 감소의 실패를 숨길 수 없다.

암달식은 또한 자원이 늘수록 병렬 부분의 시간이 줄어 순차·통신·동기화 비용의 상대적 비중이 커지는 이유를 보여준다. 병렬화되지 않는 1%가 있다는 문장보다, 자원 수가 바뀔 때 실제로 줄지 않는 시간이 무엇인지 측정하는 일이 더 중요하다.

## Gustafson이 바꾼 질문

Gustafson은 과학 계산 사용자가 더 강한 기계를 얻으면 흔히 격자를 촘촘하게 하고 시간 단계를 늘여 더 큰 문제를 비슷한 시간에 푼다고 관찰했다. 1024프로세서 시스템에서 세 응용의 거의 선형에 가까운 확대 크기 가속을 보고하며, 암달의 고정 문제 가정만으로 대규모 병렬 컴퓨팅의 가치를 판단하는 것은 부적절하다고 주장했다.

이 관점은 병렬 컴퓨팅 능력을 지연 시간뿐 아니라 **해결 가능한 문제의 규모와 해상도**로 확장한다. 같은 밤 동안 더 정밀한 기후 모형을 계산하거나, 같은 실험 시간에 더 많은 후보를 탐색할 수 있다면 컴퓨터가 제공하는 실질적 능력은 증가한다. 다만 이것은 동일한 입력의 가속이 아니라 더 큰 계산을 수행한 결과다.

## 확대 크기 가속도 공짜가 아니다

Gustafson식은 순차 시간이 거의 일정하고 병렬 작업량이 프로세서 수에 비례해 늘어난다는 첫 근사를 둔다. 실제로 문제를 키우면 통신 경계, 전역 합산, 메모리 사용량, 입출력과 초기화도 커질 수 있다. 더 큰 문제가 항상 더 정확하거나 더 가치 있는 결과를 주는 것도 아니다.

따라서 높은 확대 크기 가속을 보고할 때는 다음 질문이 필요하다.

1. 문제 크기를 어떤 변수로 얼마나 늘렸는가.
2. 늘어난 작업이 해상도·정확도·탐색 범위에 어떤 가치를 더했는가.
3. 병렬 실행 시간은 실제로 일정했는가.
4. 직렬 기준 시간은 실측값인지 확대된 작업량의 가상 추정값인지.
5. 통신·동기화·입출력과 데이터 준비를 포함했는가.

Gustafson의 1021배·1020배·1016배 결과는 특정 과학 응용과 1024프로세서 하이퍼큐브에서 보고된 사례다. 이를 모든 병렬 프로그램이 거의 선형으로 확장한다는 보장으로 읽으면 논문의 첫 근사를 보편 법칙으로 과장하게 된다.

## 둘을 함께 측정해야 하는 이유

[[병렬 확장성]]을 제대로 이해하려면 고정 크기와 확대 크기 결과를 모두 보고하는 편이 좋다.

- 고정 크기 결과는 동일 작업의 지연 시간과 자원 추가의 한계 효용을 보여준다.
- 확대 크기 결과는 자원이 늘 때 처리 가능한 문제 규모와 처리량이 어떻게 증가하는지 보여준다.
- 병렬 효율은 얻은 가속을 투입한 자원 수와 비교한다.
- 작업량 함수와 출력 품질은 “더 큰 문제”가 무엇을 뜻하는지 설명한다.

예를 들어 고정 크기 효율이 빠르게 떨어지는 프로그램도 프로세서당 충분한 일을 유지하도록 문제를 키우면 좋은 확대 크기 효율을 보일 수 있다. 이것은 모순이 아니라, 작은 고정 문제에서는 통신과 순차 비용이 지배하고 큰 문제에서는 병렬 작업이 그 비용을 상쇄한다는 뜻이다.

## 하드웨어에서 공동 설계로

2006년 [[The Landscape of Parallel Computing Research - A View from Berkeley]]는 멀티코어 전환 뒤 병렬 성능이 코어 수만으로 나오지 않으며 응용, 알고리즘, 프로그래밍 모델, 컴파일러, 운영체제와 구조를 함께 설계해야 한다고 강조했다. 암달과 Gustafson의 두 기준은 이 공동 설계가 어떤 목표를 향하는지 구분한다.

- 지연 시간을 줄이는 목표라면 순차 경로와 통신을 끝까지 줄여야 한다.
- 문제 규모를 늘리는 목표라면 병렬 작업 생성, 메모리 용량과 데이터 이동도 함께 확장해야 한다.
- 두 목표가 섞인 서비스라면 고정 입력 지연과 처리량·배치 규모를 별도 지표로 보고해야 한다.

병렬 소프트웨어는 단순히 많은 코어를 “사용”하는 것이 아니라, 사용 목적에 맞는 작업량 변화와 비용 구조를 표현해야 한다.

## 종합

병렬 컴퓨팅이 시간을 줄이는가 문제를 키우는가에 대한 답은 “둘 다 가능하지만 같은 성과는 아니다”이다. 암달은 고정된 일을 더 빨리 끝내는 능력의 상한을 드러내고, Gustafson은 더 큰 일을 같은 시간에 끝내는 능력을 드러낸다.

따라서 병렬 [[컴퓨팅 능력이란 무엇인가|컴퓨팅 능력]]을 보고할 때는 프로세서 수와 가속비만 제시하지 말고 다음 다섯 값을 함께 기록해야 한다.

`문제 크기 W(N), 실행 시간 T(N), 순차·통신 비용, 자원 수 N, 결과의 품질·가치`

이 조건이 있어야 속도의 향상과 규모의 향상을 구분하면서도, 둘을 병렬 컴퓨팅이 제공하는 서로 다른 능력으로 함께 평가할 수 있다.
