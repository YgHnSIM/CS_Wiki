## 개요

[[컴퓨팅 능력의 발달사]]는 “컴퓨터가 얼마나 빨라졌는가”라는 단일 그래프로 환원되지 않는다. 시대마다 능력의 단위와 병목, 유효한 측정 지표가 달라졌기 때문이다. 기계 이전에는 계산을 사람·절차·매체에 나누고 [[검산]]·출판하는 조직적 처리량이 중요했다. 이후 계산 절차의 원리적 경계와 새 작업을 기계에 부여하는 비용, 집적도와 클럭, 수치 결과 계약, 메모리와 병렬성, 데이터센터 규모의 처리량, 에너지 효율과 도메인 특화 성능으로 중심이 이동했다.

이 역사는 이전 한계를 완전히 제거한 과정이라기보다 **병목이 다른 층위로 이동한 과정**이다. 더 많은 트랜지스터는 연산 자원을 늘렸지만 데이터 이동과 전력을 부각했고, 더 많은 코어와 서버는 통신·동기화·가용성 문제를 키웠으며, 특화 가속기는 높은 효율과 함께 적용 범위와 소프트웨어 매핑이라는 새 제약을 만들었다.

## 0. 기계 이전의 계산 인프라

전자식 컴퓨터 이전에도 계산은 개인의 산술을 넘어선 생산 체계였다. [[인간 계산자]]는 수학적 방법을 기본 연산과 작업표로 나누어 수행했고, 중간 결과와 독립 검산을 거쳐 [[수학 표]]나 보고서를 만들었다. 이때 능력은 한 사람의 계산 속도뿐 아니라 희소한 수학적 판단을 반복 노동과 어떻게 배치하고, 오류를 얼마나 일찍 발견하며, 결과를 얼마나 널리 배포할 수 있는가로 측정할 수 있다.

드 프로니의 1790년대 표 프로젝트는 계산을 1801년에 마쳤지만 방대한 결과의 출판 비용을 감당하지 못했다. 이는 값의 계산과 사회적 이용 가능성이 서로 다른 병목임을 보여준다. 1938–1946년 Mathematical Tables Project는 최대 약 450명의 계산자, 알고리즘과 작업표, 역할별 전달과 검산을 결합해 37권의 표와 전시 계산을 생산했다. 여기서는 조직 자체가 계산 절차의 실행 환경으로 기능했다.

[[인간 계산자와 계산 공장]]은 이 전사를 배비지의 기계식 출력 구상, ENIAC 운영과 SEAC의 저장 프로그램 수치 계산에 연결한다. 이 연결은 인간 작업표가 곧바로 프로그램이 되었다는 직접 계보가 아니다. 장치 성능만으로는 설명되지 않는 작업 분해·검증·출판의 경계가 자동화 과정에서 다시 배치되었다는 비교다.

## 1. 계산 가능성에서 복잡도로

[[On Computable Numbers with an Application to the Entscheidungsproblem]]은 유한한 규칙을 수행하는 추상 기계로 “기계적으로 계산 가능하다”는 말을 형식화했다. 이 단계의 핵심은 기계 속도가 아니라 어떤 문제에 일반적인 계산 절차가 존재하는가였다. 더 빠른 하드웨어도 계산 불가능한 문제를 일반적으로 해결하게 만들지는 못한다.

[[On the Computational Complexity of Algorithms]]는 계산 가능한 문제들 사이에서도 입력 크기에 따른 시간 자원 요구가 본질적으로 다름을 분리했다. 하드웨어가 일정 배수로 빨라지는 것은 실행 시간을 줄일 수 있지만, 알고리즘의 점근적 증가율을 없애지 않는다. 이로써 컴퓨팅 능력은 “할 수 있음”과 “실용적인 자원 안에서 할 수 있음”의 두 층위로 나뉘었다.

## 2. 범용성에서 프로그래밍 가능성으로

이 전환의 연대는 계산 이론보다 앞서 시작하지만, 개념적으로는 “가능한 계산을 실제 기계에 어떻게 부여하는가”라는 다음 질문에 답한다. [[컴퓨팅의 기원 - 배비지와 러브레이스]]의 해석 기관 구상은 연산과 수 저장, 명령 순서와 반복을 분리해 하나의 장치가 여러 계산을 표현할 가능성을 보여주었다. 그러나 물리적 장치의 범용성만으로 새 프로그램을 쉽게 작성하고 축적할 수 있었던 것은 아니다.

1945년 [[First Draft of a Report on the EDVAC]]에 나타난 저장 프로그램 구조는 명령을 내부 기억장치에 데이터와 함께 두어 작업 변경을 재배선·외부 설정에서 내부 기억장치의 기호 변경으로 옮겼다. 이 문서는 집단 설계의 역사와 단독 저자 표기라는 맥락을 함께 읽어야 하며, 설계 문서의 선행성을 최초 실행이나 실용 운용의 최초성과 합쳐서는 안 된다.

[[The Preparation of Programs for an Electronic Digital Computer]]은 EDSAC에서 서브루틴, 카탈로그와 호출 규약을 프로그램 작성의 공용 기반으로 만들었다. [[The FORTRAN Automatic Coding System]]은 수학에 가까운 표기를 효율적인 IBM 704 목적 프로그램으로 번역하는 일을 컴파일러의 과제로 삼았다. 이어 [[The UNIX Time-Sharing System]]은 파일·프로세스·셸과 도구 조합을 공유 실행 환경으로 묶었다. 이 단계들에서 능력은 명령 실행 속도뿐 아니라 새 계산을 표현하고 기존 코드를 재사용하며 다른 환경으로 옮기는 비용까지 포함하게 되었다.

따라서 범용성은 한 번 얻고 끝나는 하드웨어 속성이 아니다. 저장 프로그램, 라이브러리, 언어·컴파일러와 운영체제가 추상화와 호환성의 경계를 넓힐 때 실제 [[프로그래밍 가능성]]이 된다. [[범용성은 어떻게 컴퓨팅 능력이 되었는가]]는 이 흐름을 “기계에 새 계산을 부여하는 변경 비용의 감소”로 종합한다.

## 3. 집적도와 소자 축소의 결합

1965년 [[Cramming More Components onto Integrated Circuits]]는 최소 비용 지점의 집적 회로 부품 수가 빠르게 증가하는 추세를 관찰했다. 훗날 [[무어의 법칙]]이라 불린 이 흐름은 더 많은 기능을 한 칩에 통합하고, 외부 연결을 줄이며, 캐시와 복잡한 제어 구조를 추가할 여지를 제공했다.

1974년 [[Design of Ion-Implanted MOSFET's with Very Small Physical Dimensions]]의 [[Dennard 스케일링]]은 치수와 전압을 함께 줄일 때 회로 지연과 회로당 전력을 낮추면서 이상적인 전력 밀도를 유지할 수 있음을 보였다. 집적도 증가와 일정 전계 축소가 함께 작동한 기간에는 더 많은 트랜지스터, 더 높은 클럭과 더 복잡한 단일 코어가 동시에 성능을 높일 수 있었다.

그러나 두 개념은 동일하지 않다. 집적 가능한 트랜지스터 수가 계속 늘더라도 전압을 같은 비율로 낮추지 못하면 전력과 발열 때문에 모든 소자를 높은 빈도로 사용할 수 없다. 트랜지스터 수의 증가가 실제 작업 성능으로 자동 전환된다고 보는 해석은 이 차이를 놓친다.

## 4. 잠재 성능을 프로그램 성능으로 바꾸는 구조

트랜지스터와 회로 속도의 증가는 더 큰 성능 잠재력을 제공했지만, 프로그램은 명령과 데이터를 공급받고 실제 회로의 타이밍 안에서 실행되어야 했다. 1965년 [[Slave Memories and Dynamic Storage Allocation]]은 빠른 소용량 기억장치가 최근 요구된 단어를 축적해 느린 주기억장치 접근을 줄이는 구조를 제안했다. 이 [[캐시 메모리]]는 속도가 다른 기억장치를 계층으로 결합했지만, 효과는 프로그램의 재참조 패턴과 주소 사상·교체·쓰기 정책에 달려 있었다.

1982년 [[Design and Implementation of RISC I]]은 초기 단일 칩 VLSI의 면적과 배선 제약 아래 흔한 명령 경로를 단순하고 규칙적으로 만들려 했다. [[축소 명령어 집합 컴퓨터|RISC I]]은 레지스터 간 연산, `LOAD`/`STORE`, 고정 길이 명령과 작은 제어부를 컴파일러 스케줄링과 결합했다. 이는 명령어를 무조건 줄이는 전략보다 ISA·마이크로아키텍처·컴파일러·메모리 사이에 복잡성을 다시 배분한 설계였다.

두 연구는 하나의 직접 계보가 아니다. 하나는 느린 기억장치를 재사용으로 완충했고, 다른 하나는 흔한 실행 경로와 구현 비용을 재배치했다. 그러나 둘은 최고 부품 속도만으로 실제 프로그램 성능을 설명할 수 없다는 공통점을 갖는다. RISC I의 비교 성능은 400ns 사이클을 가정한 시뮬레이션이었고 실제 칩은 놓친 임계 경로 때문에 그 목표에 미치지 못했다. 반대로 단순한 명령어 형식은 데이터 접근을 줄이면서 코드 크기와 명령어 인출을 늘릴 수 있어, 보고서 자체가 완전한 효율을 위한 명령어 캐시를 요구했다.

이 전환은 컴퓨팅 능력을 `작업 부하 × 컴파일러 × ISA × 마이크로아키텍처 × 메모리 계층`의 합성 결과로 보게 한다. [[더 빠른 프로세서는 왜 더 빠른 프로그램을 보장하지 않는가]]는 이 시기의 구조적 선택을 잠재 성능과 실현 성능의 차이로 분석한다.

## 5. 연산 속도에 수치 의미를 붙인 표준화

부동소수점 연산기는 과학·공학 계산의 속도를 높였지만, 기계마다 형식·반올림과 예외 처리가 다르면 같은 소스 식이 같은 수치 연산을 뜻한다고 가정하기 어렵다. 1985년 IEEE 754는 이진 부동소수점의 형식과 기본 연산 의미를 공통화했고, 현행 판본은 [[IEEE 754-2019 Standard for Floating-Point Arithmetic]]이다. 표준의 역사적 의의는 실수 산술을 그대로 구현한 데 있지 않고, 유한 정밀도 연산을 구현 사이에서 추론할 수 있는 계약으로 만든 데 있다.

1991년 [[What Every Computer Scientist Should Know About Floating-Point Arithmetic]]은 표현 불가능한 값, 반올림 오차, 가드 숫자와 정확한 반올림, 상쇄를 하드웨어·컴파일러·운영체제의 공동 문제로 설명했다. 이 글은 1985년판의 해설이므로 현행 표준 전체의 근거는 아니지만, 빠른 연산률만으로 계산 결과를 설명할 수 없다는 점을 컴퓨터 시스템의 언어로 정리했다.

표준화도 프로그램 전체의 동일한 답을 자동으로 만들지는 않는다. 규정 연산의 결과는 입력값·연산 순서·목적 형식에 의존하며, 컴파일러 재배열과 병렬 합산은 그 순서를 바꿀 수 있다. 따라서 [[부동소수점 정확성]]은 비트 단위 재현성, 수치 허용 오차와 과업 검증을 분리한다. [[더 빠른 계산은 같은 답을 내는가]]는 성능이 결과 계약을 통과한 작업량이어야 한다는 관점에서 이 전환을 분석한다.

## 6. 데이터 이동과 병렬성의 장벽

프로세서의 연산 능력이 빠르게 높아지자 데이터 공급이 상대적으로 느린 문제가 전면에 나타났다. [[Hitting the Memory Wall]]은 프로세서와 DRAM 개선 속도의 격차가 커지면 실행 시간이 메모리 접근에 지배될 수 있다고 경고했다. [[메모리 장벽]]은 최고 연산률보다 캐시 지역성, 지연 시간, 대역폭과 데이터 배치가 실제 성능을 결정할 수 있음을 뜻한다.

병렬 자원을 늘리는 것도 무제한 해법은 아니다. [[Validity of the Single Processor Approach to Achieving Large Scale Computing Capabilities]]가 제시한 암달의 법칙은 순차 구간이 전체 가속의 상한을 정함을 보여준다. 실제 시스템에서는 통신, 동기화, 메모리 충돌과 작업 분배 비용이 추가된다.

1988년 [[Reevaluating Amdahl's Law]]은 병렬성의 목표를 다른 방향으로 확장했다. 암달의 고정 크기 모델이 같은 문제의 실행 시간을 줄이는 정도를 묻는다면, Gustafson의 확대 크기 모델은 실행 시간을 일정하게 두고 자원에 맞춰 문제 규모를 늘릴 때 같은 시간에 얼마나 많은 일을 수행하는지 묻는다. 두 모델은 승패 관계가 아니라 고정 조건이 다른 [[병렬 확장성]]의 두 기준이다.

이 구분은 컴퓨팅 능력의 역사에서 중요한 변화를 보여준다. 더 많은 프로세서는 기존 작업의 지연 시간을 줄일 뿐 아니라 더 촘촘한 격자, 더 긴 시뮬레이션과 더 넓은 탐색을 가능하게 했다. 그러나 확대된 작업의 직렬 가상 시간과 같은 입력의 실제 시간 단축을 하나의 가속비로 섞으면 안 된다. [[병렬 컴퓨팅은 시간을 줄이는가 문제를 키우는가]]는 속도와 문제 규모를 병렬 컴퓨팅이 제공하는 서로 다른 능력으로 분리한다.

전압 축소가 둔화하고 명령 수준 병렬성의 추가 이익이 줄면서 산업은 2005년 전후 멀티코어를 기본 경로로 채택했다. [[The Landscape of Parallel Computing Research - A View from Berkeley]]는 이 전환을 전력·메모리·명령 수준 병렬성의 결합된 한계로 설명하고, 응용·프로그래밍 모델·컴파일러·운영체제·구조의 공동 설계를 연구 의제로 제안했다.

이 시점부터 성능 향상은 소프트웨어에 더 많은 책임을 요구했다. 코어가 많다는 사실만으로 프로그램이 빨라지지 않으며, 병렬 작업을 찾아 분배하고 데이터 지역성과 동기화를 관리해야 한다. 컴퓨팅 능력은 하드웨어 사양이 아니라 하드웨어와 소프트웨어가 함께 실현하는 속성이라는 점이 더 분명해졌다.

## 7. 전력과 에너지가 성능의 경계가 되다

전력 장벽은 성능 향상이 멈췄다는 뜻이 아니라 성능을 얻는 조건이 달라졌다는 뜻이다. 장기적으로는 같은 에너지로 가능한 계산량이 크게 늘었지만, 한 칩과 실제 서비스에는 순간 전력·열·기저 전력이라는 서로 다른 제약이 남았다.

[[Implications of Historical Trends in the Electrical Efficiency of Computing]]은 1946–2009년의 이질적인 컴퓨터 80대를 최고 부하에서 비교해 계산/kWh가 평균 1.57년마다 두 배가 된 추세를 보고했다. 훗날 [[Koomey의 법칙]]이라 불린 이 결과는 에너지 예산으로 가능한 계산 규모가 장기간 확대됐음을 보여준다. 그러나 최고 부하의 역사 회귀이며, 실제 장비의 유휴·부분 부하나 사회 전체의 전력 소비를 직접 측정하지 않는다.

2007년 [[The Case for Energy-Proportional Computing]]은 같은 문제를 운용 시간의 축에서 보았다. 당시 5,000대가 넘는 서버의 관측에서 중간 활용률이 흔했지만, 대표적 서버는 거의 일을 하지 않을 때도 최대 전력의 약 절반을 소비했다. [[에너지 비례 컴퓨팅]]은 최고 부하 한 점보다 유휴부터 첨두까지 처리량에 맞춰 전력이 내려가는지를 설계 목표로 만들었다.

2011년 [[Dark Silicon and the End of Multicore Scaling]]은 전력 문제를 칩 면적의 축으로 옮겼다. 집적 가능한 코어가 늘어도 고정된 전력과 프로그램의 병렬성 때문에 일부 코어 면적을 동시에 유용하게 사용하지 못한다는 [[다크 실리콘]]을 모델링했다. 논문의 22nm·8nm 비율은 당시의 미래 예측이지 후대 칩의 보편적 실측값이 아니지만, 트랜지스터 보유량과 활성 가능량이 분리된다는 문제를 정량화했다.

세 자료는 서로 다른 것을 측정한다. Koomey는 **세대 간 최고 부하의 환산 계산량/에너지**, 에너지 비례성은 **한 시스템의 부하 곡선**, 다크 실리콘은 **한 칩의 순간 활성 자원**을 묻는다. 장기 효율이 좋아져도 높은 유휴 전력과 순간 전력 밀도는 남을 수 있고, 효율 향상보다 작업 수요가 빠르게 늘면 총에너지는 증가할 수 있다. [[전력 장벽은 성능 향상의 의미를 어떻게 바꾸었는가]]는 이 차이를 성능 평가의 세 축으로 분석한다.

## 8. 칩에서 창고 규모 컴퓨터로

인터넷 서비스는 병렬성의 단위를 코어와 칩에서 수천 대의 서버로 확장했다. [[The Datacenter as a Computer]]는 서버, 네트워크, 저장 장치, 전력과 냉각, 자원 관리 소프트웨어를 하나의 창고 규모 컴퓨터(warehouse-scale computer)로 보아야 한다고 설명한다.

이 규모에서는 개별 프로세서의 최고 성능보다 서비스 전체 처리량, [[꼬리 지연 시간]], 장애가 발생해도 유지되는 가용성, 전력과 총비용이 중요하다. 한 노드의 성능을 높이는 선택이 네트워크 병목이나 전력 예산 때문에 전체 서비스 능력을 높이지 못할 수도 있다. 측정 경계가 칩에서 시스템과 시설로 넓어진 것이다.

2013년 [[The Tail at Scale]]은 이 확장의 통계적 결과를 구체적인 팬아웃 사례로 정량화했다. 논문의 실제 서비스 표에서 루트가 관측한 임의 리프 하나의 99백분위 완료 시간은 10ms였지만, 루트가 모든 리프의 완료를 관측한 시간의 99백분위는 140ms였다. 드문 구성요소 지연은 많은 응답을 합성하는 순간 흔한 서비스 지연이 될 수 있으므로, 평균 속도보다 루트 관측 분포와 꼬리 내성이 능력의 일부가 되었다.

이 전환은 성능 최적화의 목표도 바꿨다. 복제 요청과 빠른 취소, 미세 파티션, 선택적 복제는 개별 부품을 항상 일정하게 만드는 대신 변동 속에서도 전체 요청을 예측 가능하게 만든다. 그러나 중복 작업·여유 용량·결과 생략은 에너지·비용·품질을 바꾸므로, [[평균 성능은 왜 서비스의 컴퓨팅 능력을 설명하지 못하는가]]에서 이 교환을 별도로 분석한다.

2000년대 초의 [[Why Do Internet Services Fail and What Can Be Done About It]]과 [[Recovery-Oriented Computing (ROC)]]은 이 경계가 정상 상태의 응답 속도에 그치지 않음을 드러냈다. 서비스 장애를 하드웨어·소프트웨어·네트워크·운영 오류와 함께 보고, 탐지 뒤 품질이 회복될 때까지의 시간을 관리 대상으로 삼았기 때문이다. [[The Byzantine Generals Problem]]은 더 앞선 이론 차원에서 일부 참여자가 모순된 정보를 보낼 때 일관된 결과에 필요한 가정을 정식화했다. 이 흐름에서 [[결함 허용]]과 [[가용성과 복구]]는 성능의 부가 기능이 아니라 결과 계약의 일부가 된다.

## 9. 도메인 특화와 공동 설계

전력과 데이터 이동이 지배적인 제약이 되자 사용 가능한 트랜지스터를 모든 작업에 공평하게 쓰기보다 특정 계산 패턴에 집중하는 [[도메인 특화 가속기]]가 중요해졌다. 특화 회로는 범용 제어 기능을 줄이고 필요한 연산기, 정밀도, 온칩 메모리와 데이터 경로에 면적과 전력을 배분한다.

[[In-Datacenter Performance Analysis of a Tensor Processing Unit]]의 1세대 TPU는 이 전환의 구체적인 사례다. 8비트 행렬 연산과 소프트웨어 관리 메모리, 예측 가능한 실행을 신경망 추론과 데이터센터의 꼬리 지연 요구에 맞췄다. 논문이 보고한 큰 속도·에너지 이점은 특화가 유효함을 보여주지만, 특정 하드웨어 세대와 작업·정밀도·비교 조건에 한정된 결과다.

특화는 범용성을 없애는 것이 아니라 시스템 안에서 역할을 나눈다. CPU는 제어와 다양한 작업을 맡고, GPU·TPU와 여러 전용 엔진은 반복적이고 병렬적인 부분을 처리한다. 성능은 연산을 어떤 장치에 배치하고 데이터를 얼마나 적게 이동시키는지에 달려 있으므로 컴파일러, 런타임과 프레임워크가 하드웨어만큼 중요해진다.

## 측정 지표도 함께 발전했다

하드웨어의 변화는 측정 방식의 변화를 요구했다. 클럭이나 트랜지스터 수는 사용할 수 있는 자원을 나타내지만, 작업이 그 자원을 활용하는 정도를 보여주지 않는다.

- [[SPEC CPU 2026 Overview]]는 표준 작업 부하의 실행 시간과 처리량을 구분하고 컴파일러·메모리를 포함한 시스템 구성을 평가한다.
- [[The Linpack Benchmark]]은 이론적 최고 `Rpeak`와 특정 선형대수 작업에서 달성한 `Rmax`를 분리한다.
- [[부동소수점 정확성]]은 연산률과 별도로 비트 재현성, 수치 오차와 결과 검증의 경계를 묻는다.
- [[Roofline An Insightful Visual Performance Model]]은 최고 연산 성능, 메모리 대역폭과 연산 집약도를 결합해 계산·메모리 병목을 구분한다.
- [[Power Measurement Tutorial for the Green500 List]]은 달성 성능을 같은 구간의 평균 전력으로 나눠 전력당 성능을 측정한다.
- [[Koomey의 법칙]]은 서로 다른 세대의 최고 부하 계산/J를 비교하되 유휴 전력과 총수요를 설명하지 않는다.
- [[에너지 비례 컴퓨팅]]은 유휴–첨두의 처리량과 전력 곡선을 함께 측정한다.
- [[다크 실리콘]]은 고정된 칩 전력에서 구현 자원과 동시 활성 자원의 간극을 묻는다.
- [[꼬리 지연 시간]]은 평균과 분리해 종단 요청의 높은 백분위와 기한 초과를 측정한다.
- [[가용성과 복구]]는 서비스 목표가 깨진 뒤 탐지·진단·복구와 사용자 영향이 이어지는 경계를 측정한다.

이 지표들은 서로 대체 관계가 아니다. 단일 스레드 지연, 대량 처리량, 병렬 가속, 메모리 효율과 에너지 효율은 다른 질문에 답한다. [[컴퓨팅 능력이란 무엇인가]]에서 정리하듯 유효한 비교는 작업 부하, 정확성, 시스템 경계, 실행 조건과 지표를 함께 명시해야 한다.

## 종합: 속도의 역사에서 병목 이동의 역사로

컴퓨팅 능력은 다음과 같은 반복 구조 속에서 발전했다.

1. 새로운 기술이 한 층위의 자원 한계를 완화한다.
2. 이전에는 덜 중요했던 병목이 전체 실행 시간과 비용에서 더 큰 비중을 차지한다.
3. 구조와 소프트웨어가 새 병목을 숨기거나 활용하는 방식으로 바뀐다.
4. 기존 지표가 설명하지 못하는 차이를 측정하기 위해 평가 경계와 지표가 확장된다.

따라서 발전을 “연산 횟수가 지수적으로 늘었다”로만 설명하면 메모리, 전력, 병렬성, 시스템 운영과 소프트웨어의 역할이 사라진다. 더 정확한 서술은 **원리적 계산 범위는 이론으로 구분되고, 실현 가능한 능력은 알고리즘·집적 기술·데이터 이동·병렬 구조·에너지·전체 시스템과 특화 소프트웨어의 공동 진화로 확대되었다**는 것이다.
