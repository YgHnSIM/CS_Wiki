## 개요

[[메모리 장벽]](memory wall)은 프로세서가 명령과 산술을 처리하는 속도에 비해 주 메모리에서 데이터를 가져오는 지연 시간과 대역폭이 충분히 개선되지 않아, 전체 실행 시간이 데이터 이동과 대기에 지배되는 현상이다. Wulf와 McKee의 [[Hitting the Memory Wall]]이 1995년에 이 표현을 널리 알려진 문제로 정식화했다.

## 지연 시간과 대역폭

메모리 성능은 하나의 숫자가 아니다.

- **지연 시간(latency)**은 요청한 데이터가 도착할 때까지 기다리는 시간이다.
- **대역폭(bandwidth)**은 일정 시간에 옮길 수 있는 데이터의 양이다.
- **프로세서 사이클 기준 비용**은 같은 나노초의 지연도 더 빠른 프로세서에서는 더 많은 작업 기회를 잃게 함을 보여준다.

독립적인 메모리 요청을 겹치면 대역폭을 활용할 수 있지만, 다음 주소가 이전 결과에 의존하는 접근은 긴 지연 시간을 숨기기 어렵다. 따라서 같은 하드웨어에서도 순차 접근, 불규칙 접근과 포인터 추적은 서로 다른 병목을 보인다.

## 완화 방법과 한계

캐시는 자주 쓰는 데이터를 프로세서 가까이에 두고, 프리페칭은 필요한 데이터를 미리 가져온다. 데이터 배치와 블로킹은 지역성을 높이고, 여러 요청을 동시에 처리하는 메모리 수준 병렬성은 대기를 겹친다. 더 넓은 메모리 채널과 온칩 메모리도 대역폭을 늘릴 수 있다.

이 방법들은 장벽을 완화하지만 없애지는 않는다. 작업 집합이 캐시보다 크거나 접근 패턴이 불규칙하고, 공유 데이터의 일관성·동기화 비용이 크면 데이터 이동이 다시 성능을 제한한다. 또한 계층을 복잡하게 만들수록 소프트웨어가 지역성을 표현하고 활용하는 일이 중요해진다.

## Roofline과의 연결

[[Roofline An Insightful Visual Performance Model]]은 메모리 장벽을 연산 집약도와 대역폭으로 정량화한다.

`달성 성능 ≤ min(최고 연산 성능, 메모리 대역폭 × 연산 집약도)`

바이트당 연산 수가 낮은 작업은 연산기가 아무리 빨라도 메모리 대역폭 지붕에 제한된다. 이 모델은 최고 FLOP/s가 실제 성능을 대표하지 못하는 이유와, 데이터 재사용을 늘리는 최적화가 왜 중요한지를 보여준다.

## 컴퓨팅 능력에 주는 의미

메모리 장벽은 컴퓨팅 능력이 연산기의 속도만으로 결정되지 않음을 분명히 한다. 유효 성능을 비교하려면 작업의 데이터 크기와 접근 패턴, 캐시·메모리 구성, 달성 대역폭과 지연 시간을 함께 기록해야 한다. [[도메인 특화 가속기]]가 큰 온칩 메모리와 맞춤 데이터 경로를 사용하는 것도 연산보다 데이터 이동의 비용을 줄이기 위해서다.
