## 개요

[[벤치마크 계약]]은 수치 하나가 무엇을 뜻하는지 정하는 측정의 경계다. 성능·전력·에너지 결과는 숫자와 단위만으로 완결되지 않는다. 어떤 작업과 입력을 수행했는지, 어떤 답을 유효하다고 셌는지, 어떤 시스템과 시간 구간을 측정했는지, 어떤 규칙·판본 아래 계산했는지가 함께 고정되어야 결과를 다시 실행하거나 다른 결과와 비교할 수 있다.

계약은 벤치마크를 지나치게 좁게 만드는 부수 조건이 아니다. [[SPEC CPU 2026 Overview]]의 실행 시간 중심 SPECspeed와 처리량 중심 SPECrate, [[The Linpack Benchmark]]의 측정된 `Rmax`와 이론적 `Rpeak`, [[MLPerf Inference Benchmark]]의 모델·품질·시나리오·판본은 각각 서로 다른 질문을 구분한다. 같은 시스템의 수치라도 질문이 달라지면 우열의 의미가 달라진다.

## 계약을 이루는 여섯 요소

| 요소 | 반드시 고정하거나 보고할 것 | 빠지면 생기는 혼동 |
|---|---|---|
| 작업 부하 | 프로그램·모델·알고리즘, 입력 크기·분포, 데이터셋, 요청 도착 과정 | 같은 장치의 다른 강점을 같은 성능으로 오해한다. |
| 결과 조건 | 잔차·수치 오차·출력 길이·과업 품질·SLO와 검증 방법 | 더 빠른 실행이 다른 답 또는 낮은 품질을 낸 교환을 숨긴다. |
| 실행 시나리오 | 단일 작업 또는 동시 작업, 배치, 워밍업·유휴 포함 여부, 시작·종료 시점 | 실행 시간·처리량·지연·에너지가 서로 다른 구간을 가리킨다. |
| 시스템·계측 경계 | 프로세서·메모리·가속기·네트워크·저장장치, 클라이언트와 전력계의 위치 | GPU 센서값·TDP·벽면 AC·노드 전력을 같은 값처럼 취급한다. |
| 지표·단위·분모 | 시간, 요청/초, FLOP/s, `J/request`, `J/token`, 성능/W와 집계 방법 | 전력(W)과 에너지(J), 입력·출력 토큰, 평균과 백분위를 섞는다. |
| 규칙·판본·반복 | 허용된 최적화, 소프트웨어·컴파일러 구성, 규칙 판본, 반복·분산·보고 형식 | 다른 버전·구현·난이도에서 나온 수치를 직접 순위화한다. |

결과 조건은 특히 중요하다. [[MLPerf Training Benchmark]]은 학습 처리량이 아니라 정해진 품질 목표에 도달한 시간을 비교하고, [[MLPerf Inference Benchmark]]은 추론에서 품질·생성 길이와 시나리오별 지연 조건을 함께 둔다. 즉 품질은 속도 뒤에 붙는 주석이 아니라, 어떤 실행을 완료로 셀지 정하는 계약의 일부다.

## 수치의 종류도 계약의 일부다

동일한 단위를 사용하더라도 같은 종류의 수치가 아닐 수 있다. LINPACK의 `Rmax`는 지정한 문제와 검증 조건에서 실제로 달성한 성능이고, `Rpeak`는 하드웨어 사양에서 계산한 이론적 상한이다. [[Roofline An Insightful Visual Performance Model]]의 지붕도 실행 결과의 순위가 아니라, 연산 집약도와 메모리 대역폭 아래에서 가능한 성능 상한과 병목을 설명하는 분석 모델이다.

전력·에너지에서는 이 차이가 더 커진다. [[Power Measurement Tutorial for the Green500 List]]은 같은 LINPACK 실행 구간의 성능과 평균 전력을 결합해 GFLOPS/W를 정의한다. [[MLPerf Inference Power Measurement]]과 [[TokenPowerBench - Benchmarking the Power Consumption of LLM Inference]]는 성능 실행과 전력·에너지 수집의 시간 구간, 그리고 GPU·노드·전체 시스템 같은 계측 경계를 맞춰야 `J/token`이나 성능/W가 의미를 가진다고 보여 준다. 낮은 W가 같은 작업의 낮은 J를 자동으로 뜻하지는 않는다.

## 비교 전에 묻는 질문

두 결과가 같은 계약의 값인지 먼저 확인한다.

1. 작업과 입력·요청 분포가 같은가?
2. 품질·정확성·출력 길이·SLO를 같은 방식으로 통과했는가?
3. 시간·처리량·전력·에너지의 관측 구간과 집계 방식이 같은가?
4. 시스템과 계측 경계, 소프트웨어·규칙 판본이 같은가?
5. 두 수치가 모두 관측값인지, 하나가 이론 상한 또는 분석 모델인지 구분했는가?

하나라도 다르면 결과가 쓸모없어지는 것은 아니다. 그 경우에는 “A가 B보다 빠르다”라고 결론내리기보다, 각 수치가 어떤 조건에서 어떤 강점이나 병목을 보이는지 별도로 서술한다. 공통 계약으로 변환할 직접 근거가 있을 때만 정규화·환산을 사용한다.
