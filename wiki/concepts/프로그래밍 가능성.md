## 개요

[[프로그래밍 가능성]](programmability)은 계산 절차를 기계의 물리적 구조를 다시 설계하지 않고 표현하고, 변경하고, 실행하며, 다른 문제에 재사용할 수 있는 정도다. 무엇이 원리적으로 계산 가능한지를 묻는 [[계산 가능성]]과 달리, 프로그래밍 가능성은 그 계산을 사람이 실제 기계에 얼마나 효과적으로 부여할 수 있는지를 묻는다.

프로그래밍 가능성은 있음과 없음으로만 나뉘지 않는다. 천공 카드로 연산 순서를 정하는 기계, 메모리의 기계어를 바꾸는 저장 프로그램 컴퓨터, 고급 언어와 컴파일러를 사용하는 시스템은 모두 프로그램 가능하지만 절차를 표현하고 수정하는 비용과 적용 범위가 다르다.

## 보편성과의 구분

[[On Computable Numbers with an Application to the Entscheidungsproblem]]의 [[보편 튜링 기계]]는 다른 기계의 기술(description)을 입력으로 받아 그 동작을 모사할 수 있다는 이론적 보편성을 제시한다. 이는 하나의 기계가 프로그램에 따라 여러 계산을 수행한다는 관점의 형식적 기반이다.

그러나 이론적으로 보편적인 기계가 곧 사용하기 쉬운 컴퓨터인 것은 아니다. 프로그램을 비트와 주소 수준에서 직접 작성해야 하고, 입출력·적재·디버깅·재사용 수단이 없다면 같은 계산 가능 범위도 실제 이용 비용이 매우 크다. 보편성은 가능한 계산의 범위를, 프로그래밍 가능성은 그 범위에 접근하고 변경하는 비용을 설명한다.

## 역사적 층위

| 층위 | 프로그램을 부여하는 방식 | 줄어든 비용 | 새로 생긴 과제 |
|---|---|---|---|
| 절차의 외부 지정 | 카드와 기계 동작 순서로 절차를 구성 | 기계 전체를 새로 만들 필요 | 장치가 실제로 구현되지 않거나 표현이 기계에 밀착됨 |
| 보편 기계의 기술 | 다른 기계의 규칙을 입력 기호로 부호화 | 한 모델로 여러 계산 절차를 설명 | 이론적 기술을 실제 장치로 구현해야 함 |
| 저장 프로그램 | 명령과 데이터를 메모리에 저장 | 배선 대신 메모리 내용으로 작업 변경 | 기계어 입력·주소·적재와 오류 진단 |
| 로더·서브루틴·라이브러리 | 반복되는 작성·결합 작업을 소프트웨어에 위임 | 프로그램 준비와 재사용 비용 | 호출 규약·재배치·문서화 |
| 고급 언어·컴파일러 | 문제 영역의 표기를 기계어로 번역 | 기계 세부를 직접 다루는 부담 | 번역 정확성·목적 코드 효율 |
| 운영체제·이식성 | 공통 파일·프로세스·시스템 인터페이스 사용 | 여러 사용자와 기계에서 프로그램 운용 | 호환성·자원 관리·시스템 경계 |

[[컴퓨팅의 기원 - 배비지와 러브레이스]]의 [[해석 기관]]은 카드, 반복과 조건 분기로 범용 프로그램 가능성의 개념적 선구를 보여준다. [[First Draft of a Report on the EDVAC]]와 [[The Manchester Small Scale Experimental Machine - The Baby]]는 프로그램 명령을 데이터처럼 메모리에 저장하고 바꾸는 설계와 실행을 보여준다.

## 소프트웨어 계층의 역할

저장 프로그램은 배선 변경을 메모리 내용 변경으로 바꾸었지만, 프로그램 준비 자체를 자동화하지는 않았다. [[초기 소프트웨어의 탄생]]에서 다룬 [[Initial Orders]], 로더, 주소 보정과 진단 루틴은 기계 내부 형식을 다루는 반복 작업을 소프트웨어 계층으로 옮겼다.

[[The Preparation of Programs for an Electronic Digital Computer]]의 서브루틴 라이브러리는 이미 작성하고 검증한 계산을 호출 규약과 문서에 따라 재사용하게 했다. 프로그램 가능성은 새 코드를 작성하는 능력뿐 아니라 기존 계산을 찾아 조합하고 오류를 관찰하는 능력으로 넓어졌다.

[[FORTRAN Automatic Coding System for the IBM 704 EDPM]]과 [[The FORTRAN Automatic Coding System]]은 수학식에 가까운 [[소스 프로그램]]을 효율적인 [[목적 프로그램]]으로 바꾸는 [[컴파일러]]를 제공했다. 하드웨어의 속도를 유지하면서 작성 비용을 줄이는 일이 고급 언어의 실용성을 결정했다는 점에서, 프로그래밍 가능성과 실현 성능은 서로 독립적이지 않다.

[[The UNIX Time-Sharing System]]과 [[Portability of C Programs and the UNIX System]]은 파일·프로세스·시스템 호출과 이식성 있는 구현을 통해 하나의 기계에서 작성한 프로그램과 도구를 더 넓은 환경에서 운용하는 방향을 보여준다. 프로그램 가능성의 단위가 명령열에서 재사용 가능한 소프트웨어 환경으로 확장된 것이다.

## 컴퓨팅 능력과의 관계

프로그래밍 가능성은 새로운 계산 가능한 함수를 만들어 내는 물리적 자원은 아니다. 대신 제한된 인간 시간 안에 더 큰 프로그램을 작성하고, 이미 검증된 구성 요소를 재사용하며, 다른 기계로 옮길 수 있게 한다. 그 결과 이론적으로 가능하지만 구현 비용 때문에 사용할 수 없던 계산이 실제 작업 능력으로 전환된다.

따라서 [[컴퓨팅 능력이란 무엇인가|컴퓨팅 능력]]을 기계 사양만으로 설명하면 중요한 층위가 빠진다. 같은 하드웨어도 언어, 컴파일러, 라이브러리와 운영체제에 따라 사용자가 실현할 수 있는 작업의 범위와 성능이 달라진다.

## 평가할 때 확인할 질문

- 새 작업을 위해 물리적 배선, 기계어, 고급 언어 가운데 무엇을 바꿔야 하는가.
- 프로그램 작성·번역·적재·디버깅의 책임이 사람과 소프트웨어 사이에 어떻게 나뉘는가.
- 이미 작성하고 검증한 계산을 어떤 인터페이스로 재사용할 수 있는가.
- 프로그램 의미를 유지하면서 다른 기계나 운영 환경으로 옮길 수 있는가.
- 추상화가 늘어날 때 실행 성능과 오류 관찰 가능성을 어떻게 보존하는가.
