## 개요

[[연속 배칭]](continuous batching)은 자동회귀 생성 요청을 배치에 한 번 넣어 모든 요청이 끝날 때까지 고정하지 않고, 모델의 한 번 또는 작은 실행 단위가 끝날 때마다 다음에 실행할 활성 요청 집합을 다시 구성하는 서빙 방식이다. [[Orca - A Distributed Serving System for Transformer-Based Generative Models]]은 이 핵심 동작을 **반복 단위 스케줄링**(iteration-level scheduling)이라고 정의했다.

일반적인 신경망 추론은 요청 하나가 모델 한 번의 실행으로 끝날 수 있어 여러 요청을 묶은 뒤 배치 전체가 완료될 때 반환하는 방식이 자연스럽다. [[Transformer]] 기반 자동회귀 생성은 다르다. 요청마다 생성할 토큰 수가 다르고, 한 토큰이 나올 때마다 다음 토큰을 위한 모델 실행이 필요하다. 긴 요청이 포함된 고정 배치에서는 짧은 요청이 이미 끝나도 반환을 기다리고, 새 요청도 다음 배치까지 기다린다.

## 반복마다 바뀌는 활성 집합

연속 배칭 스케줄러는 한 반복이 끝나면 다음 순서로 상태를 갱신한다.

1. 이번 반복에서 종료 조건을 만족한 요청을 완료 처리한다.
2. 남은 요청의 새 토큰과 [[KV 캐시]] 상태를 보존한다.
3. 큐의 신규 요청을 사용 가능한 배치·메모리 용량 안에서 받아들인다.
4. 다음 반복에 실행할 요청과 토큰을 묶어 실행 엔진에 전달한다.

이 방식은 먼저 끝난 요청의 자리를 새 요청이 이어받게 해 가속기의 유휴 구간을 줄인다. 그러나 “계속 배치한다”는 이름이 모든 요청을 매 순간 한 텐서로 합친다는 뜻은 아니다. Orca의 선택적 배칭은 선형층·정규화 같은 위치별 연산은 토큰 축으로 합치고, 요청별 문맥을 구분해야 하는 어텐션은 각 요청의 상태 경계를 유지한다.

## 처리량과 지연의 교환

활성 요청을 많이 모을수록 행렬 연산의 하드웨어 이용률과 총 처리량은 높아질 수 있다. 반면 신규 요청을 즉시 넣을지, 기존 요청을 우선할지, 입력 토큰이 긴 요청을 한 번에 처리할지에 따라 큐 대기와 요청별 완료 시간이 달라진다. 따라서 연속 배칭은 단순한 커널 최적화가 아니라 **승인 제어·스케줄링·실행 엔진·메모리 관리**의 정책이다.

Orca의 평가 이점은 서로 다른 도착 시점과 생성 길이를 가진 요청을 다룰 때 크게 나타났다. 낮은 부하에서는 배치할 요청 자체가 적어 실행 엔진의 단일 배치 성능이 더 중요할 수 있다. 높은 부하에서도 [[KV 캐시]]가 GPU 메모리를 채우면 새 요청을 활성 집합에 넣을 수 없다. [[Efficient Memory Management for Large Language Model Serving with PagedAttention]]이 블록 기반 캐시 관리로 이 수용량 문제를 이어서 다룬 이유다.

## 측정 조건

연속 배칭의 효과를 비교하려면 다음 조건을 함께 고정하거나 보고해야 한다.

- 모델, 정밀도, GPU 수와 모델 병렬화 방식
- 입력·출력 길이의 분포와 요청 도착 과정
- 최대 활성 요청·토큰 수와 승인·선점 정책
- 요청당 지연 분포와 동일 지연 목표에서의 처리량
- KV 캐시 용량, 메모리 낭비, 재계산·교환·선점 횟수

단일한 최대 배치 크기나 초당 토큰 수만으로는 짧은 요청이 긴 요청 뒤에서 얼마나 기다렸는지 알 수 없다.

배치 확대를 평균 처리율 하나로 판단하면 어떤 요청이 기한을 놓쳤는지 사라진다. Clockwork는 일반 DNN의 예측 가능한 실행 시간을 대상으로 배치 크기와 요청 기한을 함께 판단했으며, 자동회귀 LLM은 반복 단위·KV 캐시·TTFT·TPOT 때문에 같은 정책을 그대로 쓸 수 없다. 두 경우 모두 [[대기열과 부하 제어]]는 배치 효율, 대기, 거부·취소를 같은 서비스 결과로 기록하게 한다.
