## 개요

[[소프트웨어 공학]]은 소프트웨어를 신뢰할 수 있고, 이해 가능하며, 유지보수 가능한 구조로 만드는 공학적 실천이다. 이번 소스는 소프트웨어 공학의 씨앗이 1960년대 말의 "소프트웨어 위기" 이전에도 [[EDSAC]] 주변의 디버깅, 서브루틴, 라이브러리, 언어 설계에서 이미 나타났다고 본다.

## 초기 원리

- 관찰 가능성: [[디버깅]] 루틴은 프로그램 내부 상태를 드러내 오류를 찾게 했다.
- 모듈화: [[서브루틴]]은 긴 명령열을 독립 기능 단위로 나누었다.
- 재사용: [[라이브러리]]는 검증된 루틴을 공유 가능한 자산으로 만들었다.
- 인터페이스 문서화: [[라이브러리 카탈로그]]는 루틴의 동작과 사용 규약을 정리해 현대 [[API]] 문서의 전사를 보여준다.
- 계층화: [[Initial Orders]]와 [[부트스트랩]]은 낮은 수준 절차가 높은 수준 실행을 가능하게 하는 구조를 보여주었다.
- 배치와 결합: [[재배치]]와 초기 [[링커]] 문제는 여러 코드 조각을 안정적으로 조합하는 과제를 드러냈다.
- 제어 흐름 관리: [[복귀 주소]], [[Wheeler Jump]], [[스택]]은 호출 후 정확히 돌아오는 일이 독립적인 설계 문제임을 보여준다.
- 제어 구조화: [[GOTO 문]] 논쟁과 [[구조적 프로그래밍]]은 실행 흐름을 사람이 추적하고 검증할 수 있는 구조로 제한해야 함을 보여준다.
- 표현 설계: [[프로그래밍 언어]]는 문제를 사람이 이해하고 기계가 실행할 수 있는 형태로 조직했다.
- 작성 자동화: [[자동 프로그래밍]]은 주소 보정, 서브루틴 조립, 변환 루틴처럼 반복적이고 오류가 잦은 작업을 기계에 맡기는 방향을 보여준다.
- 도구 신뢰성: [[컴파일러]]는 사람이 쓴 [[소스 프로그램]]을 신뢰할 수 있고 효율적인 [[목적 프로그램]]으로 바꾸어야 한다.
- 데이터 경계 관리: [[C 문자열]], [[정수 오버플로]], 초기화되지 않은 메모리는 낮은 수준의 [[데이터 표현]]과 경계 조건이 시스템 신뢰성과 보안으로 이어짐을 보여준다.

## 현대적 의미

현대 개발에서 테스트, 로깅, 패키지 관리, API 설계, 프레임워크, CI/CD, 관찰 가능성은 모두 이 초기 원리의 확장이다. AI 코딩 도구가 확산되어도 중요한 일은 코드 조각을 늘리는 것이 아니라, 표현 체계, 모듈 경계, 검증 전략, 책임 소재를 분명히 만드는 것이다.

[[데이터 표현과 인코딩]]은 이 원리에 보안 관점을 더한다. 추상화된 코드 위에서도 실제 값은 비트와 바이트로 저장되며, 타입 범위와 문자열 경계를 잘못 다루면 프로그램의 결정론적 동작과 정보 보호가 무너질 수 있다.

[[Structured Programming with go to Statements]]는 이 관점에 실용적 균형을 더한다. 좋은 소프트웨어 공학은 특정 문법의 사용 여부만으로 결정되지 않고, 읽기 쉬운 구조에서 출발해 올바름을 유지하며 효율과 예외적 흐름을 처리하는 설계 절차를 요구한다.

[[The FORTRAN Automatic Coding System]]은 소프트웨어 공학의 또 다른 초기 과제를 드러낸다. 사용자가 고급 언어를 신뢰하려면 컴파일러가 정확하고 빠른 목적 프로그램을 만들어야 하며, 번역 과정에서 발견한 오류를 소스 프로그램 수준으로 되돌려 줄 수 있어야 한다. 즉 컴파일러는 언어의 구현체이자 개발 생산성을 좌우하는 공학 도구다.
