## 개요

[[컴퓨팅 능력이란 무엇인가|컴퓨팅 능력]]은 어떤 시스템이 명시된 계산 작업을 정확성 조건과 자원 제약 안에서 수행할 수 있는 범위와 그 속도·효율을 뜻한다. 여기서 시스템은 프로세서만이 아니라 메모리 계층, 컴파일러와 최적화, 라이브러리를 포함한 실행 구성이며, 자원은 시간, 메모리, 데이터 이동, 병렬 처리 장치와 에너지를 포함한다.

따라서 컴퓨팅 능력은 기계에 붙어 있는 하나의 고정된 숫자가 아니다. “무엇을 계산할 수 있는가”, “문제 규모가 커질 때 필요한 자원이 어떻게 증가하는가”, “특정 구성에서 실제로 얼마나 빨리 처리하는가”, “자원을 늘렸을 때 얼마나 확장되는가”, “그 성능을 얼마의 에너지로 내는가”라는 서로 다른 질문의 묶음이다.

> 컴퓨팅 능력은 특정 시스템이 특정 계산을 정확하게, 주어진 시간·메모리·데이터 이동·병렬 자원·에너지 예산 안에서 수행하는 정도다.

## 다섯 층위

| 층위 | 핵심 질문 | 대표 표현 | 이 층위가 말하지 않는 것 |
|---|---|---|---|
| 원리적 가능성 | 이 문제를 푸는 일반 알고리즘이 존재하는가 | 계산 가능·불가능 | 실제로 빨리 풀 수 있는가 |
| 알고리즘 복잡도 | 입력 규모가 커질 때 필요한 자원은 어떻게 증가하는가 | 시간 `T(n)`, 공간 `S(n)`, 복잡도 계층 | 특정 기계에서 걸리는 절대 시간 |
| 실현 성능 | 정해진 작업을 이 구성에서 얼마나 빨리 수행하는가 | 실행 시간, 지연 시간, 처리량, 달성 연산률 | 다른 종류의 작업에서도 같은가 |
| 확장성 | 프로세서·노드 등 자원을 늘리면 얼마나 빨라지는가 | 가속비, 병렬 효율, 순차 비율 | 자원 증가가 무한히 효과적인가 |
| 자원 효율 | 같은 성능이나 작업량에 자원을 얼마나 쓰는가 | 성능당 전력, 작업당 에너지 | 절대 성능이 충분한가 |

### 원리적 가능성

[[On Computable Numbers with an Application to the Entscheidungsproblem]]은 “기계적으로 계산할 수 있다”는 말을 유한한 규칙을 따르는 절차로 형식화한다. 이 [[계산 가능성]]의 층위에서는 더 빠른 프로세서가 있어도 일반 해결 절차가 존재하지 않는 문제와, 원리적으로 절차가 존재하는 문제를 구분한다.

### 알고리즘 복잡도

계산 가능하다는 사실만으로 실용적인 자원 안에 계산할 수 있다는 뜻은 아니다. [[On the Computational Complexity of Algorithms]]는 계산 가능한 대상 사이에도 필요한 연산 수의 증가율에 본질적인 차이가 있음을 다룬다. 같은 문제라도 알고리즘에 따라 자원 요구량이 달라지며, 고정된 기계 속도의 향상은 입력 규모에 따른 점근적 증가율을 없애지 못한다.

### 실현 성능

실현 성능은 알고리즘을 구체적인 하드웨어와 소프트웨어에서 실행한 결과다. [[SPEC CPU 2026 Overview]]가 설명하듯 CPU 중심 벤치마크에도 프로세서, 캐시와 주 메모리, 컴파일러와 최적화가 함께 반영된다. 한 작업의 완료 시간을 보는 지연 시간과 여러 작업의 단위 시간당 완료량을 보는 처리량도 서로 다른 지표다.

[[The Linpack Benchmark]]의 `Rpeak`와 `Rmax`는 이론적 최고 성능과 특정 작업에서 달성한 성능의 차이를 보여준다. `Rpeak`는 하드웨어 사양으로 계산한 최고 부동소수점 실행률이고, `Rmax`는 정해진 정확도 조건 아래 LINPACK 작업으로 실제 달성한 최고 실행률이다. 어느 쪽도 모든 응용 프로그램의 성능을 대표하지 않는다.

### 확장성

자원을 더한다고 성능이 같은 비율로 증가하지는 않는다. [[Validity of the Single Processor Approach to Achieving Large Scale Computing Capabilities]]가 제시한 핵심 직관은 병렬화되지 않는 순차 부분이 전체 가속의 상한을 정한다는 것이다. 병렬화 가능한 비율을 `f`, 프로세서 수를 `p`라고 하고 추가 오버헤드를 무시하면, 통상적인 암달의 법칙은 다음과 같이 쓸 수 있다.

`S(p) ≤ 1 / ((1 - f) + f / p)`

실제 시스템에서는 통신, 동기화, 메모리 충돌과 데이터 재배치 비용이 더해져 달성 가속이 이 이상적 상한보다 낮아질 수 있다. 그러므로 코어 수나 노드 수는 능력 그 자체가 아니라, 작업의 병렬 구조와 함께 해석해야 하는 자원 수다.

### 자원 효율

절대 속도가 높더라도 지나치게 많은 전력을 요구하면 같은 자원 예산에서 수행할 수 있는 작업량은 줄어든다. [[Power Measurement Tutorial for the Green500 List]]는 LINPACK의 달성 성능을 같은 실행 구간의 평균 시스템 전력으로 나눈 `GFLOPS/W`를 사용한다. 이는 특정 작업에 대한 성능당 전력이지 모든 응용의 보편적 에너지 효율은 아니다. 또한 이 자료는 2007년판 역사 지침이므로 현재 Green500 규정을 대신하지 않는다.

## 계산과 데이터 이동의 균형

최고 연산률만으로 실제 성능을 설명할 수 없는 중요한 이유는 데이터를 연산 장치까지 옮겨야 하기 때문이다. [[Roofline An Insightful Visual Performance Model]]은 달성 가능한 성능의 상한을 다음 두 값 중 작은 쪽으로 설명한다.

`달성 성능 ≤ min(최고 연산 성능, 메모리 대역폭 × 연산 집약도)`

연산 집약도(operational intensity)는 주 메모리와 주고받은 바이트당 수행한 연산 수다. 값이 낮은 작업은 메모리 대역폭에, 값이 높은 작업은 연산 장치의 최고 성능에 제한되기 쉽다. 같은 기계에서도 데이터 배치, 캐시 활용, 벡터화와 컴파일러 최적화에 따라 병목과 달성 성능이 달라지는 이유다.

## 측정 지표의 선택

| 알고 싶은 것 | 적합한 지표 | 반드시 함께 기록할 조건 |
|---|---|---|
| 한 작업이 끝나는 데 걸리는 시간 | 실행 시간·지연 시간 | 입력, 정확성, 실행 구성 |
| 일정 시간에 완료하는 작업량 | 처리량 | 동시 작업 수, 작업 혼합, 측정 구간 |
| 부동소수점 계산의 달성률 | FLOP/s, LINPACK `Rmax` | 알고리즘, 문제 크기, 정밀도, 시스템 규모 |
| 자원 추가의 효과 | 가속비 `T(1) / T(p)`, 병렬 효율 | 프로세서 수, 순차 비율, 통신·동기화 비용 |
| 데이터 이동을 포함한 성능 상한 | 연산 집약도와 Roofline 상한 | 메모리 경계, 대역폭, 최고 연산 성능 |
| 에너지 제약 아래의 작업량 | 작업/J, 성능/W | 작업 부하, 전력 측정 경계, 동일한 시간 구간 |

지표를 고를 때는 먼저 사용 목적을 정해야 한다. 대화형 시스템은 지연 시간이 중요할 수 있고, 서버는 처리량, 과학 계산은 특정 수치 커널의 달성률, 전력 제한 시스템은 작업당 에너지가 더 중요할 수 있다. 서로 다른 질문에 답하는 지표를 하나의 순위로 합치면 해석이 흐려진다.

## 비교 가능한 측정의 조건

컴퓨팅 능력에 관한 비교가 재현 가능하려면 다음 조건을 함께 명시해야 한다.

1. **작업 부하**: 어떤 알고리즘과 프로그램을 어떤 입력·문제 크기로 실행했는가.
2. **정확성**: 결과 검증, 수치 정밀도와 허용 오차는 무엇인가.
3. **시스템 경계**: 프로세서, 메모리, 가속기, 컴파일러, 라이브러리와 전력 측정 범위에 무엇이 포함되는가.
4. **실행 조건**: 병렬 자원 수, 컴파일 옵션, 반복 횟수, 측정 구간과 시스템 부하는 무엇인가.
5. **지표와 단위**: 지연 시간, 처리량, 달성 연산률, 가속비, 성능당 전력 가운데 무엇을 보고하는가.
6. **비교 가능성**: 같은 작업·규칙·정밀도·시스템 경계의 결과끼리 비교하는가.

SPEC과 LINPACK은 표준 작업 부하와 실행 규칙을 제공하지만, 실제 사용자의 응용 프로그램을 직접 측정하는 일을 완전히 대신하지는 않는다. 벤치마크는 측정 대상과 제외 대상을 함께 밝힐 때만 의미가 있다.

## 단일 숫자가 실패하는 이유

- 클럭 주파수와 코어 수는 사용할 수 있는 하드웨어 자원의 일부만 나타낸다.
- 최고 FLOP/s는 데이터 이동, 순차 구간과 소프트웨어 오버헤드를 반영하지 않는다.
- 한 벤치마크의 높은 점수는 그 작업 부하에 대한 강점을 뜻하며 다른 작업의 우위를 보장하지 않는다.
- 실행 시간과 처리량은 서로 다른 사용 상황을 측정한다.
- 절대 성능과 에너지 효율은 어느 한쪽으로 환원되지 않는다.
- 정확하지 않은 결과를 빠르게 내는 것은 유효한 컴퓨팅 능력으로 셀 수 없다.

따라서 “이 컴퓨터의 컴퓨팅 능력은 얼마인가?”보다 “이 시스템 구성은 이 작업과 입력을 이 정확성 조건에서 어느 시간·처리량·에너지로 수행하는가?”라고 묻는 편이 더 정확하다.

## 개념의 확장

컴퓨팅 능력의 의미는 역사적으로 층위를 더해 왔다. Turing의 1936년 작업은 계산 가능한 것의 원리적 경계를 형식화했고, Hartmanis와 Stearns의 1965년 논문은 계산 가능한 문제들 사이의 시간 자원 차이를 분리했다. Amdahl의 1967년 논문은 다중 프로세서가 제공하는 가속의 한계를 작업 구조에서 찾았다.

이후 SPEC과 LINPACK 같은 표준 벤치마크는 실제 시스템에서 재현 가능한 작업별 성능 측정을 발전시켰다. Green500의 2007년 전력 측정 지침은 성능을 전력과 결합했고, 2009년 Roofline 모델은 연산 성능과 데이터 이동 사이의 병목을 하나의 상한 모델로 연결했다. 이 흐름은 컴퓨팅 능력이 “연산 장치가 얼마나 빠른가”에서 “어떤 계산을 어떤 자원과 조건으로 얼마나 잘 수행하는가”로 확장되었음을 보여준다.
