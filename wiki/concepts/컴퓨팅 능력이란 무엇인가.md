## 개요

[[컴퓨팅 능력이란 무엇인가|컴퓨팅 능력]]은 어떤 시스템이 명시된 계산 작업을 표현하고 구현해, 정확성 조건과 자원 제약 안에서 수행할 수 있는 범위와 그 속도·효율을 뜻한다. 여기서 시스템은 프로세서만이 아니라 메모리 계층, 언어, 컴파일러와 최적화, 라이브러리, 운영체제를 포함한 실행 구성이며, 자원은 인간의 프로그램 작성 노력, 시간, 메모리, 데이터 이동, 병렬 처리 장치와 에너지를 포함한다.

따라서 컴퓨팅 능력은 기계에 붙어 있는 하나의 고정된 숫자가 아니다. “무엇을 계산할 수 있는가”, “문제 규모가 커질 때 필요한 자원이 어떻게 증가하는가”, “그 계산을 실제 프로그램으로 얼마나 효과적으로 표현할 수 있는가”, “특정 구성에서 얼마나 빨리 처리하는가”, “자원을 늘렸을 때 얼마나 확장되는가”, “그 성능을 얼마의 에너지로 내는가”, “요구한 정확성과 서비스 조건을 만족하는가”라는 서로 다른 질문의 묶음이다.

> 컴퓨팅 능력은 특정 계산을 프로그램으로 표현하고, 특정 시스템에서 정확하게, 주어진 인간 노력·시간·메모리·데이터 이동·병렬 자원·에너지 예산 안에 수행하는 정도다.

### 원리적 가능성

[[On Computable Numbers with an Application to the Entscheidungsproblem]]은 “기계적으로 계산할 수 있다”는 말을 유한한 규칙을 따르는 절차로 형식화한다. 이 [[계산 가능성]]의 층위에서는 더 빠른 프로세서가 있어도 일반 해결 절차가 존재하지 않는 문제와, 원리적으로 절차가 존재하는 문제를 구분한다.

### 알고리즘 복잡도

계산 가능하다는 사실만으로 실용적인 자원 안에 계산할 수 있다는 뜻은 아니다. [[On the Computational Complexity of Algorithms]]는 계산 가능한 대상 사이에도 필요한 연산 수의 증가율에 본질적인 차이가 있음을 다룬다. 같은 문제라도 알고리즘에 따라 자원 요구량이 달라지며, 고정된 기계 속도의 향상은 입력 규모에 따른 점근적 증가율을 없애지 못한다.

### 프로그래밍 가능성

원리적으로 계산할 수 있는 절차도 사람이 매번 배선과 기계어 주소를 직접 구성해야 한다면 실제 사용 범위는 좁다. [[컴퓨팅의 기원 - 배비지와 러브레이스]]의 해석 기관 구상, [[First Draft of a Report on the EDVAC]]의 저장 프로그램 구조, [[The Preparation of Programs for an Electronic Digital Computer]]의 서브루틴 라이브러리, [[The FORTRAN Automatic Coding System]]의 컴파일러는 기계에 새로운 계산을 부여하는 비용을 단계적으로 낮췄다.

[[프로그래밍 가능성]]은 계산을 고급 표기로 표현하고, 기존 구성 요소를 재사용하며, 다른 기계와 운영 환경으로 옮길 수 있는 정도를 다룬다. 이론적 보편성과 같은 말이 아니며, 언어·컴파일러·라이브러리·운영체제가 하드웨어의 잠재력을 실제 작업 능력으로 바꾸는 층위다. 이 역사는 [[범용성은 어떻게 컴퓨팅 능력이 되었는가]]에서 종합한다.

### 실현 성능

실현 성능은 알고리즘을 구체적인 하드웨어와 소프트웨어에서 실행한 결과다. [[SPEC CPU 2026 Overview]]가 설명하듯 CPU 중심 벤치마크에도 프로세서, 캐시와 주 메모리, 컴파일러와 최적화가 함께 반영된다. 한 작업의 완료 시간을 보는 지연 시간과 여러 작업의 단위 시간당 완료량을 보는 처리량도 서로 다른 지표다.

[[The Linpack Benchmark]]의 `Rpeak`와 `Rmax`는 이론적 최고 성능과 특정 작업에서 달성한 성능의 차이를 보여준다. `Rpeak`는 하드웨어 사양으로 계산한 최고 부동소수점 실행률이고, `Rmax`는 정해진 정확도 조건 아래 LINPACK 작업으로 실제 달성한 최고 실행률이다. 어느 쪽도 모든 응용 프로그램의 성능을 대표하지 않는다.

실현 성능이 구성 전체의 속성이라는 관점은 벤치마크보다 앞선 구조 연구에서도 드러난다. [[Slave Memories and Dynamic Storage Allocation]]은 빠른 소용량 기억장치의 효과가 프로그램의 재참조 패턴과 주소 사상·교체·쓰기 정책에 달려 있음을 설명했고, [[Design and Implementation of RISC I]]은 규칙적인 ISA의 잠재력이 레지스터, 메모리 트래픽, 컴파일러와 실제 칩 타이밍을 거쳐야 성능이 된다는 사례를 제공한다.

따라서 높은 클럭이나 짧은 목표 사이클은 달성 성능과 같은 말이 아니다. [[캐시 메모리]]와 [[축소 명령어 집합 컴퓨터]]를 함께 읽으면 작업 부하·컴파일러·ISA·마이크로아키텍처·메모리 계층의 결합이 왜 필요한지 알 수 있으며, 이 관계는 [[더 빠른 프로세서는 왜 더 빠른 프로그램을 보장하지 않는가]]에서 분석한다.

### 확장성

자원을 더한다고 성능이 같은 비율로 증가하지는 않는다. [[Validity of the Single Processor Approach to Achieving Large Scale Computing Capabilities]]가 제시한 핵심 직관은 병렬화되지 않는 순차 부분이 전체 가속의 상한을 정한다는 것이다. 병렬화 가능한 비율을 `f`, 프로세서 수를 `p`라고 하고 추가 오버헤드를 무시하면, 통상적인 암달의 법칙은 다음과 같이 쓸 수 있다.

`S(p) ≤ 1 / ((1 - f) + f / p)`

실제 시스템에서는 통신, 동기화, 메모리 충돌과 데이터 재배치 비용이 더해져 달성 가속이 이 이상적 상한보다 낮아질 수 있다. 그러므로 코어 수나 노드 수는 능력 그 자체가 아니라, 작업의 병렬 구조와 함께 해석해야 하는 자원 수다.

그러나 고정된 문제를 더 빨리 푸는 것만이 병렬 자원의 용도는 아니다. [[Reevaluating Amdahl's Law]]은 병렬 실행 시간을 일정하게 두고 프로세서 수에 맞춰 문제의 병렬 부분을 키우면, 같은 시간에 더 큰 격자나 더 많은 시간 단계를 계산할 수 있다고 설명한다. 병렬 시스템에서 측정한 순차 비율을 `s'`라 하면 확대 크기 가속은 다음과 같다.

`S_scaled(p) = p - (p - 1)s'`

암달식과 Gustafson식은 서로 반박하는 공식이 아니라 고정한 조건이 다르다. 하나는 문제 크기를 고정해 지연 시간 감소를, 다른 하나는 실행 시간을 고정해 처리 가능한 문제 규모 증가를 측정한다. 따라서 [[병렬 확장성]]을 평가할 때는 무엇을 고정했는지와 순차 비율의 분모를 밝혀야 한다. 이 차이는 [[병렬 컴퓨팅은 시간을 줄이는가 문제를 키우는가]]에서 분석한다.

### 자원 효율

절대 속도가 높더라도 지나치게 많은 전력을 요구하면 같은 자원 예산에서 수행할 수 있는 작업량은 줄어든다. [[Power Measurement Tutorial for the Green500 List]]는 LINPACK의 달성 성능을 같은 실행 구간의 평균 시스템 전력으로 나눈 `GFLOPS/W`를 사용한다. 이는 특정 작업에 대한 전력당 성능이지 모든 응용의 보편적 에너지 효율은 아니다. 또한 이 자료는 2007년판 역사 지침이므로 현재 Green500 규정을 대신하지 않는다.

자원 효율도 한 숫자가 아니다. [[Implications of Historical Trends in the Electrical Efficiency of Computing]]에서 유래한 [[Koomey의 법칙]]은 여러 세대의 컴퓨터를 최고 부하의 계산/J로 비교하고, [[The Case for Energy-Proportional Computing]]의 [[에너지 비례 컴퓨팅]]은 한 시스템이 유휴부터 첨두까지 작업량에 맞춰 전력을 낮추는지를 묻는다. [[Dark Silicon and the End of Multicore Scaling]]의 [[다크 실리콘]]은 고정된 칩 전력에서 구현한 회로 중 얼마나 유용하게 동시에 활성화할 수 있는지를 다룬다.

세 관점은 각각 세대 간 최고 부하의 환산 계산량/에너지, 실제 운용의 부하 곡선, 칩 내부의 순간 활성 자원에 답한다. 환산 계산량/에너지의 장기 개선이 높은 유휴 전력이나 순간 전력 밀도를 자동으로 해결하지는 않는다. 이 차이는 [[전력 장벽은 성능 향상의 의미를 어떻게 바꾸었는가]]에서 종합한다.

### 신뢰 가능한 결과

빠르게 끝난 계산도 요구한 답을 내지 못하면 유효한 능력으로 셀 수 없다. [[The Linpack Benchmark]]의 `Rmax`는 정해진 수치 정확도 검증을 통과한 실행만 인정하고, [[SPEC CPU 2026 Overview]]도 정해진 실행 규칙과 결과 검증을 전제로 성능을 비교한다. 정확성, 수치 정밀도와 재현성은 속도 뒤에 붙이는 부가 조건이 아니라 측정 대상을 성립시키는 조건이다.

[[What Every Computer Scientist Should Know About Floating-Point Arithmetic]]은 유한한 수 표현과 반올림, 상쇄와 시스템 지원이 수치 결과에 관여함을 설명한다. [[IEEE 754-2019 Standard for Floating-Point Arithmetic]]은 지정된 입력값·연산 순서·목적 형식 아래 규범 연산의 결과를 공통화하지만, 전체 프로그램의 비트 재현성이나 알고리즘의 정확성을 자동으로 보장하지 않는다. 따라서 [[부동소수점 정확성]]은 비트 동일성, 허용 오차 안의 수치 일치, 과업 의미와 벤치마크 검증을 분리한다. 같은 결과 계약 안에서만 속도 향상을 비교해야 한다는 분석은 [[더 빠른 계산은 같은 답을 내는가]]에서 다룬다.

대화형·분산 서비스에서는 평균 처리량이 충분해도 일부 요청의 매우 긴 지연이나 장애가 사용 가능한 능력을 제한할 수 있다. [[The Tail at Scale]]은 개별 서버의 드문 지연이 큰 팬아웃에서 종단 요청의 흔한 지연으로 증폭됨을 보여준다. 따라서 [[꼬리 지연 시간]]은 평균과 별도로 측정해야 하며, 기한·가용성·결과 품질과 자원 비용의 관계는 [[평균 성능은 왜 서비스의 컴퓨팅 능력을 설명하지 못하는가]]에서 분석한다.

## 계산과 데이터 이동의 균형

최고 연산률만으로 실제 성능을 설명할 수 없는 중요한 이유는 데이터를 연산 장치까지 옮겨야 하기 때문이다. [[Roofline An Insightful Visual Performance Model]]은 달성 가능한 성능의 상한을 다음 두 값 중 작은 쪽으로 설명한다.

`달성 성능 ≤ min(최고 연산 성능, 메모리 대역폭 × 연산 집약도)`

연산 집약도(operational intensity)는 주 메모리와 주고받은 바이트당 수행한 연산 수다. 값이 낮은 작업은 메모리 대역폭에, 값이 높은 작업은 연산 장치의 최고 성능에 제한되기 쉽다. 같은 기계에서도 데이터 배치, 캐시 활용, 벡터화와 컴파일러 최적화에 따라 병목과 달성 성능이 달라지는 이유다.

|---|---|---|
| 한 작업이 끝나는 데 걸리는 시간 | 실행 시간·지연 시간 | 입력, 정확성, 실행 구성 |
| 일정 시간에 완료하는 작업량 | 처리량 | 동시 작업 수, 작업 혼합, 측정 구간 |
| 부동소수점 계산의 달성률 | FLOP/s, LINPACK `Rmax` | 알고리즘, 문제 크기, 정밀도, 시스템 규모 |
| 수치 결과의 유효성 | 비트 일치, 절대·상대·ulp 오차, 잔차 | 형식, 연산 순서, 반올림, 참조값·허용 오차 |
| 자원 추가의 효과 | 가속비 `T(1) / T(p)`, 병렬 효율 | 프로세서 수, 순차 비율, 통신·동기화 비용 |
| 데이터 이동을 포함한 성능 상한 | 연산 집약도와 Roofline 상한 | 메모리 경계, 대역폭, 최고 연산 성능 |
| 에너지 제약 아래의 작업량 | 작업/J, 성능/W | 작업 부하, 전력 측정 경계, 동일한 시간 구간 |
| 실제 운용의 전력 효율 | 활용률별 전력·작업당 에너지 | 유휴–첨두 부하 분포, 전환 비용, 서비스 수준 |
| 칩의 동시 활성 능력 | 활성 면적·활동률·전력 예산 | 코어와 언코어 경계, 작업 병렬성, 열 조건 |
| 대규모 서비스의 예측 가능한 응답 | 종단 p50·p95·p99·p99.9, 기한 초과율 | 요청 혼합, 팬아웃, 부하, 재시도·타임아웃, 결과 품질 |

지표를 고를 때는 먼저 사용 목적을 정해야 한다. 대화형 시스템은 지연 시간이 중요할 수 있고, 서버는 처리량, 과학 계산은 특정 수치 커널의 달성률, 전력 제한 시스템은 작업당 에너지가 더 중요할 수 있다. 서로 다른 질문에 답하는 지표를 하나의 순위로 합치면 해석이 흐려진다.

## 비교 가능한 측정의 조건

컴퓨팅 능력에 관한 비교가 재현 가능하려면 다음 조건을 함께 명시해야 한다.

1. **작업 부하**: 어떤 알고리즘과 프로그램을 어떤 입력·문제 크기로 실행했는가.
2. **정확성**: 비트 동일성, 수치 허용 오차, 잔차와 과업 품질 가운데 무엇을 요구하며 결과를 어떻게 검증하는가.
3. **시스템 경계**: 프로세서, 메모리, 가속기, 네트워크, 저장장치, 큐, 컴파일러와 라이브러리 가운데 무엇이 포함되며 클라이언트 관측의 시작·종료와 전력 측정 범위는 어디인가.
4. **실행 조건**: 병렬 자원 수, 컴파일 옵션, 반복 횟수, 측정 구간과 시스템 부하는 무엇인가.
5. **지표와 단위**: 지연 시간, 처리량, 달성 연산률, 가속비, 전력당 성능 가운데 무엇을 보고하는가.
6. **비교 가능성**: 같은 작업·규칙·정밀도·시스템 경계의 결과끼리 비교하는가.

SPEC과 LINPACK은 표준 작업 부하와 실행 규칙을 제공하지만, 실제 사용자의 응용 프로그램을 직접 측정하는 일을 완전히 대신하지는 않는다. 벤치마크는 측정 대상과 제외 대상을 함께 밝힐 때만 의미가 있다.

## 단일 숫자가 실패하는 이유

- 클럭 주파수와 코어 수는 사용할 수 있는 하드웨어 자원의 일부만 나타낸다.
- 최고 FLOP/s는 데이터 이동, 순차 구간과 소프트웨어 오버헤드를 반영하지 않는다.
- 한 벤치마크의 높은 점수는 그 작업 부하에 대한 강점을 뜻하며 다른 작업의 우위를 보장하지 않는다.
- 실행 시간과 처리량은 서로 다른 사용 상황을 측정한다.
- 평균 지연과 처리량은 높은 백분위의 종단 지연과 기한 초과를 보여주지 않는다.
- 절대 성능과 에너지 효율은 어느 한쪽으로 환원되지 않는다.
- 최고 부하의 성능/W는 유휴·부분 부하의 전력 곡선이나 총에너지를 대표하지 않는다.
- 트랜지스터 수는 전력 예산 안에서 동시에 사용할 수 있는 회로 수를 나타내지 않는다.
- IEEE 754 준수만으로 컴파일러·병렬 순서가 다른 전체 프로그램의 비트 동일성을 알 수 없다.
- 정확하지 않은 결과를 빠르게 내는 것은 유효한 컴퓨팅 능력으로 셀 수 없다.

따라서 “이 컴퓨터의 컴퓨팅 능력은 얼마인가?”보다 “이 시스템 구성은 이 작업과 입력을 이 정확성 조건에서 어느 시간·처리량·에너지로 수행하는가?”라고 묻는 편이 더 정확하다.

## 역사적 확장

컴퓨팅 능력의 의미는 역사적으로 층위를 더해 왔다. 해석 기관과 보편 기계는 하나의 장치에 여러 계산을 부여한다는 범용성의 개념을 제시했고, 저장 프로그램·라이브러리·컴파일러와 운영체제는 그 잠재력을 실제 [[프로그래밍 가능성]]으로 바꾸었다. Turing의 1936년 작업은 계산 가능한 것의 원리적 경계를 형식화했고, Hartmanis와 Stearns의 1965년 논문은 계산 가능한 문제들 사이의 시간 자원 차이를 분리했다. 같은 시기 [[무어의 법칙]]과 이후의 [[Dennard 스케일링]]은 더 많은 소자를 더 빠르고 효율적으로 사용하는 범용 성능 향상의 기술적 배경을 만들었다. 캐시와 RISC의 초기 연구는 이 잠재 자원을 실제 프로그램 성능으로 바꾸려면 데이터 재사용과 컴파일러·구조의 공동 설계가 필요함을 보여주었다. 1985년 IEEE 754의 표준화와 Goldberg의 1991년 해설은 빠른 부동소수점 연산에 이식 가능한 수치 의미와 오차 분석의 조건을 붙였다.

1990년대 이후에는 [[메모리 장벽]], 전력 제약과 병렬화되지 않는 작업이 최고 연산률을 실제 성능으로 바꾸는 일을 제한했다. 세대 간 계산/J가 장기간 개선되는 동안에도 높은 기저 전력과 활성 면적의 제약은 남았고, 성능 향상은 부하별 전력 관리와 선택적 활성화의 문제가 되었다. 멀티코어와 창고 규모 시스템은 능력의 경계를 칩 밖으로 넓혔으며, 팬아웃이 드문 지연을 증폭하면서 분포와 꼬리 내성도 성능의 일부가 되었다. [[도메인 특화 가속기]]는 특정 작업과 데이터 경로에 자원을 집중했다. 이 전환과 그때마다 달라진 측정 지표는 [[컴퓨팅 능력의 발달사]]에서 통합해 다룬다.
