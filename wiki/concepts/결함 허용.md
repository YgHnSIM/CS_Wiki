## 개요

[[결함 허용]](fault tolerance)은 구성요소의 고장이나 오류가 발생해도 시스템이 미리 정한 결과·서비스 조건을 유지하거나, 허용 가능한 시간 안에 그 조건으로 돌아오도록 만드는 설계 원리다. 목표는 “아무 일도 고장 나지 않게 하는 것”과 같지 않다. 어떤 실패를 가정하는지, 어떤 결과를 보존하는지, 어느 정도의 성능 저하와 복구 시간을 허용하는지를 함께 정해야 한다.

실패 모형은 설계가 무엇을 보장할 수 있는지를 바꾼다. [[The Byzantine Generals Problem]]처럼 참여자가 서로 다른 거짓 정보를 보낼 수 있는 [[비잔틴 장애]]를 다루는 경우에는 단순한 복제나 재시작보다 합의와 인증의 조건이 중요하다. 반면 실제 서비스에서는 하드웨어·소프트웨어·네트워크뿐 아니라 설정 변경과 운영 절차도 장애의 원인이 될 수 있다.

따라서 결함 허용은 중복, 격리, 오류 탐지, 상태 점검, 안전한 되돌리기와 복구를 조합하는 문제다. 중복은 독립적인 구성요소 장애를 가릴 수 있지만, 같은 설정이나 공통 코드 경로를 공유한 복제본의 오류까지 자동으로 해결하지는 않는다.

## 설계에서 먼저 고정할 것

| 질문 | 명시할 내용 |
|---|---|
| 어떤 실패를 가정하는가 | 정지·누락·느린 응답·손상된 상태·상호 모순된 메시지·운영자 오류 중 무엇인가 |
| 무엇을 보존하는가 | 안전성, 일관성, 데이터 무결성, 기한, 부분 기능, 사용자 품질 중 무엇인가 |
| 얼마나 오래 저하될 수 있는가 | 탐지·격리·복구의 시간 목표와 허용 가능한 서비스 저하 |
| 실패가 함께 일어날 수 있는가 | 공통 전원·네트워크·배포·설정·인증 체계가 만드는 상관 원인 |
| 복구가 안전한가 | 재시작·되돌리기·재시도 뒤 상태와 결과를 어떻게 검증하는가 |

## 예방·격리·복구의 구분

- **예방**은 결함을 줄이거나 배포 전에 찾으려는 활동이다. 구성 점검과 시험이 여기에 속한다.
- **격리와 마스킹**은 오류가 전체 서비스로 번지는 것을 막고, 독립적인 복제본이나 대체 경로로 사용자 영향을 줄인다.
- **탐지와 진단**은 어떤 조건이 깨졌는지 발견하고 원인을 좁히는 과정이다.
- **복구**는 서비스를 목표 품질로 되돌리고, 되돌린 결과가 안전한지 확인하는 과정이다.

이 단계는 서로 대체되지 않는다. 예방이 있어도 결함은 남고, 중복이 있어도 공통 원인은 남으며, 빠른 재시작이 있어도 잘못된 상태를 다시 사용하면 결과 계약을 지키지 못할 수 있다.

## 적용 경계

결함 허용은 “장애가 있어도 항상 같은 성능”을 뜻하지 않는다. 장애 중에는 처리량을 낮추거나 일부 부가 기능을 생략할 수 있고, 이때는 결과 품질과 서비스 수준이 정상 상태와 같은지 따로 밝혀야 한다. 또한 특정 합의 알고리즘의 정족수 조건을 일반 서비스의 복제 개수로 옮기거나, 장애 보고 연구의 원인 비율을 다른 조직에 그대로 적용해서는 안 된다.
