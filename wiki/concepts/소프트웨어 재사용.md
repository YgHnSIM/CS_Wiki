## 개요

[[소프트웨어 재사용]]은 기존 소프트웨어 지식이나 산출물을 바탕으로 새 소프트웨어를 만드는 실천이다. 가장 좁게는 [[서브루틴]]이나 코드 조각을 다시 쓰는 일을 뜻하지만, 넓게는 [[라이브러리]], [[API]], [[소프트웨어 컴포넌트]], 설계 문서, 테스트, 아키텍처, 도메인 지식까지 포함한다.

재사용은 단순 복사와 다르다. 재사용 가능한 산출물은 무엇을 하는지 설명되어야 하고, 찾을 수 있어야 하며, 필요한 상황에 맞게 선택·변형·결합될 수 있어야 한다. 따라서 재사용의 핵심은 코드의 존재가 아니라 추상화, 명세, 검색 가능성, 변경 가능성, 신뢰성이다.

## 역사적 흐름

초기 [[EDSAC]]의 [[라이브러리]]와 [[라이브러리 카탈로그]]는 검증된 루틴과 사용 명세를 공유 자산으로 만들었다. [[Mass Produced Software Components]]는 이 흐름을 산업적 컴포넌트와 제품군 사고로 확장했다. [[On the Criteria To Be Used in Decomposing Systems into Modules]]는 재사용 가능한 구조가 변경 가능한 설계 결정을 감추는 [[모듈화]]를 요구한다고 보여준다.

1990년대의 [[Software Reuse]]는 재사용을 추상화, 선택, 특수화, 통합의 문제로 정리했다. [[Sixteen Questions about Software Reuse]]는 재사용과 [[이식성]]을 구분하고, 체계적 재사용이 조직과 경험적 실천의 문제이기도 하다는 점을 강조한다.

## 성공 조건

- 추상화: 사용자가 내부 구현을 읽지 않아도 기능을 이해할 수 있어야 한다.
- 발견 가능성: 필요한 산출물을 새로 만드는 것보다 빠르게 찾을 수 있어야 한다.
- 인터페이스: 입력, 출력, 전제 조건, 오류 처리, 비용이 명확해야 한다.
- 특수화: 상황에 맞게 정밀도, 견고성, 성능, 데이터 표현을 조정할 수 있어야 한다.
- 통합: 기존 시스템과 안정적으로 결합될 수 있어야 한다.
- 신뢰: 테스트, 검증, 문서, 사용 경험이 재사용 비용을 낮춰야 한다.

## 한계

재사용은 항상 비용을 줄이지 않는다. 낯선 코드를 찾고 이해하고 수정하고 디버깅하는 비용이 새로 만드는 비용보다 크면 재사용은 실패한다. 또한 범용성을 과하게 높이면 컴포넌트가 무거워지고, 반대로 특수성을 과하게 높이면 다른 상황에 적용하기 어렵다.

## 관련 항목

- [[서브루틴]] — 호출 후 원래 위치로 돌아오는 재사용 가능한 코드 단위.
- [[라이브러리]] — 검증된 재사용 코드 단위를 모아 공동체의 소프트웨어 자산으로 만든 체계.
- [[라이브러리 카탈로그]] — 서브루틴의 동작, 사용법, 실행 시간, 정밀도, 저장 공간을 문서화한 초기 API 문서형 체계.
- [[API]] — 기능 제공자와 사용자 사이의 동작, 호출 규약, 비용, 전제 조건을 정리한 프로그래밍 인터페이스.
- [[소프트웨어 컴포넌트]] — 다른 시스템에서 재사용할 수 있도록 경계, 명세, 변형 가능성, 품질 기준을 갖춘 소프트웨어 산출물.
