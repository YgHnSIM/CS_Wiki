## 개요

[[프리필과 디코드]]는 자동회귀 [[Transformer]] 추론 한 요청의 실행을 두 단계로 나누는 구분이다. **프리필(prefill)**은 프롬프트의 입력 토큰을 처리해 첫 출력 토큰과 층별 [[KV 캐시]]를 만든다. **디코드(decode)**는 저장된 키·값을 사용해 이후 출력 토큰을 하나씩 생성하고, 새 토큰의 상태를 캐시에 덧붙인다.

두 단계는 같은 모델 가중치와 어텐션을 사용하지만 병렬성과 지연 의미가 다르다. 프리필은 한 요청의 여러 입력 위치를 병렬로 계산할 수 있고, 충분히 긴 입력에서는 큰 행렬 연산으로 계산 장치를 활용하기 쉽다. 디코드는 다음 토큰이 이전에 생성한 토큰에 의존하므로 한 요청 안에서 순차적이다. 여러 요청을 [[연속 배칭]]해야 큰 행렬 연산을 만들 수 있으며, 매 반복마다 가중치와 누적 캐시를 읽는 메모리 대역폭의 영향을 크게 받을 수 있다.

## 사용자 지연에서의 경계

프리필이 끝날 때까지 사용자는 첫 출력을 받지 못한다. 따라서 첫 토큰 시간(time to first token, TTFT)은 큐 대기, 프리필 실행과 첫 토큰 반환 경로를 포함하는 사용자 지연이다. 이후 생성은 토큰 간 지연 또는 출력 토큰당 시간(time per output token, TPOT)으로 관찰한다. 전체 응답 시간은 이 둘과 실제 출력 길이에 함께 좌우된다.

TTFT와 TPOT를 하나의 평균 요청 지연으로 합치면 원인을 찾기 어렵다. 긴 프롬프트는 프리필을 늘리고, 긴 출력은 디코드 반복 수를 늘린다. 같은 종단 지연이어도 첫 출력이 늦고 이후 토큰이 빠른 서비스와 첫 출력은 빠르지만 토큰 간 대기가 긴 서비스는 상호작용 경험이 다르다.

## 공배치와 단계 분리

한 GPU에서 프리필과 디코드를 함께 처리하면 가중치를 공유하고 KV 상태를 네트워크로 옮기지 않아도 된다. 그러나 큰 프리필 작업이 끼어들면 디코드 반복 사이 간격이 길어질 수 있고, 디코드 우선 정책은 새 요청의 TTFT를 늘릴 수 있다. 두 단계에 같은 GPU 수와 병렬화 구성을 적용해야 하는 자원 결합도 생긴다.

[[DistServe - Disaggregating Prefill and Decoding for Goodput-optimized Large Language Model Serving]]는 프리필과 디코드를 서로 다른 GPU 군에 배치해 각 단계의 병렬화·복제와 SLO를 독립적으로 조정한다. 대신 모델 가중치를 중복 적재하고 프리필이 만든 KV 캐시를 디코드 군으로 전송한다. 따라서 단계 분리가 유리한지는 입력·출력 길이 분포, 요청 도착률, GPU 메모리, 네트워크 대역폭과 TTFT·TPOT 목표에 달려 있다.

## 측정할 것

| 단계 | 주요 작업 | 사용자 지연 | 자원·스케줄링 조건 |
|---|---|---|---|
| 프리필 | 입력 토큰 처리, 첫 토큰과 초기 KV 상태 생성 | TTFT | 입력 길이, 큐 대기, 청크 크기, 계산 병렬성 |
| 디코드 | 저장된 KV를 읽어 이후 토큰을 순차 생성 | TPOT·토큰 간 지연 | 출력 길이, 활성 요청·토큰, 메모리 대역폭, 배치 |
| 단계 사이 | KV 상태와 요청 소유권 전달 | 전송·재대기 지연 | KV 크기, 네트워크 대역폭, 배치와 배치 위치 |

“프리필은 계산 병목, 디코드는 메모리 병목”은 유용한 출발점이지만 보편 법칙은 아니다. 짧은 입력, 작은 모델, 매우 큰 배치, 양자화, 병렬화와 하드웨어에 따라 병목이 달라질 수 있다. 단계 이름만으로 병목을 추정하지 말고 실제 연산률, 메모리 트래픽, 큐와 지연 분포를 함께 측정해야 한다.
