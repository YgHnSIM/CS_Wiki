## 개요

[[축소 명령어 집합 컴퓨터]](Reduced Instruction Set Computer, RISC)는 자주 실행되는 명령을 규칙적이고 빠르게 처리하도록 명령어 집합 구조(ISA), 데이터 경로, 제어부와 컴파일러의 역할을 함께 조정하는 설계 접근이다. 이름의 “축소”는 단순히 명령어 목록의 길이가 짧다는 뜻보다, 복잡하고 드문 기능이 흔한 실행 경로의 사이클과 구현을 지배하지 않게 하려는 선택에 가깝다.

초기 RISC I의 대표 선택은 제한된 주소 지정 방식, 레지스터 간 산술, 메모리를 `LOAD`와 `STORE`로만 접근하는 구조, 고정 길이 명령 형식과 작은 제어부였다. 이 규칙성은 파이프라인과 코드 생성을 단순하게 만들려 했지만, 실제 성능은 컴파일러가 레지스터와 명령 순서를 얼마나 잘 활용하는지에 달려 있었다.

## RISC I의 문제 설정

[[Design and Implementation of RISC I]]은 메인프레임 구조를 초기 단일 칩 VLSI에 그대로 옮기지 않고, 제한된 면적·배선·핀과 설계 시간을 어디에 배분할지 다시 물었다. 연구팀은 C와 Pascal 프로그램의 동적 실행 빈도를 조사해 단순 스칼라 연산과 프로시저 호출이 자주 나타난다고 보았다.

RISC I은 많은 레지스터와 겹치는 레지스터 윈도를 사용해 지역 변수와 호출 매개변수를 칩 안에 두려 했다. 지연 분기는 분기 직후의 슬롯을 컴파일러가 유용한 명령으로 채우게 했다. 다만 레지스터 윈도와 지연 분기는 RISC I의 구체적 해법이며 RISC 전체를 정의하는 필수 조건은 아니다.

## 복잡성은 사라지지 않는다

RISC의 단순화는 기능을 없애기보다 책임을 다른 층으로 옮긴다.

| 설계 선택 | 기대한 이점 | 이동한 비용 |
|---|---|---|
| 규칙적인 고정 길이 형식 | 빠른 해독과 단순한 제어 | 코드 크기와 명령어 인출 증가 가능성 |
| 레지스터 간 연산과 `LOAD`/`STORE` | 데이터 경로와 메모리 접근의 명료화 | 명시적인 데이터 이동 명령 필요 |
| 작은 제어부 | 짧은 사이클과 검증 가능한 배치 | 복합 동작을 여러 명령 또는 소프트웨어로 구성 |
| 컴파일러 스케줄링 | 파이프라인 빈칸 활용 | 코드 생성기와 [[컴파일러 최적화]]의 책임 증가 |

RISC I 보고서는 데이터 메모리 접근을 크게 줄였다고 평가하는 동시에, `NOP`와 고정 길이 인코딩 때문에 명령어 접근과 코드 크기가 늘 수 있다고 인정한다. 결론에서 명령어 캐시가 필요하다고 한 점은 단순한 ISA도 [[캐시 메모리|메모리 계층]]과 분리해 평가할 수 없음을 보여준다.

## 성능 주장을 읽는 법

RISC I의 여섯 C 프로그램 비교는 당시 VAX 11/780, Z8000과 서로 다른 컴파일러를 사용한 제한된 사례다. RISC I 시간은 400ns 사이클을 가정한 시뮬레이션이었고, 실제 칩은 놓친 임계 경로 때문에 그 목표에 도달하지 못했다. 그러므로 이 자료로 “RISC는 언제나 다른 명령어 집합보다 빠르다”고 일반화할 수 없다.

더 안전한 결론은 RISC가 작업 부하의 빈도, VLSI 구현 비용과 컴파일러 능력을 근거로 복잡성을 재배분한 역사적 설계 전략이라는 것이다. 명령 수, 사이클 수, 코드 크기, 메모리 트래픽과 실제 회로의 타이밍이 함께 측정되어야 어느 선택이 유리한지 판단할 수 있다.

## 컴퓨팅 능력에 주는 의미

RISC는 [[컴퓨팅 능력이란 무엇인가|컴퓨팅 능력]]이 ISA나 클럭 하나에 내재한 숫자가 아님을 보여준다. 고급 언어의 프로그램이 컴파일러를 거쳐 명령열이 되고, 그 명령열이 레지스터·파이프라인과 메모리 계층을 통과해 실행될 때 비로소 달성 성능이 생긴다. 이 변환 과정은 [[더 빠른 프로세서는 왜 더 빠른 프로그램을 보장하지 않는가]]에서 캐시 사례와 나란히 분석한다.
