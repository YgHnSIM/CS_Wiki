## 개요

[[컴파일러 최적화]]는 [[컴파일러]]가 프로그램의 의미를 유지하면서 더 빠르거나 더 작은 [[목적 프로그램]]을 만들기 위해 코드 구조를 바꾸는 작업이다. Fortran의 초기 성공은 고급 언어가 편하다는 점뿐 아니라, 번역 결과가 손으로 짠 코드에 가까운 성능을 낼 수 있다는 주장과 구현에 달려 있었다.

## Fortran 사례

1957년 Fortran 논문은 산술식 번역에서 같은 값을 불필요하게 다시 계산하지 않거나, 특정 지수 연산을 더 싼 곱셈으로 처리하는 예를 보여준다. 그러나 더 큰 난점은 산술식 자체보다 `DO` 반복, 첨자 변수, 입출력, `FORMAT`, 인덱스 레지스터 관리 같은 주변 작업이었다.

Fortran 번역기는 목적 프로그램의 제어 흐름을 분석하고, IBM 704의 제한된 인덱스 레지스터에 맞게 프로그램을 변환했다. 매뉴얼의 `FREQUENCY` 문은 프로그래머가 분기나 반복의 예상 빈도를 알려 주어 목적 프로그램의 자원 배치를 돕는 초기 최적화 힌트로 볼 수 있다.

## 구조와의 공동 설계

[[Design and Implementation of RISC I]]은 컴파일러 최적화가 하드웨어 설계와 분리된 사후 작업이 아님을 보여준다. RISC I의 규칙적인 명령 형식, 많은 레지스터와 지연 분기는 컴파일러가 값을 레지스터에 배치하고 독립적인 명령을 재배열할 수 있다는 가정을 포함했다. 지연 슬롯을 유용한 명령으로 채우지 못하면 `NOP`가 남고, 레지스터를 충분히 활용하지 못하면 메모리 접근이 늘어난다.

이 사례에서 [[축소 명령어 집합 컴퓨터|RISC]]의 단순성은 성능을 보장하는 성질이 아니라 컴파일러가 활용할 기회를 제공하는 구조적 선택이다. 같은 ISA와 프로세서도 코드 생성에 따라 명령 수, 명령당 사이클, 코드 크기와 메모리 트래픽이 달라질 수 있다. 이 변환 조건은 [[더 빠른 프로세서는 왜 더 빠른 프로그램을 보장하지 않는가]]에서 메모리 계층과 함께 분석한다.

## 수치 의미와 최적화

부동소수점에서는 “프로그램 의미를 유지한다”는 말에 수치 계약이 필요하다. [[What Every Computer Scientist Should Know About Floating-Point Arithmetic]]이 설명하듯 각 연산에 반올림이 들어가므로 실수 대수에서 같은 식도 연산 순서를 바꾸면 결과 비트와 예외 발생 시점이 달라질 수 있다. 중간 결과를 어느 정밀도와 형식에 두는지도 결과에 관여한다.

[[IEEE 754-2019 Standard for Floating-Point Arithmetic]]은 입력값·연산 순서·목적 형식이 주어진 규범 연산의 결과를 공통화하지만, 컴파일러가 다른 순서를 선택한 전체 프로그램까지 같은 비트로 만들지는 않는다. 따라서 최적화기는 언어와 옵션이 요구하는 비트 동일성, [[부동소수점 정확성|수치 허용 오차]]와 예외 의미 안에서 재배열·벡터화·병렬화를 수행해야 한다. 이 조건을 넘어서 얻은 속도는 같은 계산의 단순한 가속으로 셀 수 없으며, [[더 빠른 계산은 같은 답을 내는가]]에서 별도의 정확성·성능 교환으로 분석한다.

## 의미

컴파일러 최적화는 고급 언어와 기계 효율 사이의 불신을 줄이는 역할을 했다. 프로그래머가 수학적 표기로 절차를 쓰더라도, 컴파일러가 기계의 구체적 제약을 대신 처리해 준다면 고급 언어는 실용적 선택지가 된다.
